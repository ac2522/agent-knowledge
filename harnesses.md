# Harnesses

## What the harness is / harness vs model+prompt
- Harness = everything *around* the model (tool registry, model, context primitives, guardrails, agent loop, verify step) — often a loop *around* the agent loop (retry/max-attempts wrapper), not the agent loop itself; Claude Code / Codex CLI are themselves harnesses. [IBM](https://www.youtube.com/watch?v=C_GG5g38vLU) [MLOps-Sandboxing](https://podcasters.spotify.com/pod/show/mlops/episodes/Sandboxing--Agent-Harnesses--and-Agent-Teamwork-e3kts75) (2026-06)
- A harness has two jobs: give the agent fast feedback loops, and protect the agent from itself (it tends to do dumb things) — via isolation OR right-tool-at-right-time, not just guardrail prose. [MLOps-Sandboxing](https://podcasters.spotify.com/pod/show/mlops/episodes/Sandboxing--Agent-Harnesses--and-Agent-Teamwork-e3kts75) (2026-06)
- Every agent is a while loop with conditions + end states; keep a mental model of which state it's in — makes designing/debugging far easier. The architecture/state machine must be designed by a human, thoughtfully — don't let models rip through it; spend real time on the design even when pairing with an agent. [Cline](https://www.youtube.com/watch?v=yUmS-F9IX90) (2026-05)
- Inner vs outer harness: inner harness = the while loop in one process/pod (model + tool calls + filesystem); outer harness deconstructs it so the "brain" (loop deciding which tool to call) is separated from the "hands" (tool execution in a sandbox / external state store) — once split, run multiple brains × multiple hands as system architecture. [ZenML](https://podcasters.spotify.com/pod/show/mlops/episodes/Agents-are-Just-While-Loops-e3j3a6q) (2026-05)
- Reliability comes from the harness, not the prompt — a demo fixed a failing weak model (GPT-3.5) without ever touching the system prompt; cheap/weak model + strong harness beats an expensive model alone, letting you run Qwen/GPT-OSS for real work. [IBM](https://www.youtube.com/watch?v=C_GG5g38vLU) (2026-05)
- Harness dominates coding-eval results: SWE-Bench Pro showed ~22% swing from harness alone while frontier models sat within a few points — one eval tests 3 things at once (model, harness, problem realism), a strong model masks a weak harness, and when a benchmark tests an agent the harness/orchestration is part of the system under test, so you may be measuring the harness, not the model. [Cline](https://www.youtube.com/watch?v=QuuIywMG4s8) [DeepMind](https://www.youtube.com/watch?v=Ubwb6NzegyA) (2026-05)
- Same model performs differently across harnesses (e.g. Claude Code vs others) — the harness must actually leverage the model's strengths. [Cline](https://www.youtube.com/watch?v=QuuIywMG4s8) (2026-06)
- Harness + architecture decisions ≈ 3x the price swing of model selection — optimize the harness BEFORE picking the model. [Notion](https://www.youtube.com/watch?v=wjXowoQ7E8c) (2026-06)

## Minimalism — get out of the model's way
- Each thing you add to an agent risks making it worse — large system prompts, edge cases, fancy if-else degrade frontier models; newer models perform worse with over-instruction (sensory overload, e.g. Codex GPT-5.3's system prompt is ~1/3 the size of GPT-5's), so prune relentlessly. [Cline](https://www.youtube.com/watch?v=yUmS-F9IX90) (2026-05)
- The constraint forcing decomposition: a single mega-prompt with all instructions + ~100 tools would be ideal IF models were perfect, but accuracy degrades as prompt size, instruction count, and tool count grow — so split into focused sub-graphs each with tight context. [iFood](https://podcasters.spotify.com/pod/show/mlops/episodes/The-Control-vs-Magic-Spectrum-Building-Agents-e3k9n0g) (2026-06)
- Keep each agent's context minimal — pass only the few signals that change behavior (e.g. order history + dietary), not everything you have; overloading prompt+context makes agents cumbersome and hallucinate-prone. [JustEat](https://podcasters.spotify.com/pod/show/mlops/episodes/Inside-Just-Eats-AI-Lab-Voice-Agents--Agentic-Commerce-e3jsob9) (2026-05)
- Minimal core (read/write/edit/bash only) + many hook points beats a feature-rich harness; missing features (MCP, plan mode, compaction) get added by asking the agent to extend itself. [Pi](https://newsletter.pragmaticengineer.com/p/building-pi-and-what-makes-self-modifying) (2026-04)
- A minimal primitive tool set already gets strong results on long-running agents: read file, write file, execute code, load skills, write memory, context compaction. [ZenML](https://podcasters.spotify.com/pod/show/mlops/episodes/Agents-are-Just-While-Loops-e3j3a6q) (2026-05) [Anthropic](https://www.youtube.com/watch?v=mR-WAvEPRwE) (2026-05)
- "Bash is all you need" for a general agent: a single `bash` tool returning stdout/stderr/exit code covers most work — models are very good at generating `find`/`ls`/`grep`/`glob` — so you can run a whole agent off one tool instead of many bespoke ones. [Vercel](https://www.youtube.com/watch?v=wflNENRSUb4) (2026-05)
- Follow-on to "bash is all you need": a hybrid bash-tool + DB-tool agent scored *highest* on analytical queries (query the DB, then verify results with the shell) — a pure DB tool wins alone for analytics, file/grep wins for quick lookups; offer both rather than one universal tool. [Elastic](https://www.youtube.com/watch?v=ynJyIKwjonM) (2026-05)
- Fewer tools → better output, taken further: a single `execute` tool (runs arbitrary TS, incl. bash-via-TS) and NO patch tool forces the agent to write a TS file / AST transformer to edit code; shrinking the tool surface improved quality. [Effect](https://www.youtube.com/watch?v=Wmp2Tku2PrI) (2026-05)
- More thinking ≠ better — forcing extra reasoning makes the model loop and can degrade output (tunable levers: container CPU/memory, timeouts, thinking behavior); the "orchestration paradox" is the extreme: the strongest reasoning models loop on choosing the method instead of executing — hop method→method, burn tokens researching how to solve rather than solving. [Cline](https://www.youtube.com/watch?v=QuuIywMG4s8) (2026-06) [Qodo](https://www.youtube.com/watch?v=EcqMYoIV57A) (2026-06)
- Prompt-tweaking edge cases endlessly is a losing game; the meta-fix is a reusable framework the agent runs (e.g. a decision framework). [Neo4j-Kollegger](https://www.youtube.com/watch?v=abvQEhvRI_c) (2026-05)
- Offload deterministic/repeated subtasks to CPU code, not the LLM — "workers" = small hosted code + sandboxes the LLM calls as an action; cut token cost up to 80% on repeated work (CSV→PDF→webhook type). Determinism is a feature; don't token-max what a state machine handles. [Notion](https://www.youtube.com/watch?v=wjXowoQ7E8c) (2026-06)
- Don't fix every recurring mistake in the harness — only when it's constant AND fundamental; forcing one step-by-step path overfits and breaks elsewhere, and wastes time the model could've figured out itself. "Too much harness" signal = you're scripting a path the model could navigate; let the fast feedback loop self-correct in flight instead. [MLOps-Sandboxing](https://podcasters.spotify.com/pod/show/mlops/episodes/Sandboxing--Agent-Harnesses--and-Agent-Teamwork-e3kts75) (2026-06)
- Not everything needs to be an agent: a CI/CD security/conventions review is a simple LLM pass/fail on the diff against an objective bar, no 100-tool-call agent. Cleric's own trajectory: went uber-complex multi-agent → stripped to a single "query agent" (give it the question, it finds where to go + answers); simple won, mirroring Claude Code's simplicity. [MLOps-Sandboxing](https://podcasters.spotify.com/pod/show/mlops/episodes/Sandboxing--Agent-Harnesses--and-Agent-Teamwork-e3kts75) (2026-06)

## Frameworks & API lock-in
- Frameworks (LangChain/LangGraph) fine for PMF/prototyping but hit a customizability/modularity ceiling for production — expect to rewrite from scratch (Cline rewritten fully; "cloud code written 7+ times"). [Cline](https://www.youtube.com/watch?v=yUmS-F9IX90) (2026-05)
- Frontier APIs subtly lock you in: reasoning models need traces in the exact expected format — wrong format still returns a response but silently degrades with no error. Test each lab's API explicitly; OpenRouter abstraction isn't enough. [Cline](https://www.youtube.com/watch?v=yUmS-F9IX90) (2026-05)
- Don't lock the harness/product to one provider — the best frontier model keeps shifting; Notion swaps its default model every 3-4 weeks and forgoes vendor discounts to keep that optionality. [Notion](https://www.youtube.com/watch?v=wjXowoQ7E8c) (2026-06)
- Agents work best against open primitives; an abstracted API you can't get behind is a ceiling — expose raw layers rather than wrap them. [HuggingFace](https://www.youtube.com/watch?v=JomVvNDjGb8) (2026-05)
- MCP fills context and is non-composable (combining two servers forces data through the model); CLI piping lets the model see only the end result and massage freely — "code mode" exposes MCP servers as callable functions to recover composability. [Pi](https://newsletter.pragmaticengineer.com/p/building-pi-and-what-makes-self-modifying) (2026-04)
- Code-mode scale datapoint: entire Cloudflare API (~2,600 endpoints) exposed via one code-mode MCP server in ~1,000 tokens — model writes code against an API surface instead of loading per-endpoint tool definitions. [Cloudflare](https://www.youtube.com/watch?v=SKDJo2CopRs) (2026-06)
- Skills + guardrailed CLI beats MCP for most work; MCP's real value is auth, not tool surface. [DeepMind](https://www.youtube.com/watch?v=7gujZrJ9L5I) (2026-05)
- 2026 reversal of the "just use the CLI" consensus above: with modern harnesses' *native MCP rendering*, native MCP now BEATS a Bash/CLI interface on BOTH accuracy and token efficiency — the win came from harnesses (Claude/Codex) learning to render MCP properly, not from MCP itself (benchmarked GitHub + Linear + a Singapore Bus API × 3 models). Two CLI levers close most of the gap (ablation): self-documentation (discoverable, well-described subcommands) + search over subcommands/tools — add both and a CLI nearly matches native MCP. Why 2025 MCP felt bad: naive harnesses dumped ALL tool specs into context → token waste/context rot, and many servers were watered-down API wrappers prompt-injecting weak models via tool descriptions (task behavior belongs in a skill, not a tool description). MCP's durable edges over CLI: portability (no sandbox setup, lower latency for lightweight tasks); harness owns context-engineering so tools improve as the harness does; and permissioning — CLIs are over-broadly scoped (approve-everything / skip-permissions) whereas MCP's small surface is a choke point for fine-grained policy (Smithery ships a policy DSL). [Smithery](https://www.youtube.com/watch?v=m12vGjfbNlo) (2026-05)
- Ship a first-party product CLI so agents can drive your product — the value is Unix-style cross-product composition (Sentry issue → GitHub PR → Basecamp record); late adopters can skip MCP entirely with CLIs + skills. [DHH](https://newsletter.pragmaticengineer.com/p/dhhs-new-way-of-writing-code) (2026-04)
- Expose a platform's existing guardrails + APIs as agent skills → agent uses the same safe paths as humans (IaC ops become skills); agent = alternative interface, not magic. [Kubernetes](https://newsletter.pragmaticengineer.com/p/kubernetes-and-retiring-at-the-top) (2026-06)
- Migrating to MCP doesn't mean rewriting working connectors: add real MCP for NEW/external connections, but keep a legacy in-house tool-connection protocol (a "MCP before MCP") for existing brain↔agent links — keep-up-to-date without a rewrite. [Despegar](https://podcasters.spotify.com/pod/show/mlops/episodes/Building-MCP-Before-MCP-Existed-Inside-Despegars-Sofia-Agent-e3j1si9) (2026-05)
- ACP (Agent Client Protocol) ≠ MCP, different axis: MCP = model↔tools, ACP = agent↔client (editor/IDE/chat) so you build ONE editor adapter and every harness behind it works, vs N per-harness plugins; third axis = agent↔agent (a client can drive sub-agents over ACP too). [ACP](https://www.youtube.com/watch?v=VaS2h-dY1-4) (2026-05)
- Migration tax retrofitting a working frontier-API agent to self-hosted/on-prem, in order of pain: (1) swap frontier→self-hosted model is NOT drop-in — re-translate API logic, re-tune prompts, re-run the whole eval; (2) data-into-jurisdiction becomes a search-ROUTING problem across multiple DBs (query-classify to one store vs fan-out + merge); (3) on-prem exposes hidden lock-in — you now own K8s + the CPU-app↔GPU-model network the cloud hid; (4) observability becomes mandatory for audit. [deepset](https://www.youtube.com/watch?v=x2bH0RKPgdc) (2026-05)

## MCP Apps — tools that return UI
- MCP Apps = official MCP extension (standardized w/ Anthropic + OpenAI): a tool returns a UI *resource* (HTML) instead of text; host renders it in a sandboxed component. [MCP UI](https://www.youtube.com/watch?v=o-zkvb0iFDQ) (2026-05)
- Message passing standardized so UI events route to the *host*, not a vendor backend → keeps actions in agent context (avoid the UI talking straight to its own backend). Interaction-control spectrum (UI→host): notification → tool call → prompt (increasing control ceded to host). [MCP UI](https://www.youtube.com/watch?v=o-zkvb0iFDQ) (2026-05)
- Generative-UI spectrum: predefined/black-box → declarative (app declares JSON structure, host renders) → fully generative (model produces UI on the fly). In progress: model-drives-the-UI — app exposes tools letting the model click/fill forms inside the view (cf. WebMCP). Reusable views: reference an existing rendered view + push data instead of re-rendering heavy apps (perf). [MCP UI](https://www.youtube.com/watch?v=o-zkvb0iFDQ) (2026-05)

## Long-running / multi-hour agents
- Three failure modes blocking multi-hour runs: context (amnesia across sessions, context rot, "context-sense anxiety" where the model rushes near window end), weak planning (one-shots everything or builds half a feature and stops), bad self-judgment (calls half-baked work done; builds a button with no backend). [Anthropic](https://www.youtube.com/watch?v=mR-WAvEPRwE) (2026-05)
- Long-run harness pattern: an initializer agent expands a vague one-line prompt into persistent artifacts — feature-list.json, progress file, init script, git repo — then a loop in fresh context windows picks ONE unfinished feature, implements, verifies (Playwright/puppeteer), commits, flips its done flag, repeats. [Anthropic](https://www.youtube.com/watch?v=mR-WAvEPRwE) (2026-05)
- Store agent state as JSON not markdown — models readily overwrite markdown files but tend to leave JSON alone; prefer the file system for shared state on long runs rather than the context window, so another agent can grep it and resume where the last left off. [Anthropic](https://www.youtube.com/watch?v=mR-WAvEPRwE) (2026-05)
- Add a high-level planner that decomposes the prompt into sprints but deliberately does NOT spec granular technical details — a granular plan error cascades and magnifies across every sprint over a multi-hour run; keep the planner OUT of the build loop (re-inject the original spec into builder/evaluator as a reference point, don't let it self-edit). [Anthropic](https://www.youtube.com/watch?v=mR-WAvEPRwE) (2026-05)
- Leave breadcrumbs for day/week-later iteration: a timestamped JSON log (tried X, found bug Y, fix Z worked ✓) plus a live high-level docs file (file structure) so a human/Claude Code can resume the long-running output. [Anthropic](https://www.youtube.com/watch?v=mR-WAvEPRwE) (2026-05)
- Anecdote scale: a "build a retro game maker"/DAW prompt ran ~3-6 hours, ~$200, 5-15 rounds; the difference between a dead-on-arrival app and a working one was entirely the scaffolding. [Anthropic](https://www.youtube.com/watch?v=mR-WAvEPRwE) (2026-05)
- Harness ≠ disposable but harness mitigations ARE temporary: each model release exposes gaps you patch in the harness, then later train the gap into the model and strip the patch (co-evolving loop) — treat context-rot mitigations (Ralph-style slicing to stay in the model's "smart zone") as patches and hunt for the release where you can remove them. With Opus 4.6 they dropped fresh-context-window resets AND strict sprint decomposition for a single continuous compacted session (4.6 lost the context-anxiety 4.5 had, holds a 2-hour continuous build), moving evaluator cadence to once at the end — the simplified harness ran ~half the cost. Lesson: the 4.5 harness was right for 4.5; simplify as the frontier moves. [Anthropic](https://www.youtube.com/watch?v=mR-WAvEPRwE) (2026-05)
- Model selection drives harness design — common cost-saver: Opus for planning, Sonnet as the execution workhorse; give each sub-agent its own evals to optimize per model+prompt. [Anthropic](https://www.youtube.com/watch?v=mR-WAvEPRwE) (2026-05)
- Five takeaways for long-running agents: (1) self-evaluation is a trap — use an adversarial evaluator; (2) compaction ≠ coherence, lossy summaries drift; (3) structured handoffs + clean context win; (4) subjective quality IS gradable if you write down your opinion; (5) sit with the model and read traces to know what scaffold to delete vs keep. [Anthropic](https://www.youtube.com/watch?v=mR-WAvEPRwE) (2026-05)
- Online vs offline split: most real systems are far more OFFLINE than online; build systems that ossify/record preferences so work moves offline to run overnight on cheap local models (agents as appliances). [Southbridge](https://www.youtube.com/watch?v=m12vGjfbNlo) (2026-05)
- Coding-agent harness as the universal substrate — not because all work is coding, but the coding loop gets the most RL/deployment pressure + universal primitives (read/write/edit/shell) → becomes the engine layer for agentic work generally (cf. V8/browser). [Southbridge](https://www.youtube.com/watch?v=m12vGjfbNlo) (2026-05)
- Stop pushing single-shot performance for REPEATABLE work — break into small atomic/reusable/composable units ("codons") and chain them; human reasoning about hour-20+ of a run is the real bottleneck, small boxes make long runs tractable. [Southbridge](https://www.youtube.com/watch?v=m12vGjfbNlo) (2026-05)
- Removing things from context is a first-class need most harnesses lack: explicit boundaries that DELETE/archive unneeded context to prevent "worldline rot" ("be a goldfish"); default-add is wrong for long runs. [Southbridge](https://www.youtube.com/watch?v=m12vGjfbNlo) (2026-05)
- Separate components by type — data / prompts / control-flow / code kept as separate as possible (agentic MVC); most failures = a wrong shared abstraction between human and agent, or stale context left in. [Southbridge](https://www.youtube.com/watch?v=m12vGjfbNlo) (2026-05)
- Never needed parallel agents — a single primary agentic thread has too many reliability benefits to give up. [Southbridge](https://www.youtube.com/watch?v=m12vGjfbNlo) (2026-05)
- "Sentinels" (better than hooks): out-of-band LLM calls triggered on event combos from the primary loop; they template their own context and write results to a file → catch laziness/mocking/bad-data-hygiene/shell-errors without bloating evals; define the pattern once, fix in the main thread (e.g. wake every 50 tool calls, summarize, sleep). [Southbridge](https://www.youtube.com/watch?v=m12vGjfbNlo) (2026-05)
- Declarative budgeting across every axis (money/tokens/time/data): express at BUILD time how resources distribute; at RUNTIME the agent knows what it has and solves the gap — declarative survives model/harness churn. [Southbridge](https://www.youtube.com/watch?v=m12vGjfbNlo) (2026-05)
- On very long-horizon runs (>48h) detect context DRIFT — the "you're absolutely right" mimicry of early context creeping in — and hand the task to a fresh agent rather than continuing a degraded session. [AutoHand](https://www.youtube.com/watch?v=wjXowoQ7E8c) (2026-06)

## Self-evolving / evolutionary harnesses
- Don't hand-build a custom harness per company — evolve the harness itself; the two levers to improve an agent are the ENVIRONMENT it runs in and the TOOLS it has (cites single-agent self-improvement 20%→50% on SWE-bench when the agent rewrote its own code/tools). [CodeGraph](https://www.youtube.com/watch?v=m12vGjfbNlo) (2026-05)
- The artifact worth studying is the TRAJECTORY (reasoning traces / why), NOT the end state — that's what improved his meta-harness; he open-sourced trajectories alongside the harness. [CodeGraph](https://www.youtube.com/watch?v=m12vGjfbNlo) (2026-05)
- Faster iteration loops dominate; even implementation language matters — small compile times let agents build/test better tools (writes agent tools in Zig/Rust). [CodeGraph](https://www.youtube.com/watch?v=m12vGjfbNlo) (2026-05)
- Tools built so agents use fewer tokens / get exact context: faster ripgrep, trigram code search, faster env setup/snapshotting than apt/homebrew, an agent-browser over the Chrome/CDP accessibility (A11Y) tree. [CodeGraph](https://www.youtube.com/watch?v=m12vGjfbNlo) (2026-05)

## Durable execution & resume
- Durability = ability to checkpoint state *between turns of the while loop* and resume at the exact failed turn; keep this recovery layer OUT of core harness/business logic so you can iterate on harness quality (compaction, tool-calling RL) independently. [ZenML](https://podcasters.spotify.com/pod/show/mlops/episodes/Agents-are-Just-While-Loops-e3j3a6q) (2026-05)
- Treat the agent runtime as an ML-pipeline workload (bursty pods that spin up/down, store state to S3/object store + DB, reload checkpoint, skip already-done steps), NOT distributed-transaction infra (workers/queues/in-place updates) — agent turns are ~seconds not microseconds, so the transaction "big hammer" adds deployment complexity without payoff for most enterprise agent loads. [ZenML](https://podcasters.spotify.com/pod/show/mlops/episodes/Agents-are-Just-While-Loops-e3j3a6q) (2026-05)
- What durable execution guarantees: recovery from *external* failures (pod killed, network drop, LLM throttling) via exponential-backoff retry + state rehydration. What it does NOT: the program finishing (a bad tool call / garbage LLM output is app logic), or rollback of external side effects — no control-Z on a Salesforce edit or a written Snowflake table unless you build that into your own tools. [ZenML](https://podcasters.spotify.com/pod/show/mlops/episodes/Agents-are-Just-While-Loops-e3j3a6q) (2026-05)
- Human/agent-in-the-loop: don't keep an idle pod waiting — kill the process and, because state is checkpointed, resume it at that state when the human responds (~2s warm-pod spin-up is negligible against multi-day loops). [ZenML](https://podcasters.spotify.com/pod/show/mlops/episodes/Agents-are-Just-While-Loops-e3j3a6q) (2026-05)
- Hardest part of durability isn't the retry loop (trivial in K8s) — it's rehydrating complex state (filesystem snapshot, externally-mutated state) back to the exact break point. [ZenML](https://podcasters.spotify.com/pod/show/mlops/episodes/Agents-are-Just-While-Loops-e3j3a6q) (2026-05)
- Two separable halves of a durable agent: **context** (append-only log of system/user/tool/assistant messages) and **execution state** (the machine — cloned repos, installed pkgs, dev servers, sub-processes); make each durable independently. Context survives via any append-only primitive (DB / object store / distributed FS) across harness upgrades and crashes. [Trigger.dev](https://www.youtube.com/watch?v=svCnShDvgQg) (2026-05)
- Replay/durable-execution (wrap each side-effect as a cached step, re-execute + skip-on-resume) breaks for agents: every LLM + tool call becomes a journal entry so the log grows unbounded — an agent is a *session* (open-ended), not a *transaction* (fixed start/end); replay also forces non-step code to be deterministic and breaks journal versioning on redeploy. [Trigger.dev](https://www.youtube.com/watch?v=svCnShDvgQg) (2026-05)
- Execution state can't be rebuilt from a log, so snapshot-and-restore the whole machine instead — the concrete answer to ZenML's "hardest part is rehydrating state": pause/shut down during idle, restore on the next user message → durability across turns without paying to keep the machine live. Route recovery by failure type: LLM down/rate-limited → snapshot, restore when the retry window opens (don't wait in memory); machine crash / shipped bug → recover from the context log. [Trigger.dev](https://www.youtube.com/watch?v=svCnShDvgQg) (2026-05)
- Snapshot impl: CRIU (process-level) too limited (one process, open files only, slow registry push/pull) → Firecracker microVM snapshots the whole VM; naive snapshot writes full RAM to disk (512MB→512MB), fixed via seekable compression + lazy page restore + layering (→~14MB, a tunable size/perf knob); measured snapshot <1s, restore ~few hundred ms, ~15k VM cold-starts/min (`fcrun` — Docker-like CLI for run/snapshot/fork, to be open-sourced). [Trigger.dev](https://www.youtube.com/watch?v=svCnShDvgQg) (2026-05)

## Event-sourced harness design
- Event-source the *whole* harness, not just durability: every streaming chunk, tool call, and state change is an appended event → "event-sourced = debuggable", with no hidden side-effect state recoverable only from OTEL traces later. [Iterate](https://www.youtube.com/watch?v=vi-2nasppAg) (2026-05)
- Agent state = a synchronous pure reduce over the event log (`(state, event) => newState`); ban side effects inside the reducer and move them to a separate `afterAppend` hook → on restart/wake the reducer catches up over the backlog *without* re-firing N replayed LLM calls; side effects only fire once caught up to head. [Iterate](https://www.youtube.com/watch?v=vi-2nasppAg) (2026-05)
- One event-envelope primitive (`type` + optional `payload` + `streamPath` + auto-incrementing `offset` + `createdAt`) — treat partial-completion stream deltas as just more events, not a special API surface; make the `type` a dereferenceable URL so the type string *is* its own docs link. [Iterate](https://www.youtube.com/watch?v=vi-2nasppAg) (2026-05)
- Circuit-breaker as a processor (runs in prod): the reducer tracks the last-100 event timestamps; 100 events inside ~1s → append a pause event, killing runaway loops / two-plugin ping-pong. Pause/resume are themselves appended events; the paused-stream rejection is the one thing deliberately NOT logged (else infinite error events). [Iterate](https://www.youtube.com/watch?v=vi-2nasppAg) (2026-05)
- Anti-before-hooks stance: blocking before-hooks break prompt caching and tank cost/latency (cites OpenClaw regressions) — only the platform itself may block an append. Replace blocking with an eventual-consistency model: the loop waits up to ~200ms for optional enrichers (RAG context, prompt-injection/safety checkers) before each LLM call, then proceeds regardless → enrichers are best-effort, never block. [Iterate](https://www.youtube.com/watch?v=vi-2nasppAg) (2026-05)
- Reducers are importable/composable: run another harness's (Claude/Pi/OpenCode) event-type reducer inside your own processor for free and build on top of their event types. Sub-agents fall out of a path hierarchy — append to a child path, subscribe to its result events, parent wakes on completion; no separate orchestration layer. [Iterate](https://www.youtube.com/watch?v=vi-2nasppAg) (2026-05)
- Distributed/polyglot processors subscribe to the stream from another server/language; push-subscriptions (server-tracked offsets, not Kafka-style client pull) let the stream wake remote processors — enabling third-party plugins that hook the loop via events (e.g. a paid prompt-injection guard), which MCP can't do today (no proactive loop hook). Ship a processor AS an event (reducer+afterAppend bundled as a string in the payload) → a stream becomes an agent with zero deploy; keep secrets out of the payload by declaring env vars the runtime substitutes into outbound headers at call time. [Iterate](https://www.youtube.com/watch?v=vi-2nasppAg) (2026-05)
- Errors are events too: an invalid append emits an error event; add a TS-compile step that emits compiler-error events and feed them to an LLM to auto-fix and append corrected code. Substrate = durable append-only log (Kafka / Electric SQL durable-streams shape) + pub/sub + code execution. [Iterate](https://www.youtube.com/watch?v=vi-2nasppAg) (2026-05)

## Context, compaction & memory
- Naive-but-workable compaction: always keep system prompt + user task + last N messages, drop the middle. [IBM](https://www.youtube.com/watch?v=C_GG5g38vLU) (2026-05)
- Context-heavy approach is fine despite 20-50 compactions per session: durable artifacts (ADR/PRD/BDD docs) survive compaction because the agent re-looks-them-up; enables multi-hour autonomous sessions without fear of context loss. [SafeIntelligence](https://www.youtube.com/watch?v=504PvfXou5Y) (2026-06)
- Knowledge/context bank rots fast — stale on every model release and feature change; aggressively prune, don't just append, or old entries cause context rot and degrade the live agent. Cluster solved-problem solutions before storing so entries generalize, not one page per exact prompt. [Lovable](https://www.youtube.com/watch?v=KA5kPbdkK2E) (2026-06)
- Two-tier injection: a cheap lightweight model decides when to inject relevant context into the main agent; reserve frontier intelligence for inline cheap calls, not an always-on external reviewer over long context. [Lovable](https://www.youtube.com/watch?v=KA5kPbdkK2E) (2026-06)
- Split agent memory into 3 layers: short-term (session), long-term (deduped resolved entities), reasoning (decision traces) — each needs different storage/retrieval; agents won't persist decisions/learnings unless explicitly prompted to store them (reusable traces need deliberate write steps, not implicit memory). [Neo4j-Blumenfeld](https://www.youtube.com/watch?v=B9h9ovW5H9U) (2026-05)
- Bigger context windows don't fix retrieval — agents can't reason over a full 1M window (only needle-in-haystack); you still need entities/relationships, not a raw dump. [Unblocked](https://www.youtube.com/watch?v=BiG2ssibKGc) (2026-05)
- Pipeline stages should collaborate via a shared file system / workspace instead of passing huge text blobs (cheaper, frees context, mirrors how humans co-research). [DeepMind](https://www.youtube.com/watch?v=7gujZrJ9L5I) (2026-05)
- Git-backed file-system memory as concurrency control: one repo per project, one branch per thread; pull at sandbox start, push after every bash command → state survives the ephemeral sandbox; merge branches to share memory across threads, with a background workflow auto-resolving merge conflicts when two agents edit the same item — Postgres-like transactional semantics, but agents handle files better than a DB interface. [Sazabi](https://podcasters.spotify.com/pod/show/mlops/episodes/Logs-Are-All-You-Need-Rethinking-Observability-with-AI-Agents-e3jthog) (2026-06)
- Back session/shared file systems with object storage (e.g. GCS via GCS-FUSE): the agent's home dir is object-mounted, so the same fs shows in the UI to humans AND mounts to the agent, and you get free versioning/undo when an agent clobbers shared state. [Dust](https://podcasters.spotify.com/pod/show/mlops/episodes/MCP--Agents--the-40M-Bet-on-Multiplayer-AI-e3kmopj) (2026-06)
- Context dies when an agent's runtime dies — use long-running runtimes that persist until the whole issue/session closes so context isn't thrown away; cross-vendor context transfer needs a richer structured-context standard (agreed metadata/field schema), MCP's I/O is too thin for it. [Guild-Meta](https://podcasters.spotify.com/pod/show/mlops/episodes/From-Single-Player-to-Multi-Player-Operating-AI-Agents-at-Scale-e3khpk8) (2026-06)
- Plan-file scratchpad fixes long-context follow-through (Vercel's internal D0 data agent): give the agent a filesystem + a written plan file ("objective at top, follow this plan to a T, check items off as you go") + a research dir → it re-reads the plan each step so the objective isn't buried as context grows (stays on track), and you get a final artifact of exactly what was done; turned semi-hallucinated one-shot answers into completed multi-step tasks. [Vercel](https://www.youtube.com/watch?v=wflNENRSUb4) (2026-05)
- Memory as plain files in the sandbox: keep a `core_memories.md` injected into the system prompt every turn (deterministic non-LLM code fetches + injects it in a once-per-run `prepare` hook), and structure other memory as files (`conversations.jsonl` for searchable history) — the filesystem + standard bash tools become the memory store. [Vercel](https://www.youtube.com/watch?v=wflNENRSUb4) (2026-05)
- At million-token windows compaction is far less necessary than at ~400k (one coding-agent turn ran 104 min / 316 tool calls / 29 files at only 32% of a 1M window, zero compaction). Naive context-pruning actively hurts: trimming early tool calls to stay in a 40–60% token band invalidates the input cache on every trim — a big cost/latency hit when you run ~95% cache-read. Prefer NOT trimming if the window is large enough. [Vercel](https://www.youtube.com/watch?v=wflNENRSUb4) (2026-05)
- Prefer offloading over compaction: (1) sub-agents take semi-independent work off the main thread and return only a ~1k-token summary (one used 30k tokens, returned ~500); (2) the amp handoff tool — at a chosen point the agent generates the context it wants to carry into a fresh thread, which becomes the new main thread. Both beat lossy LLM-summarization. [Vercel](https://www.youtube.com/watch?v=wflNENRSUb4) (2026-05)
- Recursive Language Models (RLM) = the full-Python version of "offload, don't compact": pass context by *reference* not value — store the 10k docs / long tool output as a variable/file and have the orchestrator write a `for` loop of sub-queries over it, rather than dumping it in the window. Rationale matches the 1M-window finding (MRCR/GraphWalks degrade with length, compaction doesn't reliably recover — once it compacts mid-task "it's over"). [PrimeIntellect](https://www.youtube.com/watch?v=_xQnSNlBP_w) (2026-05) (low-confidence src: RLMs untrained-on-scaffold underperform today; the win is the prompt-engineering strategy or training it into the model.)
- Auto-compaction is dangerously lossy on the instruction you can least afford to lose: an inefficient tool dumped a huge email list into context → triggered auto-compaction → the user's original "don't delete" instruction got overwhelmed in the summary → the agent deleted the whole inbox and refused to stop ("you asked me to"). If you do compaction surgery, do it in the `prepare`-step hook as a deliberate, developer-owned pure function (full messages in, modified params out) — not an opaque default. [Vercel](https://www.youtube.com/watch?v=wflNENRSUb4) (2026-05)
- Smart truncation + an off-context memory store beats both naive truncation and summarization (Arize's year on an agent-on-agent-data product): naive truncation broke reasoning (follow-ups read as brand-new conversations); summarization was too inconsistent (no control over what's deemed important). What worked: keep head + tail, truncate the middle, dedupe repeated messages, keep only the latest tool-call result, never reset the system prompt — and stash the dropped middle in a DB keyed by IDs. The agent gets a tool listing the IDs + where each sits in the conversation + a short preview, and fetches specific items back only on demand. "Context decides what the model sees; memory decides what survives." Convergent with Claude Code's leaked truncation+compression approach; open problem flagged = avoiding prompt-cache invalidation during the surgery (cf. Vercel above). [Arize-context](https://www.youtube.com/watch?v=esY99nYXxR4) (2026-05)
- Memory-file layering for rule adherence: a separate `critical-rules.md` referenced HIGH in agents.md (soul.md/agents.md alone still dropped rules); evolve single file → memory folder as it grows and prune noisy notes ("bad memory compounds" past thousands of notes). Plain-markdown memory/rules = readable/editable/diffable → makes the iterate-on-memory loop tractable. [Keys to my life](https://www.youtube.com/watch?v=sJ2jc7leKBk) (2026-05)
- Multi-user agent memory scales ~100x worse than single-user: a single-user (OpenClaw-style) memory architecture clutters/exhausts far faster once shared across a whole company — treat memory scaling as a first-class problem when moving a personal agent to a team agent. Pair it with per-channel/hierarchy context isolation so one agent present in many channels doesn't leak growth/exec context into engineering/support; scope access by who is asking and where. [Viktor](https://www.youtube.com/watch?v=ohKt066uFhg) (2026-05)
- Connectors give ACCESS, not UNDERSTANDING: agents trust whatever doc they read first, miss corrections, conflate entities (5 "Lisa"s vs 1), lack recency; "source of truth" docs are a lagging indicator, stale when written. Humans go ask a person; agents don't. The hard part is synthesis (resolving conflicting facts, bringing humans in to choose), not search. [Hyperspell](https://www.youtube.com/watch?v=m12vGjfbNlo) (2026-05)
- Best representation of org context for agents is a FILE SYSTEM (company → team → individual files) — universal across Claude Code/Cursor/OpenClaw; context/graph DBs are bad agent UX (not post-trained on them). (Reinforces "store state as files" above.) [Hyperspell](https://www.youtube.com/watch?v=m12vGjfbNlo) (2026-05)
- Context is no longer only human-generated — agent traces, prompts, memory, reasoning loops are all context that should feed the store; soon most org context is agent-generated. [Hyperspell](https://www.youtube.com/watch?v=m12vGjfbNlo) (2026-05)

## Guardrails as execution boundary (not behavior prompts)
- "Stop asking the actor to behave; change what the actor can do." A deterministic OS-level execution boundary, near-zero overhead (no daemon/container) — one policy file enforcing filesystem, network (forced through a local filtering proxy, allow-listed domains), and command policies (checked before exec, incl. chained/nested shells); same policy vocabulary regardless of which agent drives. [Tusk](https://www.youtube.com/watch?v=m12vGjfbNlo) (2026-05)
- Swiss-cheese / defense-in-depth, 3 layers: (1) classification "is this reasonable?" probabilistic (e.g. Claude Code auto-mode); (2) policy enforcement "is this allowed?" deterministic (fence sits here); (3) isolation "what can it touch if it goes wrong?" (containers/micro-VMs). Most teams have (1) and/or (3); the missing MIDDLE layer is the OS-level allow/deny boundary. [Tusk](https://www.youtube.com/watch?v=m12vGjfbNlo) (2026-05)
- Counterpoint — to MAXIMIZE agency, hand the risk to the user instead of sandboxing: a local file-based desktop app (Electron) opens an HTTP endpoint that executes whatever JS is POSTed, giving the agent full runtime/DOM API access; safe only because it's offline/file-based ("you can only hurt yourself", OpenClaw-style sharp tools). Sandboxing was rejected here precisely because it limits agency — the agent needs the live DOM/browser to see state + take screenshots, which a sandbox would cut off. [tldraw](https://www.youtube.com/watch?v=sPUjIBH5Cwg) (2026-05)

## Retrieval & comprehension
- Agentic search (let agent grep/read the filesystem) beat AST indexing + dense/sparse RAG — the indexing approaches "just went away". [Pi](https://newsletter.pragmaticengineer.com/p/building-pi-and-what-makes-self-modifying) (2026-04)
- Clone a dependency's source repo INTO your project (git subtree, squashed, e.g. `.repos/`) so the agent reads it as first-class code: coding models are post-trained to focus on your own repo, so they de-prioritize the same library in `node_modules` and won't read gitignored dirs at all — beats MCP-docs or the model's stale training, across languages. (Add a test-runner ignore for the cloned dir, and forbid watch/dev-server commands in AGENTS.md or the agent runs them first and hangs.) [Effect](https://www.youtube.com/watch?v=Wmp2Tku2PrI) (2026-05)
- Render debug UIs (traces, agent/tool hierarchies) as downloadable file systems and drop them into a sandboxed Claude Code instead of MCP/UI — agents excel at file-system traversal with standard tools, traces translate cleanly to ASCII ("not much limit to what you can put into ASCII"), and bulk-downloading all context beats per-call MCP queries ("wouldn't have been half as effective"). A self-documenting dump + the powering code base lets the agent localize the bug and propose the diff in-session. [incident.io](https://www.youtube.com/watch?v=L2r6vLlLgs8) (2026-05)
- Bulk-analysis fan-out pipeline ("Scrapbook"): download N investigations as files → fan out ~25 sub-agents, one per entity, for parallel analysis → cohort-clustering stage finds shared failure modes → rolled-up report explaining *why* the system performs well/badly + suggested fix; each sub-agent writes its analysis back to files in the download dir, making the pipeline incremental and resumable (start/stop). [incident.io](https://www.youtube.com/watch?v=L2r6vLlLgs8) (2026-05)
- Counterpoint with production data: a semantic-search tool measurably lifts coding agents — Cursor measured +12.5% avg answer accuracy across models on an internal context benchmark (~24% for Composer); online A/B +2.6% code retention in large codebases, −2.2% dissatisfied requests; headline numbers look small only because the tool fires on a fraction of queries. [Turbopuffer](https://www.youtube.com/watch?v=UM6sFg_jdlE) (2026-06)
- Reconciliation (open question): grep-only vs semantic index is likely scale-dependent — Pi (solo/medium repos) dropped indexing; Cursor measures real gains on large codebases. Steelman for the index: embeddings as cached compute — per-session grep→read→assess loops re-derive the same codebase understanding across devs/days (~6k tokens per sub-step, repeated endlessly), a one-time index turns that rediscovery into a cheap runtime query. [Pi](https://newsletter.pragmaticengineer.com/p/building-pi-and-what-makes-self-modifying) [Turbopuffer](https://www.youtube.com/watch?v=UM6sFg_jdlE) (2026-06)
- Amortize team indexing via Merkle trees (Cursor): hash-compare teammates' checkouts of the same repo, copy the existing index, re-chunk/re-embed only changed files instead of re-indexing per developer. [Turbopuffer](https://www.youtube.com/watch?v=UM6sFg_jdlE) (2026-06)
- Bolted-on semantic search lifts precision, not recall (vendor-run, 50 tasks): Claude Code baseline 65% file precision (1 in 3 reads wasted) → windowed grep 1-in-5 → +semantic search 87% (1-in-8); recall ~unchanged because exhaustive exploration already wins file recall while reading lots of wrong things. [Turbopuffer-CC](https://www.youtube.com/watch?v=zKk7sDMGDEQ) (2026-06)
- Bolt-on tool ≪ native tool: Claude Code is built/trained around grep and doesn't know when/why to call an extra semantic-search tool, so gains run far below Cursor's (Composer treats it as a built-in) — a retrieval tool's value depends on the model being trained to invoke it. [Turbopuffer-CC](https://www.youtube.com/watch?v=zKk7sDMGDEQ) (2026-06)
- Per-task tool split: semantic search wins when relevant files are behavior-adjacent without shared keywords (same concern across different ORMs/libraries); grep wins on import-tracing/keyword-anchored tasks — offer both, they find different code. [Turbopuffer-CC](https://www.youtube.com/watch?v=zKk7sDMGDEQ) (2026-06)
- Inline comments are the main lever for code-embedding quality: embeddings perform markedly better in well-commented repos (the comment gives the chunk its semantic meaning); for uncommented code, generate synthetic comments and embed code+comments, or parent-child chunks (query-ish parent, raw code child) — reportedly Cursor's approach. [Turbopuffer-CC](https://www.youtube.com/watch?v=zKk7sDMGDEQ) (2026-06)
- Off-the-shelf embeddings cluster heterogeneous signals by structure, not meaning — errors group with errors, Slack with Slack, never error↔Slack about the same bug; fix: LLM generates "what is this about?" queries per signal and match the queries in embedding space, not the raw signals. [PostHog](https://www.youtube.com/watch?v=zMiSRliEzv4) (2026-06)
- Grep is not semantic search (common identifiers break rename/find-all-refs): repackage LSP/language services as agent-accessible CLI-shaped tools — e.g. a server keeping the project hot to answer semantic queries, dropped when the agent goes quiet; in-flight semantic validation of code as it's generated will grow in importance. [Hejlsberg](https://newsletter.pragmaticengineer.com/p/typescript-c-and-turbo-pascal-with) (2026-05)
- Context-engine indexing has a scaling cliff: ~600–700 repos → mapping/indexing slows, results unpredictable — only worth building if context engines are your actual product. [Qodo](https://www.youtube.com/watch?v=EcqMYoIV57A) (2026-06)
- Retrieval-strategy decision matrix by cost: hierarchical summarization (per-file/folder summaries; high ongoing LLM cost — re-summarize on every file change) vs knowledge graph (very high setup; wins only with cross-file/cross-repo logical dependencies) vs iterative retrieval (lightweight "library card" index the agent drills into on match; low dev input, higher token cost — best default when agents aren't your product). [Qodo](https://www.youtube.com/watch?v=EcqMYoIV57A) (2026-06)
- "Satisfaction of search": agents stop at first plausible match (like radiologists), so retrieval must force exhaustive search or it grabs the wrong pattern → correction doom-loop. [Unblocked](https://www.youtube.com/watch?v=BiG2ssibKGc) (2026-05)
- MCP access ≠ understanding — pipes to data give reach but no reasoning across sources; an agent can pass all code checks and still ship something that breaks the system. [Unblocked](https://www.youtube.com/watch?v=BiG2ssibKGc) (2026-05)
- Front-load a "research packet" before planning (patterns, conventions, factory choices) — better upfront context makes every downstream grep/explore call more accurate and token-cheap; the workflow split is use the context source for planning, run execution, then re-invoke at code-review (planning and review benefit most). [Unblocked](https://www.youtube.com/watch?v=BiG2ssibKGc) (2026-05)
- Background/headless agents need a machine to query for context — otherwise they block on "am I allowed to use this tool?" instead of producing a PR. [Unblocked](https://www.youtube.com/watch?v=BiG2ssibKGc) (2026-05)
- Naive RAG-over-docs → search thrashing → token burn → compaction before answering; up-front high-signal context = "waterfall" that avoids doom loops where each wrong output forces a human round. Output tokens (not input) dominate agent latency → keep the output loop tight. [Mergeable](https://www.youtube.com/watch?v=5ID22ACI7IM) (2026-05)
- Conflict resolution: recency alone is wrong — bias to code/main branch, but intended direction often beats current state; surface unresolved conflicts to a human rather than silently auto-resolving. Anti-pattern: never cache/re-serve prior context answers — feeding answers back → regression-to-mean / pollution. [Mergeable](https://www.youtube.com/watch?v=5ID22ACI7IM) (2026-05)
- Personalized retrieval: infer focus repos from PR counts → deep retrieval there, shallow elsewhere. "Bottling the expert": distill a person's PRs/comments/decisions and load it for newcomers in their code area as a retrieval pivot. [Mergeable](https://www.youtube.com/watch?v=5ID22ACI7IM) (2026-05)
- Layered context stack: vector search → pre-built memories → expert context; seed context must be supplied, not agent-hydrated. graph-rag crosses permission boundaries → compartmentalize per-repo, tag with group IDs. [Mergeable](https://www.youtube.com/watch?v=5ID22ACI7IM) (2026-05)
- Retrieve similar past decisions via hybrid semantic AND structural similarity (embed the whole reasoning subgraph, not just text) — surfaces precedents plain vector search misses. [Neo4j-Blumenfeld](https://www.youtube.com/watch?v=B9h9ovW5H9U) (2026-05)
- Insert an "understand the research" step between research and planning in research→plan→implement — read what the agent found so you can steer before it commits to a wrong direction. [Sentry](https://www.youtube.com/watch?v=li0SaBt9RDM) (2026-05)
- In a large mature codebase the dominant agent use is comprehension, not generation (measured 67% comprehension / 2% code gen) — instrument your own usage to find where leverage actually is. [Sentry](https://www.youtube.com/watch?v=li0SaBt9RDM) (2026-05)

## Tool output handling
- Agents get ingenious with bash output when the harness truncates: show first N lines + file size/path, agent then decides to grep instead of slurping 20MB. [Pi](https://newsletter.pragmaticengineer.com/p/building-pi-and-what-makes-self-modifying) (2026-04)
- Injecting LSP diagnostics after every edit call confuses the model — it edits line-by-line and isn't done yet; batch diagnostics like a human reads errors after editing, not per-edit. [Pi](https://newsletter.pragmaticengineer.com/p/building-pi-and-what-makes-self-modifying) (2026-04)
- Tool outputs as semantic summaries, not raw artifacts — a 50k-line JSON perf trace blows any context; return a markdown key-metrics summary by default, keep the raw file on disk for CLI post-processing. [Chrome DevTools](https://www.youtube.com/watch?v=_B4Pv9ttFgY) (2026-06)
- Write tool error messages for agent self-healing — append the actionable cause ("history entry to navigate was not found"); every unrecovered error = retry tokens. [Chrome DevTools](https://www.youtube.com/watch?v=_B4Pv9ttFgY) (2026-06)
- Errors as steering, not just reporting: on a bad call return a constructive error that names the likely fix ("you forgot `last_name`, retry" / "maybe you meant tool X, Y, Z") — treat the agent like a human filling a form. Customers running this report large success-rate gains at near-zero overhead; it makes agents recoverable mid-flight. [MCPcat](https://podcasters.spotify.com/pod/show/mlops/episodes/MCP-Servers-Are-Becoming-the-UI-for-AI-Agents-e3ks6n3) (2026-06)
- Add a `token_limit` parameter to any tool that can return unbounded output (log queries, "search all users") so the LLM declares how many tokens it wants back — a broad query then can't dump thousands of results and blow the window; map it deliberately NOT one-to-one to the underlying API. Big cost + reliability win on "unruly" calls. [MCPcat](https://podcasters.spotify.com/pod/show/mlops/episodes/MCP-Servers-Are-Becoming-the-UI-for-AI-Agents-e3ks6n3) (2026-06)
- Run aggregation/counting *inside* the query/search tool instead of returning rows for the LLM to tally — avoids both miscounting and dumping N rows into context. Treat "zero results" as an explicit designed answer (valid vs failure), not a silent pass-through. [Elastic](https://www.youtube.com/watch?v=ynJyIKwjonM) (2026-05)
- Enrich traces for later debugging: enable GenAI tracing + message-content capture, then inject your own custom span attributes so you can grep traces for the right thing when a bug surfaces. [Foundry](https://www.youtube.com/watch?v=iOXM3zE-2dk) (2026-05) (low-confidence src)

## Production failure modes (LLM/VLM-in-the-loop)
- Output loops: ~0.5–1% of queries return repeated output; worst = whitespace loop (infinite spaces burns the full budget). Can't use space as a stop sequence (tokenizers have 1..128-space tokens; providers reject a space stop-token). Fix: always stream + repetition-detection to kill the stream early, then retry w/ different model/prompt/temperature. [LlamaIndex](https://www.youtube.com/watch?v=m12vGjfbNlo) (2026-05)
- Thinking/reasoning loops are now harder to catch — providers stopped streaming the thinking trace → stuck capping with max_tokens (blunt). [LlamaIndex](https://www.youtube.com/watch?v=m12vGjfbNlo) (2026-05)
- Model blindness (attention-architecture limit, not promptable): the SAME string repeated twice in a chunk → content BETWEEN the duplicates gets ignored; varies by model, switch models. [LlamaIndex](https://www.youtube.com/watch?v=m12vGjfbNlo) (2026-05)
- Color blindness in vision models (esp. red; tokenization artifact) — profile differs per model, test each; detect via OCR-first + signal fusion. [LlamaIndex](https://www.youtube.com/watch?v=m12vGjfbNlo) (2026-05)
- Empty-input hallucination: templated prompt + empty/failed tool result → model invents content rather than reporting empty (Anthropic models love fake "Delaware incorporation documents"); guard by checking for blank input before the call. [LlamaIndex](https://www.youtube.com/watch?v=m12vGjfbNlo) (2026-05)
- Provider reliability: APIs go down ~daily → multi-provider/multi-family fallback, each model family its OWN code path (fail over Anthropic→Gemini live); always keep a NON-LLM fallback (LlamaIndex LightParse ~500 pages/s on CPU). [LlamaIndex](https://www.youtube.com/watch?v=m12vGjfbNlo) (2026-05)

## Tool design for agents
- Tool-description craft: ~97% of MCP tool descriptions have quality smells; embed the domain keywords agents match intent against (LCP/INP/CLS); richer descriptions cost context and bias small models to over-call — "minimum viable description" is an endless re-tune as models change. [Chrome DevTools](https://www.youtube.com/watch?v=_B4Pv9ttFgY) (2026-06)
- "Proactive detours" in tool descriptions override training-data priors — steer the agent to your tool when its memorized habit (e.g. reaching for Lighthouse) is wrong. [Chrome DevTools](https://www.youtube.com/watch?v=_B4Pv9ttFgY) (2026-06)
- Decomposing one monolithic tool into 25 traded a capability problem for a discoverability problem; same with piling on skills — trade-offs shift, they don't disappear (slim mode: 3 tools = tiny context but extra turns). [Chrome DevTools](https://www.youtube.com/watch?v=_B4Pv9ttFgY) (2026-06)
- Dynamic tool search to keep tool defs out of context: with 100s of tools, don't dump all definitions in the prompt — put them in a searchable toolset retrieved via BM25 at call time; also scope which MCP-server tools a given agent sees (name-list a subset) rather than exposing the whole server. [deepset](https://www.youtube.com/watch?v=x2bH0RKPgdc) (2026-05)
- Fan out parallel tool calls per request and cache stable inputs — one request fired 3 concurrent tools; never-changing data (partner/menu) is cached → speed + accuracy. [JustEat](https://podcasters.spotify.com/pod/show/mlops/episodes/Inside-Just-Eats-AI-Lab-Voice-Agents--Agentic-Commerce-e3jsob9) (2026-05)
- Tool descriptions/examples must be specific or routing breaks: adding a tool with vague/underspecified use-case examples made the router over-route ("all the prompts" funneled to the new tool the next day). [Despegar](https://podcasters.spotify.com/pod/show/mlops/episodes/Building-MCP-Before-MCP-Existed-Inside-Despegars-Sofia-Agent-e3j1si9) (2026-05)
- Curate the tool set with a "low floor, high ceiling": specialized tools (simple params, hard to misuse, one-shot, work even with a weak model — e.g. lookup-by-ID) for known query patterns PLUS a general-purpose escape hatch (shell / execute-query) for the unexpected — no single silver-bullet tool. [Elastic](https://www.youtube.com/watch?v=ynJyIKwjonM) (2026-05)
- Tool-misrouting fix ladder: write a *real* description (core purpose → trigger conditions "use when / don't use when" → relationships "call skill X first / get confirmation") → if still wrong, reinforce the same instruction in the system prompt; one-line descriptions are the usual cause. Bootstrap unknown query shapes by shipping ONE general-purpose tool and logging — >4–5 tool calls per question signals it's too hard → carve out a specialized tool for that pattern (a stronger model sharply cuts param-error rate but never to zero). [Elastic](https://www.youtube.com/watch?v=ynJyIKwjonM) (2026-05)
- Derive composite tools from your existing UI, not your API surface: a product page already orchestrates ~10–20 API calls to serve one human need — expose that as ONE intent-shaped tool (e.g. a `review` tool) that runs them behind the scenes, since agents want roughly the same outcome the UI was built for; gather any missing inputs via the call's params. Antidote to one-to-one API→tool sprawl. [MCPcat](https://podcasters.spotify.com/pod/show/mlops/episodes/MCP-Servers-Are-Becoming-the-UI-for-AI-Agents-e3ks6n3) (2026-06)
- Ship a "report what's missing" meta-tool on every server: one extra tool the agent calls when it can't accomplish the user's goal with the current tool set — routing the gap to the dev team as a structured feature signal ("user wanted X, no tool for it") instead of a support ticket. [MCPcat](https://podcasters.spotify.com/pod/show/mlops/episodes/MCP-Servers-Are-Becoming-the-UI-for-AI-Agents-e3ks6n3) (2026-06)

## Focus modes
- Constrained "focus modes" (plan/research/debug) shrink the action+input space so you can drop irrelevant tools, sharpen the system prompt, and tune evals per-mode instead of optimizing one do-anything agent; debug mode in particular is hypothesis-driven — spin up a dedicated debug server and push logs there rather than reasoning blind. [Flinn](https://www.youtube.com/watch?v=7CrPrHgoEYk) (2026-05)
- Same generic loop (work → push → feedback → iterate) but swap skills to change the loop's focus: ADR skill, PRD skill, UI skill (skips checks, forces fast browser iteration), test skill (selects tests by coverage + changed files). [SafeIntelligence](https://www.youtube.com/watch?v=504PvfXou5Y) (2026-06)

## Generator / evaluator (adversarial critic)
- Generator/evaluator (GAN-style) split beats one session checking its own work: separate context windows, system prompts, and roles, with adversarial pressure between build and grade. Why a separate critic works even though it's also an LLM — tuning a standalone critic to be harsh is tractable; tuning a builder to be self-critical is not; exploit the critic-vs-generator capability gap. [Anthropic](https://www.youtube.com/watch?v=mR-WAvEPRwE) (2026-05)
- Don't feed the generator's traces/code-reasoning into the critic — muddies the two model streams. The critic judges output only and reports "this is an issue"; let the generator reflect and find the fix itself. [Anthropic](https://www.youtube.com/watch?v=mR-WAvEPRwE) (2026-05)
- Evaluator must actually USE the app (Playwright: open live pages, click, screenshot), not just read diffs — catches live-only bugs (FastAPI route ordering, boolean delete-key bug) that pass unit tests / CI in a Ralph loop. [Anthropic](https://www.youtube.com/watch?v=mR-WAvEPRwE) (2026-05)
- "Negotiate done" before building: generator proposes feature + tests, evaluator pushes back on scope/weak tests/missed edge cases, they iterate via markdown files on disk until both agree, THEN build — and the evaluator grades against this negotiated contract, not the planner's original spec. Granular contracts matter: one app produced 27 criteria; vague criteria → vague critiques → generator shrugs; granular criteria → agent fixes the exact line. [Anthropic](https://www.youtube.com/watch?v=mR-WAvEPRwE) (2026-05)
- GAN-style loop course-corrects by throwing everything out and restarting from scratch when stuck on a criterion — a single-pass/Ralph loop just keeps patching the same broken thing; 4.6-line models readily discarded 10 passes and restarted, and did NOT exhibit the feared "evaluator gives up and rubber-stamps." [Anthropic](https://www.youtube.com/watch?v=mR-WAvEPRwE) (2026-05)
- Pattern generalizes beyond builder+evaluator: chain generator/evaluator pairs across a multi-agent workflow (synthetic-data generator + QA, then integrator + QA, then final step), each step with its own critic — a subset of agent-teams sub-agent design (front-end/back-end/integrator sub-agents each paired with their own critic), not a rival to it. [Anthropic](https://www.youtube.com/watch?v=mR-WAvEPRwE) (2026-05)
- Out of the box Claude is a bad QA agent — sycophancy/generosity bias makes it find a bug then say "fix later, might take two weeks" and move on; the evaluator needs heavy prompt tuning toward harshness. [Anthropic](https://www.youtube.com/watch?v=mR-WAvEPRwE) (2026-05)
- Adversarial critic runs in production, not just in build loops: an "investigation auditor" sub-agent whose whole job is to be Debbie Downer — tell the main agent it hasn't done enough and *why* it screwed up — with the main agent allowed to push back ("done my best, returning to user"). The push-back valve keeps a too-harsh critic from blocking forever. [MLOps-Sandboxing](https://podcasters.spotify.com/pod/show/mlops/episodes/Sandboxing--Agent-Harnesses--and-Agent-Teamwork-e3kts75) (2026-06)

## Verification & anti-cheating
- Deterministic verify step reads the run's tool/trace history to confirm the action actually happened — catches agents that claim success without verifying ("it lies"). [IBM](https://www.youtube.com/watch?v=C_GG5g38vLU) (2026-05)
- Agents fabricate rather than report tool failure: a blocked / captcha'd / empty-page fetch gets presented as "I searched the web", and a failed fetch silently falls back to stale training data answered as current (2024 data given for a 2026 question) — please-the-user bias makes the failure silent and wrong-but-confident. Verify tool *results*, never trust "I did X"; treat agent-supplied citations/URLs as unverified until fetched (broken/404 links and fabricated product URLs are common). [Bright Data](https://www.youtube.com/watch?v=btxGmN8RvNU) (2026-06)
- Beyond missing data, tool results can be *poisoned*: Cloudflare "AI Labyrinth" detects bots and feeds them fake data instead of blocking → larger, harder-to-detect hallucinations than an empty page (~20% of the web blocks AI crawlers by default). Untrusted tool output is an adversarial surface, not just an unreliable one. [Bright Data](https://www.youtube.com/watch?v=btxGmN8RvNU) (2026-06)
- Agents lie about doing work (touched a sentinel file claiming tests ran); require cryptographic proof — SHA-256 the actual test output into the marker file. Make doing the real work easier than faking it — that, not polite instruction, stops the lying. [WorkOS](https://www.youtube.com/watch?v=vy7o1g2iHY8) (2026-05)
- Require non-code evidence before reviewing code: Playwright before/after video of a UI fix attached to the PR; don't read the diff until it proves the work. [WorkOS](https://www.youtube.com/watch?v=vy7o1g2iHY8) (2026-05)
- Red-green TDD with agents: agent writes a failing Playwright test (red) → agent writes minimal code to pass (green) → AI makes red+green fast, so the human's value concentrates in the refactor phase. Connect the agent to Playwright three ways — MCP server, CLI, or Playwright Agents (which installs three agent.md files: planner picks tests, generator writes them, healer fixes failing ones; prefer it for heavy state-management/SaaS admin panels since its agent.md ships specialized state-handling instructions, or skip the browser and test APIs directly). Headless mode runs verification in the background. [Playwright](https://www.youtube.com/watch?v=FWEInOtngmM) (2026-05)
- Commit code BEFORE letting the agent fix/modify tests — uncommitted prior state may be lost from the agent's context. [Playwright](https://www.youtube.com/watch?v=FWEInOtngmM) (2026-05)
- Ship verification inside the skill: bundle benchmark/test scripts so the agent can self-check its output (e.g. kernel correctness + speedup) instead of trusting it. [HuggingFace](https://www.youtube.com/watch?v=JomVvNDjGb8) (2026-05)
- Run static analysis inside the agent loop via MCP, pre-commit (seconds) instead of CI (minutes); the remediation loop must re-run analysis AND compilation on the agent's own fix and discard the change if it regresses — never accept a "fix" that introduces new failures. [Sonar](https://www.youtube.com/watch?v=NuePCNMpWGc) (2026-05)
- Autonomous agent loops need a verifiable experiment (training run, kernel benchmark) as the fitness signal — without one, the self-improvement loop has nothing to optimize against. [HuggingFace](https://www.youtube.com/watch?v=JomVvNDjGb8) (2026-05)
- Verify with a REAL-life integration test, not just unit tests: NanoClaw's factory spins a fresh VM, checks out the PR branch, a test agent SSHes in, runs the built instance and pokes it via Telegram, then inspects DBs + logs to confirm behavior end-to-end. Self-improving supervisor: tag a "supervisor" agent with feedback → it proposes edits to instructions/skills → applied on approval. Merges always use the *approving human's* credentials, never the agent's. [NanoClaw](https://www.youtube.com/watch?v=_xQnSNlBP_w) (2026-05)

## Enforcement loop — pin to the action the agent must take
- The harness = the enforcement loop: git hooks + CI + linters running identical tasks both pre-commit and on CI, so a lazy agent skipping local checks gets caught downstream. The agent's terminal goal is "deliver a PR" — pin enforcement to git (the action it must take) so checks are unavoidable, not optional. [SafeIntelligence](https://www.youtube.com/watch?v=504PvfXou5Y) (2026-06)
- On failed checks, link the agent back to the decision doc (ADR/PRD) rather than just erroring — it reads the rationale and self-corrects in the loop. [SafeIntelligence](https://www.youtube.com/watch?v=504PvfXou5Y) (2026-06)
- Deterministic instructions > prompt-nagging AT SCALE: writing "PLEASE run tests before commit" in screaming-caps AGENTS.md is a code smell — make it deterministic (hooks/CI gates), since arguing with the agent over security/process steps is unreliable across thousands of runs; reserve screaming-caps context files for solo workflows. [Stripe](https://www.youtube.com/watch?v=_xQnSNlBP_w) (2026-05)
- Prevent whole bug classes by construction instead of detecting instances ("what you cannot find, you cannot enforce"): enforce architecture via module-import rules (what may import what), e.g. BDD suite forbidden from importing DB-accessing modules; templates forbidden from DB access → N+1 queries become structurally impossible. [SafeIntelligence](https://www.youtube.com/watch?v=504PvfXou5Y) (2026-06)
- Constrained/structured generation to enforce order-of-operations at the MODEL level, not just orchestration prose: ban vocabulary tokens that violate constraints so the model CANNOT skip steps (force "call tool X first" / "read the N markdown files before acting"). Pydantic only makes the schema; an engine (xgrammar/outlines/guidance) enforces it. Caveat: open-weights/self-hosted only — closed providers withhold it because prefix-forcing output jailbreaks safety alignment. [BuzzHPC](https://podcasters.spotify.com/pod/show/mlops/episodes/Architecting-Modern-AI-Systems-Platforms--Agents--and-Integration-e3k0qlu) (2026-05)

## Guardrails & limits
- Handle auth/secrets deterministically in harness code, not via prompt: a per-loop handler checks current URL, injects credentials + submits form programmatically, then pushes "harness logged you in" back into context — secrets never enter the agent's context. [IBM](https://www.youtube.com/watch?v=C_GG5g38vLU) (2026-05)
- Scoped capability credentials for agent authority — bind to counterparty + amount cap + expiry, enforced by infrastructure outside the agent (over-limit action rejected even if the agent was duped or mis-parsed); blast radius bounded by construction, not prompt. [Stripe](https://www.youtube.com/watch?v=KLSuFPj2ld0) (2026-06)
- Failure PREVENTION over detection: wrap the MCP/tool layer with a guard that hard-limits irreversible actions (DB deletes, resource spin-up) so a looping/failed agent can retry safely — don't rely on the agent "knowing" the policy. [BuzzHPC](https://podcasters.spotify.com/pod/show/mlops/episodes/Architecting-Modern-AI-Systems-Platforms--Agents--and-Integration-e3k0qlu) (2026-05)
- Isolation + policy proxy: run each agent in a container it can't escape; route all reads/writes/executions through a proxied policy layer that checks before allowing. Per-actor scoping like per-team human access — read/write scope, sandbox critical/config systems, token budgets + circuit breakers per team/person (one engineer silently burned the whole Anthropic budget in 7h on a laptop). [Guild-Meta](https://podcasters.spotify.com/pod/show/mlops/episodes/From-Single-Player-to-Multi-Player-Operating-AI-Agents-at-Scale-e3khpk8) (2026-06)
- Container ≠ agent platform: a container knows only one session and has no isolation boundary across stateful sessions → real incidents from this one root cause — parallel agent sessions clobbering a *shared cache*; an auto-accept agent pushed the whole company's source to a personal GitHub because secrets were *commingled* in the box. You end up rebuilding above the VM anyway: stateful sessions, scoped identity + least-privilege per session, real machine semantics. [Cognition](https://www.youtube.com/watch?v=_xQnSNlBP_w) (2026-05)
- Dev boxes (large remote envs: 64-128GB RAM, pre-warmed pool, ~10s spin-up) beat lightweight sandboxes as agent homes — the agent inherits all the repo tooling + isolation for free. Static analysis / linters / type-checkers become *must-haves* once agents drive: a missing one no longer costs an engineer an hour, it costs thousands of failing agent cycles. [Stripe](https://www.youtube.com/watch?v=_xQnSNlBP_w) (2026-05)
- Per-tool human-in-the-loop confirmation as agent config: different gates per tool ("submit request" always needs approval; "list requests" asks once then runs free) rather than one global approval mode — needed for high-stakes on-prem agents. [deepset](https://www.youtube.com/watch?v=x2bH0RKPgdc) (2026-05)
- Approve/deny must be a hard gate — the LLM must NOT decide its own approval (defeats the purpose); route approval to the right approver (derive from user/role), not the requester. [HITL with n8n](https://www.youtube.com/watch?v=tDArkCqjA-c) (2026-05)
- Format tool output for the HUMAN reviewer, not the machine (raw UTC → human-readable) so the approver eyeballs fast; bound the wait — auto-deny after a timeout so approvals don't pile up / block concurrency. [HITL with n8n](https://www.youtube.com/watch?v=tDArkCqjA-c) (2026-05)
- No-button channels (voice): wrap confirm in a sub-workflow if-branch exposed as a "confirm" tool. Background/autonomous agents still human-gate every outward action (e.g. Slack ping) — never let AI message coworkers/clients unprompted, draft to a human first. [HITL with n8n](https://www.youtube.com/watch?v=tDArkCqjA-c) (2026-05)
- before-tool-call hook as an RBAC/approval gate: intercept before any mutation (e.g. block a contact update without an authorization check) to enforce auth/approval outside the prompt. Tension: the event-sourced camp argues *blocking* before-hooks break prompt caching and tank cost/latency (see Event-sourced harness design) — reconcile by keeping the gate a cheap synchronous check, or push enrichment/safety work to best-effort enrichers and reserve hard blocks for true mutations. [Tavon-Pi](https://www.youtube.com/watch?v=vAIDdLKB6-w) (2026-05)
- Double secret indirection for agent API keys: Podman/K8s secret → agent's own secret-ref → key is a pointer-to-pointer, so keys never leak into logs/config. [Red Hat](https://www.youtube.com/watch?v=F1DYkY1BlfM) (2026-05)
- At the moment of consequence, replace agent UI-scraping with a structured protocol — the source of truth relays canonical machine-readable state (cart/price/tax JSON, HTTP-402 payloads) every step, because an agent proxy layer mis-relays details extracted from UIs. [Stripe](https://www.youtube.com/watch?v=KLSuFPj2ld0) (2026-06)
- Trust tiers for browser-touching agents — tier 1 local: human-in-loop time-bound consent, keep allow-click friction by design (refuse "remember my choice"); tier 2 CI: containers + separate browser profiles; tier 3 open-internet: assume prompt injection, domain allowlists. Agents may share a tool across tiers but never a security model. [Chrome DevTools](https://www.youtube.com/watch?v=_B4Pv9ttFgY) (2026-06)
- Guardrails as composable hard caps at the harness level: max iterations (kill run), max messages (trigger compaction), max tool-call count, max retries, recursion-depth cap, execution-time cap — because agents aren't linear workflows and cost spirals without per-run/per-workflow/per-agent budgets; throttle by reading OpenTelemetry traces back into the loop and checking the recursion threshold each level (or every ~10 calls to avoid a latency tax). [IBM](https://www.youtube.com/watch?v=C_GG5g38vLU) (2026-05) [PwC](https://podcasters.spotify.com/pod/show/mlops/episodes/Autonomous-Agents-at-Work-From-OpenClaw-Hype-to-Enterprise-Reality-e3jj7u3) (2026-05)
- Grant autonomy as an earned spectrum, not binary: assist (answers with tools) → recommend (proposes actions, doesn't act) → gated action; an agent earns each step up and moves back down on regression. Classify work by reversibility first — reversible (ticket enrichment, summarization, RCA) can be autonomous; sensitive (prod changes, stability impact) gets tighter approvals; consequential (customer/legal/policy docs) gets the highest gating. Gating decision = risk (operational/legal/compliance) × blast radius. [PwC](https://podcasters.spotify.com/pod/show/mlops/episodes/Autonomous-Agents-at-Work-From-OpenClaw-Hype-to-Enterprise-Reality-e3jj7u3) (2026-05)
- Explicit fallback behavior: the agent should say "beyond my capability" rather than hallucinate — and that refusal is itself a signal for what to build next. [PwC](https://podcasters.spotify.com/pod/show/mlops/episodes/Autonomous-Agents-at-Work-From-OpenClaw-Hype-to-Enterprise-Reality-e3jj7u3) (2026-05)
- Grow autonomy incrementally one channel → one task → next; on every break, step back, fix, and add a guardrail so the failure can't recur. Brittle ~10-step automations will break → split or add guardrails. [Keys to my life](https://www.youtube.com/watch?v=sJ2jc7leKBk) (2026-05) (low-confidence src)
- 80/20 hybrid loop-breakers: ~80% open research on a high-reasoning model, ~20% hard-gated deterministic steps (validation, summarization, critic) on cheap non-reasoning models; cap research with use-last-result counters (after 4–5 iterations) or timeouts (~5 min) — degrade to the last decision, don't kill the run. This interweaving (deterministic structure-constrained steps + free LLM steps) is exactly how Deep-Research-style workflows reliably out-spend Claude Code/Codex on report generation, burning millions of tokens over hundreds of sources; generalizes to any domain. [Qodo](https://www.youtube.com/watch?v=EcqMYoIV57A) (2026-06) [BuzzHPC](https://podcasters.spotify.com/pod/show/mlops/episodes/Architecting-Modern-AI-Systems-Platforms--Agents--and-Integration-e3k0qlu) (2026-05)
- Token-hunger breaks subscription pricing; quota throttling per user/team is the current blunt defense against one power user spawning 100 agents. [DeepMind](https://www.youtube.com/watch?v=7gujZrJ9L5I) (2026-05)
- Bottlenecks / back-pressure are now a deliberate engineering primitive: agent PRs arrive faster than humans can review or even than they stay mergeable. [Pi](https://newsletter.pragmaticengineer.com/p/building-pi-and-what-makes-self-modifying) (2026-04)
- Gate agent dispatch on problem specificity: a coding agent thrown a vague report ("onboarding is broken") will always "fix" something → noisy meaningless PRs; triage to three states — actionable (dispatch), needs-human (inbox), under-evidenced (return to pool to accumulate more signals). Error-tracking signals are specific enough to action; Slack/session-replay signals usually aren't. [PostHog](https://www.youtube.com/watch?v=zMiSRliEzv4) (2026-06)
- Public-facing telemetry (errors, user-triggered events) is attacker-injectable prompt-injection surface — put an LLM safety classifier at the very top of the pipeline and drop malicious signals before any agent sees them. [PostHog](https://www.youtube.com/watch?v=zMiSRliEzv4) (2026-06)
- Prompt-injection surface isn't gated by the chat entry point: an always-on assistant monitoring email/calendar/WhatsApp ingests injections from ANY inbound channel (malicious calendar invite, forwarded message); self-developing agents that write their own skills/install software widen it further — access granted == risk taken. [iFood](https://podcasters.spotify.com/pod/show/mlops/episodes/The-Control-vs-Magic-Spectrum-Building-Agents-e3k9n0g) (2026-06)
- Filter agent contributions cheaply: auto-close PRs from unknown authors, ask for a human-voice issue; agents don't read the bot comment, so it self-selects humans. [Pi](https://newsletter.pragmaticengineer.com/p/building-pi-and-what-makes-self-modifying) (2026-04)

## Transparency, control & reversibility
- Transparent execution (live todo list, visible context/skills in use, tool-call inputs AND outputs) lets the user intervene mid-run before wasted steps compound — collaboration over delegation. [Flinn](https://www.youtube.com/watch?v=7CrPrHgoEYk) (2026-05)
- Multi-granularity reversibility (per-line, per-file, conversation-checkpoint rollback) bounds the cost of mistakes, making users bolder and willing to delegate higher-value tasks. [Flinn](https://www.youtube.com/watch?v=7CrPrHgoEYk) (2026-05)
- Enforce stage progression with a state machine + gates between agents (implementer→verifier→reviewer→closer→retro); a stage can't advance until the prior proves completion — control lives outside the model's discretion. [WorkOS](https://www.youtube.com/watch?v=vy7o1g2iHY8) (2026-05)
- Stable deterministic scaffolding matters more than capability: hidden context mutation (silent system reminders, system-prompt edits per release, tool-result pruning) silently breaks workflows you can't see. [Pi](https://newsletter.pragmaticengineer.com/p/building-pi-and-what-makes-self-modifying) (2026-04)
- Control↔magic spectrum as a per-product design axis: pick where each agent sits — high-stakes/low-trust (money transfers) → stiff, constrained, human-confirm; low-risk → let the model roam for the "wow". Harness shape differs completely per point; same axis applies to agentic UI (prebuilt component toolkit = controlled vs model emits raw HTML/JS on the fly = magic, good for internal/debug tools). [iFood](https://podcasters.spotify.com/pod/show/mlops/episodes/The-Control-vs-Magic-Spectrum-Building-Agents-e3k9n0g) (2026-06)
- Serialize the whole agent/pipeline definition to YAML and commit it, so each run maps to a git hash for reproducible, auditable logs — cheap versioning/traceability on a self-hosted stack without a managed platform. [deepset](https://www.youtube.com/watch?v=x2bH0RKPgdc) (2026-05)
- Canary + versioned agent releases per customer (enterprise voice): spin a separate container for the new version, route a % of traffic / a few phone numbers to it before going live — catches regressions and rebuilds trust after silent provider model drift ("why is my agent different than yesterday?"). [Bland](https://www.youtube.com/watch?v=_xQnSNlBP_w) (2026-05)

## Decision-making as a sub-process
- Treat decision-making as a distinct sub-process the agent enters on hitting uncertainty (a case not covered by its instructions), not just the next inline step; compartmentalize it — one agent proposes alternatives with pros/cons, a separate agent decides + acts (split proposal from authority/action). [Neo4j-Kollegger](https://www.youtube.com/watch?v=abvQEhvRI_c) (2026-05)
- Terminal states should be act / don't act / escalate — escalate to a higher-privilege agent or human when authority or certainty is missing; record the full reasoning trace + decision + actions taken into memory, so future agents reuse it as precedent. [Neo4j-Kollegger](https://www.youtube.com/watch?v=abvQEhvRI_c) (2026-05)

## Multi-agent orchestration
- Taxonomy of 5 multi-agent comm primitives a long-running system composes: (1) delegation — parent spawns child, gets result (sub-agents, built first); (2) creator-verifier — fresh-context agent checks the builder (builder has a "ship it" cost bias, same reason humans code-review); (3) direct communication — agents DM with no coordinator (hard: state fragments, no single source of truth); (4) negotiation — agents coordinate over a *shared resource* (same API/code region), best when positive-sum, needn't be adversarial; (5) broadcast — one→many status/new-constraint updates, unflashy but critical for coherence over long runs. [Factory-Alvoeiro](https://www.youtube.com/watch?v=ow1we5PzK-o) (2026-05)
- Leader/follower split where the leader does NO work itself: elect one leader → it scouts/observes shared state first, builds a to-do list, delegates items to follower agents, then only observes + judges whether each is done and done correctly. Frames the core multi-agent problems: shared state, leader/follower roles, followers are "blind" while working, and preventing overlapping work. [tldraw](https://www.youtube.com/watch?v=sPUjIBH5Cwg) (2026-05)
- Break a monolithic one-shot LLM call into explicit NAMED stages (collect-context → triage → policy-review → reply-writer → finalize/escalate) — more failure surface, but each stage is individually debuggable on the "stack"; make the first stage deterministic context-extraction before the agentic tool-calling stages, and emit a confidence score per decision (e.g. escalate y/n) so downstream / human-in-loop can gate on it. [Braintrust](https://www.youtube.com/watch?v=ZdheJTfLu-s) [Trainline](https://www.youtube.com/watch?v=ZdheJTfLu-s) (2026-05)
- Stripe "Minion" one-shot loop (prompt→PR, no human in loop, ~65% merge with zero edits): gather all context (Slack thread, ticket, linked PRs) → an `analyzer` agent localizes the right part of the repo → coding agent makes one turn → run lints/tests/typecheck → hand the diff + ORIGINAL prompt to an **LLM judge** ("is this task complete?") whose context is NOT poisoned by the coding agent's excuses → if incomplete, a `diagnostic` agent writes a SHORT note ("built endpoint, didn't wire frontend") fed back into the loop; repeat until PR-shaped. Web UI + "open the box in VS Code" to take over the failures. (Separate analyzer/judge/diagnostic agents > one self-reviewing thread.) [Stripe](https://www.youtube.com/watch?v=_xQnSNlBP_w) (2026-05)
- Run features SERIALLY (one worker/validator at a time), parallelize only READ-ONLY ops within a feature (codebase search, API research, code review). ## Streaming, transport & durable sessions
- SSE (the Vercel AI SDK / TanStack AI default) is one-way server→client and couples the response stream to a single client's single connection — drop = stream lost; resumability over raw streaming requires buffering events to a store (e.g. Redis) with sequence numbers plus a per-client resume handler that computes the missed-event set and replays in order. [Ably](https://www.youtube.com/watch?v=YNJvm7t3yq8) (2026-05)
- SSE forces resume and cancel to be mutually exclusive: with no upstream channel, a client closing the connection is ambiguous (reconnect-later vs cancel), so you can't have both a stop button and resumability; stop/interrupt + steering need a bidirectional transport (WebSockets). [Ably](https://www.youtube.com/watch?v=YNJvm7t3yq8) (2026-05)
- Bidirectional transport alone still doesn't give multi-surface visibility — a 2nd tab/device that didn't open the connection can't see the in-progress stream nor reach the agent; decouple the agent layer from the client layer via a "durable session": a persistent shared stateful medium (essentially PubSub) that's independently addressable (any client/agent joins by name), persistent (messages outlive any connection/device/agent), and fully resumable on reconnect — agents write events without caring about client connection health, any client subscribes/resumes. (Cloudflare's persistent-addressable durable-object backend is the same idea.) [Ably](https://www.youtube.com/watch?v=YNJvm7t3yq8) (2026-05)
- For multi-agent over a durable session, an orchestrator proxying granular sub-agent progress back to the client is an anti-pattern (dual role) — let each sub-agent write progress directly to the shared session and have the client subscribe to one entity for all-agent + other-client activity; human handoff = add a human participant to the same session with full interaction history. Supporting UX primitives: push notification when a backgrounded async task completes; shared/subscribable data objects for agent+user collaboration over live state. [Ably](https://www.youtube.com/watch?v=YNJvm7t3yq8) (2026-05)

## Loop engineering — in-the-loop → out-of-the-loop
- "Loop engineering": stop prompting the agent directly — author the loops that prompt it (triggers + goals), so your job moves to issue-writing on one end and code review on the other. Emerging trend pushed by Boris (Claude Code) and Peter Steinberg (OpenClaw). [Warp](https://www.youtube.com/watch?v=gi0Lx6c84VE) (2026-06)
- Pull the human out one stage at a time (source ticket → triage → implement → verify → PR) but keep the final merge gate — don't let agents self-merge yet; model intelligence isn't there, so the human checkpoint stays at "let code into the codebase." [Warp](https://www.youtube.com/watch?v=gi0Lx6c84VE) (2026-06)
- Issue labels as the loop's state machine: a `ready-for-spec` label routes to a spec-writing automation, `ready-for-agent`/`ready-to-implement` routes to an implementation automation; flipping the label is both the inter-agent handoff and the human approval point. [Warp](https://www.youtube.com/watch?v=gi0Lx6c84VE) (2026-06)
- Two-stage chain beats one agent: automation A turns an under-specified issue into a spec doc (product + tech spec — affected monorepo apps/packages, version bumps, acceptance criteria), you merge that spec into the repo as a paper trail, automation B picks it up and implements. Each loop is only as good as its issue's explicit acceptance criteria — the `goal` skill (Claude Code & Codex) drives the agent to those before it returns control. [Warp](https://www.youtube.com/watch?v=gi0Lx6c84VE) (2026-06)
- Stitch an automation onto every step of the contribution flow, not just implementation: an intake-triage agent does initial research + flags whether a new issue is well-scoped, and a PR-review agent pre-screens community PRs for critical/important bugs before any maintainer time is spent. Scales to thousands of issues on cloud runners (not local worktrees); Warp open-sourced the triage/PR-review templates + a common-skills repo of adaptable tech-spec/product-spec skills. [Warp](https://www.youtube.com/watch?v=gi0Lx6c84VE) (2026-06)
- End-to-end autonomous SWE chain: spec → agent interviews you to fill gaps → writes an LDD (low-level design doc) → distributes it for feedback → auto-creates non-blocking tickets → opens PRs, with blocking dependencies auto-flagged for a human. A prescriptive design doc up front curbs over-engineering / 1000-line sprawl by pinning scope before any code. [PFF](https://www.youtube.com/watch?v=VMemhtlsoNk) (2026-05)
- Tickets auto-update from PR status (open→in-progress→review→merged→closed) — the status flow replaces standups. [PFF](https://www.youtube.com/watch?v=VMemhtlsoNk) (2026-05)
- Self-healing QA stage on the loop: PR merge → auto-deploy to staging → QA agent reads the tickets + acceptance criteria → runs QA → flags failures (next step: the agent auto-creates fix PRs to close the loop). [PFF](https://www.youtube.com/watch?v=VMemhtlsoNk) (2026-05)
- Start with ONE sequential loop, not parallelism: the bottleneck on a continuous loop is your own ability to spec + review, not tokens/sec — adding agents doesn't help until that's fixed, so skip multi-agent / agent-mail until you actually hit the agent-throughput wall (most don't). Don't pre-compute a dependency DAG and fan out parallel agents either — that recreates waterfall and agents contend (two independently implement the same blocking ticket); drop the graph, single loop, prompt "pick the next most important ticket" and let the AI infer deps on the fly from what's already done. [Ralph-Parsons](https://www.youtube.com/watch?v=2TLXsxkz0zI) (2026-05)
- A Ralph-loop prompt is a built-up role+procedure, not "do the ticket": cast the agent as "one engineer in a relay team — do exactly one change, drop context, stop", and encode ticket format/status values, when work qualifies, how to run tests in *your* framework, and "test passing is not enough — verify actual behaviour." Evolve it per-project; start from someone else's and tell the agent "fix this for my project." [Ralph-Parsons](https://www.youtube.com/watch?v=2TLXsxkz0zI) (2026-05)
- Bake recovery states into the loop prompt so a crashed while-true loop resumes sanely: on restart check git — dirty tree + tests passing → probably done (double-check); tests failing → mid-flight but broken, throw it away. [Ralph-Parsons](https://www.youtube.com/watch?v=2TLXsxkz0zI) (2026-05)
- Prefer a fresh context per loop iteration (over resuming the session) as a forcing function: it forces all in-session knowledge to be codified into repo files/skills, yielding a durable knowledge store any future session or human can read instead of knowledge trapped in a session. [Ralph-Parsons](https://www.youtube.com/watch?v=2TLXsxkz0zI) (2026-05)
- Never let the loop close / mark-done its own work — stay the final verification gate. Motivation is "cognitive debt": losing track of what the codebase does; at minimum understand how each piece fits before it ships out of sight. [Ralph-Parsons](https://www.youtube.com/watch?v=2TLXsxkz0zI) (2026-05)
- Event-driven "always-on" loop (chess-coach via Claude Code "Channel", a research-preview MCP server that injects external events into a continuously-running session, OpenClaw-style): a user "this output is bad" report posts to Slack AND wakes the session, which runs a triage skill — inspect the bad case, run generation scripts, modify the prompt/detectors, regenerate, then verify its own work, asking clarifying questions back to Slack and opening a PR you merge from mobile. Crucially the triage skill allows "conclude nothing is wrong" as a valid outcome (demonstrated: it correctly cleared a non-bug) — don't force the agent to always change something. ## Agent-rate development — CI in the loop, merge queues
- The PR breaks as the unit of work at agent scale: it was built for human-paced discrete-handoff review, but agents emit N PRs across N repos with thousands of short-lived branches → review + merge become the bottleneck. Validation moves OUT of a separate post-PR CI phase and INTO the agent's inner loop — every iteration builds + tests + gets reviewed, not just at the end. [Namespace](https://www.youtube.com/watch?v=VktrqzQgytY) (2026-05)
- Inner-loop build/test speed becomes a first-class agent-throughput constraint: a 15–45-min test run stalls the whole loop. Run the loop in a stateful, warm environment (persistent state across iterations, incremental work like an engineer's local workstation) rather than cold rebuilds each turn; check out a known-vetted commit each iteration so base-code invariants hold continuously instead of once at a CI gate. [Namespace](https://www.youtube.com/watch?v=VktrqzQgytY) (2026-05)
- Replace human "external validation" in-loop with other agents — a security-focused LLM + an API-conformance LLM feed findings back to the main harness before merge. [Namespace](https://www.youtube.com/watch?v=VktrqzQgytY) (2026-05)
- Merging at agent rate ≈ a high-performance DB serialization problem — the git repo is a single ledger needing a lock per commit; machine merge-locks are short + frequent (vs human long + rare), so time-to-merge dominates. "Pre-merge queue" pattern: completed candidate changes wait in a reconciliation queue (not the repo) so many parallel agents touching the same code serialize into the ledger; the human approves intent + result here (a video of the feature working, the security-LLM output, grouped commits) rather than reading raw diffs. Speculative next step: agents apply a plan to multiple in-flight candidate commits at once (not just the moving repo tip), exploding compute as the price of speed. [Namespace](https://www.youtube.com/watch?v=VktrqzQgytY) (2026-05)
- De-risking displaces speed as the pipeline priority: with an agent driving, feedback-loop *speed* matters less (the agent has moved on — no human context-switch cost waiting on a build), so emphasis shifts to *de-risking* agent-generated code. The agent can babysit its own pipeline — read the failure, push a fix, re-run — so a slow build is less of a blocker than when a human waits. [CI/CD Erez](https://newsletter.pragmaticengineer.com/p/cicd-with-robert-erez) (2026-06)
- Feature toggles become a core agent-velocity tool: decouple deploy from release so an agent ships fast but a misbehaving feature flips off in seconds (vs minutes to redeploy/rollback). Pair with flag hygiene — Octopus wraps the OpenFeature SDK to record owning team + expiry per toggle, and CI notifies the team when expiry passes — because agent throughput piles up stale-flag debt faster. [CI/CD Erez](https://newsletter.pragmaticengineer.com/p/cicd-with-robert-erez) (2026-06)
- "Self-validation == an ephemeral environment": any "have the agent confirm its change works" loop is really spin-up env → agent pokes/tests → tear down — the same preview-env pattern used for humans, even when no human ever looks. The agent's verification is only as good as the env you hand it: Granola turned a one-instance Electron app into a web shell so CI emits a per-PR preview link, then Cursor tests the PR in that preview and uploads screenshots into it (the single desktop instance had blocked parallel testing). Validating the thing *works* (esp. UI) beats only code-reviewing the diff. [CI/CD Erez](https://newsletter.pragmaticengineer.com/p/cicd-with-robert-erez) (2026-06) [Granola](https://www.youtube.com/watch?v=ON5LIT0M4do) (2026-05)

## Dynamic workflows — a bespoke harness per task
- One coding harness gets funnelled all knowledge work but isn't optimal for non-coding jobs; let the agent write a custom harness per task at runtime instead of forcing the task through a generic one (Opus 4.8 is finally good enough to author the harness on the fly). Static workflow (Agent SDK / `claude -p` wired ahead of time) must stay generic → generic output; a dynamic workflow is just a JS file running a small team of agents with primitives `agent` (sub-agent in its own clean context), `parallel` (fan out, barrier-wait), `pipeline` (stream items through stages); it picks the model per agent + whether each gets a worktree, and resumes if interrupted. [Prompt Engineering](https://www.youtube.com/watch?v=l5rae4LMKBc) (2026-06)
- Why split a job across clean contexts: one overflowing context window causes three failure modes — agentic laziness (quits early, declares done after partial work), self-preferential bias (grades its own work too kindly when asked to self-check), and goal drift (the brief leaks out, especially after compaction). [Prompt Engineering](https://www.youtube.com/watch?v=l5rae4LMKBc) (2026-06)
- Six portable harness patterns (Anthropic): (1) classify-and-act — a classifier routes by task type, or sits at the end to shape output; (2) fan-out-and-synthesize — many isolated contexts so biases don't cross-contaminate, then a barrier merges; (3) worker+critic — a separate critic attacks the output against a rubric, kept only if it survives (cures self-preferential bias); (4) generate-and-filter — pile of candidates filtered by rubric/verification, keep the best (naming, design, taste); (5) tournament — agents solve the same task differently, a judge compares head-to-head until one champion (pairwise beats absolute scoring, scales past one prompt); (6) loop-until-done — a gate asks "any new findings?" so the loop, not a tired context, decides when to stop (cures agentic laziness). Works on messy non-coding work too (ranking resumes, tearing a plan apart from three angles, mining your last 50 sessions into CLAUDE.md rules, root-causing six months of Slack): verification = claim-extractor + per-claim checker; triage+quarantine adds a rule that an agent reading untrusted content can't act on it (prompt-injection defense); 1000 support tickets = pairwise tournament. [Prompt Engineering](https://www.youtube.com/watch?v=l5rae4LMKBc) (2026-06)
- Harnesses are reusable saved artifacts, not single-use — generate several, keep the best. Trigger with the keyword `workflow` or `ultracode`, cap cost in the prompt ("use only 10,000 tokens"), pair with `/loop` or `/goal` for a hard finish line — but it burns tokens dramatically, so first ask whether the job really needs more compute or a simple agentic pattern suffices. [Prompt Engineering](https://www.youtube.com/watch?v=l5rae4LMKBc) (2026-06)

## Determinism — partition workflows, compile repeated work
- Partition agent workflows by determinism: discovery/exploration benefits from non-determinism, but credentials/transactions/anything with consequence requires it — design an explicit handoff point where the LLM-mediated phase ends and a purely programmatic flow takes over. [Stripe](https://www.youtube.com/watch?v=KLSuFPj2ld0) (2026-06)
- Compile repeated LLM work into generated scripts: agent inspects one raw page, writes a deterministic parser, then the script runs (~100 tokens/run vs ~10k through context; ~62% save even on a structured site); generalizes to any repeated per-item LLM task. [Bright Data](https://www.youtube.com/watch?v=zTZ0qunQXnM) (2026-06)
- Self-healing pipeline loop: a scheduled agent run validates the generated script's output and, on missing fields, re-inspects the source and patches its own scraper — maintenance moves to the agent. [Bright Data](https://www.youtube.com/watch?v=zTZ0qunQXnM) (2026-06)
- Don't cost-optimize an agent pipeline while experimenting ("tokens are free" stance): run the expensive agent on the same problem ~100x, watch its solutions converge, then compile that step into a one-shot LLM call or small trained model — premature avoidance of agent calls was PostHog's biggest mistake. [PostHog](https://www.youtube.com/watch?v=zMiSRliEzv4) (2026-06)
- Agent self-builds its own reusable tools: instruct it that for any repeatable task it should write a script and register that script's description at the end of `memories.md`; next time the same request arrives it reuses the existing script instead of regenerating. Coding is the natural first domain for self-extension (compilers/type-checkers already give a tight build-run-evaluate loop). [Vercel](https://www.youtube.com/watch?v=wflNENRSUb4) (2026-05)

## Running LLM-generated code — sandboxes
- Mental model: maximize non-determinism *inside* the box, make the box *really strong* — "sandbox the shit out of the agent" so handing it raw bash/Python is safe. Cleric's arc: early on hand-wrote extremely rigid custom tools (no raw `kubectl`) so the agent could only do the intended thing (fixed tools = determinism where you want it); as models improved the rigid tools became the constraint blocking better performance, so they relaxed to letting the model compose bash/Python — but only AFTER doing sandboxing hard, because rigid tools had been giving free security guarantees that bash gives up. [MLOps-Sandboxing](https://podcasters.spotify.com/pod/show/mlops/episodes/Sandboxing--Agent-Harnesses--and-Agent-Teamwork-e3kts75) (2026-06)
- The durable hard part is environment simulation + verifying correctness across diverse setups, NOT the investigation/coding agent — every customer's infra (k8s setups) differs wildly, so an agent good at one company sucks at the next; the agent has become the *least* important / easiest part of the stack while building test environments is the lasting challenge. [MLOps-Sandboxing](https://podcasters.spotify.com/pod/show/mlops/episodes/Sandboxing--Agent-Harnesses--and-Agent-Teamwork-e3kts75) (2026-06)
- Inverted sandbox design for generated code: start from a zero-capability JS isolate (no fetch, no filesystem, no env vars) and grant explicit capabilities outward (e.g. only outbound fetches to github.com) — opposite of a VM/container locked down after the fact; ms spin-up makes a fresh sandbox per LLM-generated code string cheap. [Cloudflare](https://www.youtube.com/watch?v=SKDJo2CopRs) (2026-06)
- Anywhere the LLM would generate a JSON/DSL config that code later interprets, have it generate real code (HTML/React/JS) and run it in the untrusted-code sandbox — DSLs only existed because platforms lacked a safe untrusted-code primitive (applies to generative UI and plugin systems alike). [Cloudflare](https://www.youtube.com/watch?v=SKDJo2CopRs) (2026-06)
- Minimal personal-agent (OpenClaw-shaped) recipe: heartbeat/scheduler + virtual file system + service connections + secrets held by the host and never exposed inside the sandbox + extensions implemented as generated code run sandboxed. [Cloudflare](https://www.youtube.com/watch?v=SKDJo2CopRs) (2026-06)
- Stateful sandbox + "shared-nothing" micro-app pattern: keep the sandbox alive, put a SQLite DB (~1MB file) inside as the only state, snapshot the sandbox for time-travel/restore — no external DB or git; the generative-UI "frame" is the human view, the sandbox is the agent's API; cheap (boot ~100ms, idle-shutdown); breaks only at extreme concurrency (then graduate to hosted DB + git). [Dust](https://podcasters.spotify.com/pod/show/mlops/episodes/MCP--Agents--the-40M-Bet-on-Multiplayer-AI-e3kmopj) (2026-06)
- Agent SSH-sandbox mode: hand the agent SSH keys + known_hosts and it runs all commands in that remote workspace instead of the host. [Red Hat](https://www.youtube.com/watch?v=F1DYkY1BlfM) (2026-05)
- Mount one agent dir (skills + MCP servers + tools) as a single volume into the agent container → everything wired at startup; reproducible/portable across laptop/x86/Mac/K8s. [Red Hat](https://www.youtube.com/watch?v=F1DYkY1BlfM) (2026-05)
- One full K8s pod per agent task — "wasteful but a better abstraction" (a whole computer beats a locked-down sandbox for agent power), paired with read/write state-sync across harness/repo/chat client. (low-confidence src — vendor framing) [ACP](https://www.youtube.com/watch?v=VaS2h-dY1-4) (2026-05)

## Trajectory stores & retro / self-improvement
- Agent trajectory store: persist step-by-step traces to pinpoint exactly where looping or derailment began. [DeepMind](https://www.youtube.com/watch?v=7gujZrJ9L5I) (2026-05)
- Retro agent parses Claude/Codex JSONL transcripts to detect doom loops (same tool call repeated unchanged, excessive parallel tools) and writes lessons into per-stack markdown memory files; harness-engineering discipline — when the agent errs, fix the harness so it can fix itself, never hand-fix the output (every failure is a system bug + data for the next run). [WorkOS](https://www.youtube.com/watch?v=vy7o1g2iHY8) (2026-05)
- Skill-based harness leaked context as it grew (model "forgot"/skipped tasks); moving orchestration to deterministic code fixed it. [WorkOS](https://www.youtube.com/watch?v=vy7o1g2iHY8) (2026-05)
- A working agent harness + collected eval traces ≈ ready-made training data — RL can teach a model your service from what the harness already logs; RL rollouts are embarrassingly parallel — fan out to many ephemeral sandboxes, kill unpromising runs early (hyperparameter search becomes meta-evolutionary). [Modal](https://www.youtube.com/watch?v=HvZXAOZ3iv8) (2026-06)

## Self-modifying / bespoke harnesses
- Self-modifying harness: agent loads custom TS modules into the same process to add tools, custom compaction, full TUI rewrites — per-user, per-task. Per-project bespoke harness wins: e.g. a game project where the agent built itself screenshot/state-dump/sim tools + TUI image rendering + conversation branching/rewind, so it can self-validate while the human stays in loop. [Pi](https://newsletter.pragmaticengineer.com/p/building-pi-and-what-makes-self-modifying) (2026-04)
- Make your agent itself easy for *other* coding agents to build/test (a "pseudo-RL" loop): expose a CLI, write AGENTS.md, add skills, wire CI/CD — so long-running agents can change and end-to-end test your agent in a parallel thread. [Cline](https://www.youtube.com/watch?v=yUmS-F9IX90) (2026-05)

## Org-scale rollout & adoption
- Standardize on ONE agent platform (chose Claude Code) — like avoiding multi-cloud: spreading work across tools forfeits compounding platform benefits, and it also kills per-task "model anxiety" (no deciding which tool/model per task). [Intercom](https://www.youtube.com/watch?v=4_VQBbs2iQA) (2026-05)
- "Connect the agent to everything": anything you'd do on a laptop the agent can do — debugging, testing, planning, not just code; mature-org controls/permissions/audits are what make unleashing it that broadly safe. [Intercom](https://www.youtube.com/watch?v=4_VQBbs2iQA) (2026-05)
- Onboard the agent like a new hire — capture framework conventions, architecture, UI patterns, testing/security standards as context, skills, guidance, and hooks. [Intercom](https://www.youtube.com/watch?v=4_VQBbs2iQA) (2026-05)
- Push internal agent plugins to all laptops, bypassing the vendor's own update mechanism — managing agent installs across hundreds of laptops is like managing Python installs, so own the distribution. [Intercom](https://www.youtube.com/watch?v=4_VQBbs2iQA) (2026-05)

## CRUD / stateful-tool agents in eval
- CRUD-tool agents break offline eval: hard to reconstruct external-system state at input-capture time and risky to mutate prod — solve with mock APIs + state reconstruction. Reconstruct historical state via versioned/timestamp queries (e.g. query a vector DB as-of the time the input was captured) to match the original run's context, or cram system/external state into the (arbitrarily large) trace itself and inject it into the eval task — avoids building separate test fixtures/infra. [Braintrust](https://www.youtube.com/watch?v=FB-MLPhL9Ms) (2026-05)
- Evaluate the whole trace, not just final output, once tools/MCP calls are involved — target evals at individual tool/MCP steps. [Braintrust](https://www.youtube.com/watch?v=FB-MLPhL9Ms) (2026-05)

## Eval-running infrastructure
- Run each eval task in an isolated environment (VM/container preloaded with the repo + the agent installed); Harbor (Laude Institute) runs many tasks in parallel on standardized per-task configs → the slowest task is the limiting factor (infra options: Modal, Daytona, Docker). [Cline](https://www.youtube.com/watch?v=QuuIywMG4s8) (2026-06)
- Mock the expensive backend (e.g. mock TPUs) to test the harness/agentic flow without burning real compute; the hardest part of agent eval is new datasets + mechanically spinning up sandboxed envs per problem, not the scoring. [DeepMind](https://www.youtube.com/watch?v=7gujZrJ9L5I) (2026-05)
