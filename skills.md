# Skills

## What a skill is
- A skill is just a detailed prompt with clear goals in a markdown file — human language, no code [Sentry](https://www.youtube.com/watch?v=li0SaBt9RDM) (2026-05)
- Anthropic-style structure: `skill.md` (metadata + one-line desc) + optional scripts/assets; needs no special training — generic strong function-calling + thinking suffices [LiteRT-LM](https://www.youtube.com/watch?v=BKWpYIWvAo4) (2026-05)
- Convert hard-to-code branching logic into a skill (a procedure description) — solves cases plain code can't [Tessl](https://www.youtube.com/watch?v=bSG9wUYaHWU) (2026-05)
- Versioned, source-controlled, file-based context the agent opens/closes on demand — turns a zero-shot task into few-shot by shipping worked examples [HuggingFace](https://www.youtube.com/watch?v=JomVvNDjGb8) (2026-05)
- Limit of example-based skills: retrieving high-quality expert trajectories as in-context examples helps, but near-identical cases diverge on tiny policy-sensitive attributes (account tier, location) that look the same to humans too — retrieval alone can't discriminate the right trajectory; a tuned model is still needed [ElevenLabs](https://podcasters.spotify.com/pod/show/mlops/episodes/Voice-Agent-Use-Cases-e3ileg6) (2026-05)
- Skills authored by domain experts hand the agent (and other engineers) that expertise "for free" [DeepMind](https://www.youtube.com/watch?v=7gujZrJ9L5I) (2026-05) — skills packs make formerly-scarce expertise abundant (e.g. property-based + deterministic/system testing, Hypothesis-style, via a skill, beating ad-hoc Playwright use) [Huntley](https://www.youtube.com/watch?v=wjXowoQ7E8c) (2026-06)
- Triggered by mention: e.g. when an ADR is referenced, the ADR skill loads how to find/operate on affected code [Safe Intelligence](https://www.youtube.com/watch?v=504PvfXou5Y) (2026-06)
- A skill can also be a *bundle of small CLIs* the agent drives, not a native integration — a Cowork Excel skill = Pandas + openpyxl + LibreOffice CLIs packaged together; "make it easy for the coding agent" by handing it command-line tools it's reliably good at rather than bespoke tool-call plumbing [Tavon-Pi](https://www.youtube.com/watch?v=vAIDdLKB6-w) (2026-05)
- Skill and MCP are complementary, not rivals: skill = guidance, MCP = tools already present; the live debate has moved past "MCP vs skill" to "MCP vs CLI" [Supabase](https://www.youtube.com/watch?v=JT3OzDKrucU) (2026-05)
- Skills carry security knowledge MCP alone lacks: an agent with only the Supabase MCP created a Postgres view over an RLS table, silently bypassing RLS (missed `security_invoker=true`); same agent + skill got it right [Supabase](https://www.youtube.com/watch?v=JT3OzDKrucU) (2026-05)
- Shift from "run skill X to do this" to just describing the problem and letting the agent pick which skills to invoke — a well-written skill auto-fired on a security-incident triage the author didn't even know existed [Intercom](https://www.youtube.com/watch?v=4_VQBbs2iQA) (2026-05)

## Less is more — skills can actively hurt
- 10k lines of docs-derived skills produced WORSE results than 553 lines of hand-written "common gotchas" — cover the landmines, not the whole product [WorkOS](https://www.youtube.com/watch?v=vy7o1g2iHY8) (2026-05)
- Sharp negative constraints beat a prescriptive novel: 3 crisp "never" rules ("never cite code without file:line + commit ref", "never be vague") outperform a long markdown spec, and over-prescription can actively REGRESS a model already good at the task — one measured skill dropped a task 97%→77%, and a Next.js installer skill caused a ~30% accuracy drop by being dogmatic about something Claude already did well (both caught only via evals); every skill must earn its place [WorkOS](https://www.youtube.com/watch?v=vy7o1g2iHY8) (2026-05) [WorkOS-Skills](https://www.youtube.com/watch?v=pFsfax19yOM) (2026-05)
- Skills/slash-commands earn their place only for narrow workflow automation (e.g. "generate testing-patterns.md from the cloned repo"), NOT for stuffing framework internals — a skill per library-internal just pollutes context [Effect](https://www.youtube.com/watch?v=Wmp2Tku2PrI) (2026-05)
- Big skill set sends the model on goose chases checking many things; lean gotchas keep it focused and cut run time (68min → 6min evals) [WorkOS](https://www.youtube.com/watch?v=vy7o1g2iHY8) (2026-05)
- Each thing you add to an agent risks making it worse — large prompts/skills can degrade frontier models; prune relentlessly, get out of the model's way (newer models perform worse with over-instruction; Codex GPT-5.3 system prompt ~1/3 the size of GPT-5's) [Cline](https://www.youtube.com/watch?v=yUmS-F9IX90) (2026-05)
- Knowledge/context banks rot fast — stale on every model release + feature change; aggressively prune (don't just append) or old entries cause context rot and degrade the live agent [Lovable](https://www.youtube.com/watch?v=KA5kPbdkK2E) (2026-06)
- This applies to LOADED context, not idle skills on disk: only each skill's description (~75–150 tokens) sits resident, the ~1.3k-token body loads on demand — so unused *local* skills aren't worth deleting; pruning bites in prod/CI where the skill-listing budget (default ~1% of context, ~15–25 skills on a 200k model) truncates descriptions [Supabase-SkillIssue](https://www.youtube.com/watch?v=GmAQKINjv1E) (2026-05) [claudefa.st](https://claudefa.st/blog/guide/mechanics/skill-listing-budget) (2026-05)
- Reconciliation: "less is more" is about quality/leanness, NOT "no skills." Sources that advocate many skills [HuggingFace](https://www.youtube.com/watch?v=JomVvNDjGb8) [DeepMind](https://www.youtube.com/watch?v=7gujZrJ9L5I) still demand each be lean, expert-authored, evaluated, and curated — the failure mode is dumping whole docs, not having skills at all.

## What to put in a skill
- Models already know how to code; skills should encode product-specific intricacies and gotchas, not re-teach general competence — guide don't prescribe, with specific conditional rules ("in the Next.js proxy you can't call redirects") rather than dumping summarized docs [WorkOS](https://www.youtube.com/watch?v=vy7o1g2iHY8) (2026-05)
- Ship verification inside the skill: bundle benchmark/test scripts so the agent self-checks its output (e.g. kernel correctness + speedup) instead of trusting it [HuggingFace](https://www.youtube.com/watch?v=JomVvNDjGb8) (2026-05)
- A skill can own infra/capacity reasoning, not just the command: a fine-tune skill does the napkin math (VRAM for the model at a given batch size → which GPU instance fits, e.g. 4-bit Gemma in a 24GB L4), then asks only the genuinely-open questions (instance, validation split) before launching the remote job — so the agent self-provisions. Generalizes to any batch job (pick a cheap model off a benchmark → agent writes the processing script → does the cost/instance math → kicks off a re-runnable one-off job; OCR'd 30k papers this way) [HuggingFace-Hermes](https://www.youtube.com/watch?v=OV56RddyFuU) (2026-05)
- A model-selection skill should optimize for the downstream *use*, not the leaderboard top row — "don't be fooled by the #1 benchmark result"; asked "best OCR model for fine-tuning" it weighs fine-tuning suitability + size constraints, not raw score [HuggingFace-Hermes](https://www.youtube.com/watch?v=OV56RddyFuU) (2026-05)
- Structure exploration into fixed modes + structured output, not prose: "Catch Me Up" comprehension skill uses fixed modes (architecture, conventions, feature trace, syntax, testing, history) returning org charts/tables [Sentry](https://www.youtube.com/watch?v=li0SaBt9RDM) (2026-05)
- Onboarding-to-unfamiliar-code pattern: state your role ("I am a new contributor") + a specific yes/no question to verify, not just "explore this repo" [Sentry](https://www.youtube.com/watch?v=li0SaBt9RDM) (2026-05)
- Personalization via reusable playbooks/memory: encode the user's own method (e.g. how a firm reviews a contract) so the agent does it the right way, not just any way [Flinn AI](https://www.youtube.com/watch?v=7CrPrHgoEYk) (2026-05)
- SOP-as-spec: encode agent behavior as the same procedure docs (SOPs) domain experts already wrote for humans — but make them far longer/more detailed since there's no human reading-fatigue limit; lets non-engineer experts own the skill [ElevenLabs](https://podcasters.spotify.com/pod/show/mlops/episodes/Voice-Agent-Use-Cases-e3ileg6) (2026-05)
- Package grading rubrics as skills to inject your QA/design standards into the normal dev flow [Anthropic](https://www.youtube.com/watch?v=mR-WAvEPRwE) (2026-05)
- SKILL.md = style/process, NOT facts: encode HOW to implement (ask follow-ups before deciding; propose plan + approval gate before irreversible/data-leaving actions); push reference content out to live docs [Langfuse](https://www.youtube.com/watch?v=vNCY9kXXyDQ) (2026-05)
- Written rule files (agents.md / semantic-memory rules) are followed only ~7/10 at best — a model limitation, not a config fix; don't over-rely on them, and make must-happen steps deterministic (hooks/CI) rather than prose (cf. "deterministic instructions > prompt-nagging" in harnesses.md) [AutoHand](https://www.youtube.com/watch?v=wjXowoQ7E8c) (2026-06) (low-confidence src)
- Markdown playbooks/runbooks encode repeatable structured analysis pipelines — "creating an AI runbook saves days/weeks" [incident.io](https://www.youtube.com/watch?v=L2r6vLlLgs8) (2026-05)
- Structured templates beat open-ended chat for parallelizing agents reliably; for brownfield migrations, do docs-first — have the agent write docs over the legacy code (e.g. 2M LOC of undocumented COBOL) and migrate from the docs, not the source [Cognition](https://www.youtube.com/watch?v=_xQnSNlBP_w) (2026-05) (low-confidence src: product-pitch framing; transferable bit = structured-template-over-chat + docs-first migration)
- Red-green runbook for fixing a bug via prompt edits: write a failing eval reproducing the bug → modify the prompt until it passes → re-run the full suite for regressions → final pass to consolidate/simplify the bloated prompt; bake the simplify pass in since iterative edits accumulate bloat [incident.io](https://www.youtube.com/watch?v=L2r6vLlLgs8) (2026-05)
- Reference dynamic content, don't inline it — pasting docs into a skill creates a stale local cache (same staleness as pre-training data) + a conflicting representation; point the skill to your single source of truth and be stubborn/persistent telling the agent to actually go fetch it (agents default to stale training data and are lazy to fetch) [Langfuse](https://www.youtube.com/watch?v=vNCY9kXXyDQ) (2026-05) [Supabase](https://www.youtube.com/watch?v=JT3OzDKrucU) (2026-05)
- Don't band-aid each tool error into the system prompt one at a time (you end up re-inlining the whole query-language doc) — promote real per-tool documentation into a skill instead [Elastic](https://www.youtube.com/watch?v=ynJyIKwjonM) (2026-05)
- "If it can be skipped, it will be skipped": agents are lazy to load reference files — a skill that loads usually pulls ZERO or at most ONE reference file; two is "almost impossible," three/four never. Put anything must-not-miss (e.g. a security checklist) directly in skill.md, not a reference file — Supabase moved their RLS/security checklist into skill.md after watching it get skipped [Supabase](https://www.youtube.com/watch?v=JT3OzDKrucU) (2026-05)
- Be opinionated — encode the workflow you know is best so the agent doesn't freelance: Supabase bakes the schema workflow into the skill (run direct DDL freely on dev/staging → run the security/perf advisor → fix → only THEN write a migration file), preventing the agent from spamming a migration on every schema change [Supabase](https://www.youtube.com/watch?v=JT3OzDKrucU) (2026-05)
- Skill that reads ALL prior artifacts so new work matches org ethos: an LDD (Lightweight Design Document) skill reads every past LDD before generating the next; tool-agnostic, not Claude-specific [PFF](https://www.youtube.com/watch?v=VMemhtlsoNk) (2026-05)
- Encode the team's own design patterns as composable skills — service-repository pattern for APIs, feature-flag creation for trunk-based dev, branch naming, analytics generation [PFF](https://www.youtube.com/watch?v=VMemhtlsoNk) (2026-05)
- Pre-training knowledge is a LIABILITY for a maturing tool: agent confidently writes SDK/CLI calls from memory (old-version methods/params), "verifies", fails, then fetches docs — slow + wrong; skill must force fetch-first [Langfuse](https://www.youtube.com/watch?v=vNCY9kXXyDQ) (2026-05)
- Agent-friendly defaults invert human defaults: extra prompts/vars cost the agent nothing, so make the skill ASK (e.g. data region) rather than assume a config-minimizing default [Langfuse](https://www.youtube.com/watch?v=vNCY9kXXyDQ) (2026-05)
- Progressive disclosure / on-demand loading is the core scaling mechanism: keep only each skill's name + description (+ file-store location) resident, load the full body on instantiation and offload it once the step is done, then reference deterministic code — works on-device too (gate body behind a model-callable "load skill" tool) and the same on-demand fetch applies to compaction (push prior tool results to the store, re-fetch when needed) [Anthropic](https://www.youtube.com/watch?v=mR-WAvEPRwE) (2026-05) [Elastic](https://www.youtube.com/watch?v=ynJyIKwjonM) (2026-05) [Function-Gemma](https://www.youtube.com/watch?v=-TiET_K-E_g) (2026-05)
- Contrast with MCP, which loads all tool specs up front: agent sees only one-line skill descriptions, a `load_skill` tool pulls full skill.md on demand — minimal on-device orchestrator = 3 tools (`load_skill`, `run_javascript`, `run_intent`); flow = read metadata → load_skill returns skill.md → model runs functions [LiteRT-LM](https://www.youtube.com/watch?v=BKWpYIWvAo4) (2026-05)
- Split skills into input tools (fetch: Wikipedia/weather/CRM/RAG) vs output tools (render: maps/cards) [LiteRT-LM](https://www.youtube.com/watch?v=BKWpYIWvAo4) (2026-05)
- The skill `description` IS the runtime routing rule (the LLM reads only descriptions to decide load/no-load) — write it to disambiguate near-duplicate skills by stuffing the acronyms/terms/conditions it should trigger on (e.g. blog-image-gen vs work-S3-image-gen); a thin "skill-router" SKILL.md pointing to a reference map, each reference .md loaded only on its sub-task, is progressive disclosure in practice [WorkOS-Skills](https://www.youtube.com/watch?v=pFsfax19yOM) (2026-05). Concrete authoring tip: start the `description` with the verb `use` + the purpose — raises load probability (claimed Claude-specific) [Supabase-SkillIssue](https://www.youtube.com/watch?v=GmAQKINjv1E) (2026-05)
- Debug description-routing empirically: define both should-fire AND should-NOT-fire scenarios, run them via the CLI, and tune the description against the negatives (false-positive loads) — not just the positives [Supabase-SkillIssue](https://www.youtube.com/watch?v=GmAQKINjv1E) (2026-05)
- Three escalating force-load levers when routing is unreliable: plain prompt = pray it fires; `use <skill-name>` in the prompt ≈100%; `/<skill-name>` slash command = 100% guaranteed [Supabase-SkillIssue](https://www.youtube.com/watch?v=GmAQKINjv1E) (2026-05)
- `!`-backtick syntax in SKILL.md = shell interpolation (like JS `${}`): Claude runs the backtick'd command and substitutes its *output* into context before acting (e.g. inline a `git log` in your exact format instead of telling it to fetch commits) — deterministic + token-saving, claimed Claude-specific [WorkOS-Skills](https://www.youtube.com/watch?v=pFsfax19yOM) (2026-05)
- Confidence-gate inside a skill: have it self-score sub-dimensions (problem clarity / goal / success criteria / scope) summing to 100 and refuse to execute until ≥95, forcing it to "show its work" on why it's confident (claiming "I'm sure" → asked why → realizes it isn't); pull missing inputs via the built-in ask-user-question (multiple-choice) rather than open prompts [WorkOS-Skills](https://www.youtube.com/watch?v=pFsfax19yOM) (2026-05)
- Skills can embed executable code (JS) the harness runs as part of execution (render a UI, run a picker) — app-side code triggered by the agent, not just text [Function-Gemma](https://www.youtube.com/watch?v=-TiET_K-E_g) (2026-05)
- Prefer structured *text* output over vision models when generating structured data: text-output models have far more training data, while vision data self-conflicts (y-axis-up vs screen-y-down, ambiguous "left") — getting predictable structured output from vision still takes heavy prompt engineering [tldraw](https://www.youtube.com/watch?v=sPUjIBH5Cwg) (2026-05)
- Let the orchestrator model author downstream tool inputs rather than hand-writing them — it produces the tool's "native" input format (the tool was often trained on the orchestrator's own outputs) better than a human one-liner; and bias those inputs toward verbose/explicit, since downstream models internally rewrite/expand short inputs before acting so one-liners yield weak results [Gemini](https://www.youtube.com/watch?v=BcWFc3H7Khg) (2026-05)

## Designing skills around existing systems
- Expose a platform's existing guardrails + APIs as agent skills so the agent uses the same safe paths humans do — don't build bespoke "magic" for the agent. Infrastructure-as-code operations become skills (skills.md as implementation detail); the agent is an alternative interface, not a magic layer [Pragmatic Engineer / Kelsey Hightower](https://newsletter.pragmaticengineer.com/p/kubernetes-and-retiring-at-the-top) (2026-06)
- Skills + a guardrailed CLI beat MCP for most work; MCP's real value is auth, not tool surface [DeepMind](https://www.youtube.com/watch?v=7gujZrJ9L5I) (2026-05)
- Vendor-published skills as integration aid: Google ships official coding-agent skills for its Gemini APIs incl. the Live API — install vendor skills when wiring agents to finicky real-time/streaming APIs; steers the agent where docs alone fail [DeepMind-Schaeff](https://www.youtube.com/watch?v=Bc6Ojl2XS1w) (2026-06)
- Domain skill that maps a structured DSL pays off: Chronon ships a Claude skill where you describe a feature in freeform text and it emits valid DSL feature definitions ("pretty good, maybe slight tweaking") — practitioners expect to stop hand-writing the DSL almost entirely. The leverage is the constrained DSL itself: a structured construct (vs free-form hand-rolled logic) is what makes agent codegen reliable, with correctness invariants (here point-in-time joins) baked into the construct rather than left to the model — designing a clean high-level abstraction + good docs/examples is what lets "any coding assistant just make it happen" (cf. model-level constrained generation in harnesses.md) [Chronon](https://podcasters.spotify.com/pod/show/mlops/episodes/Zipline-Roundtable-episode-Building-Real-Time-ML-Systems-with-Zipline--Chronon-e3ktnth) (2026-06)
- Anthropic ships an MCP-app scaffolding skill in the modelcontextprotocol repo — runnable through any agent CLI to generate working MCP-app templates incl. handlers and tool-visibility config (model-only / model+app / app-only invocation) [MCP Apps](https://www.youtube.com/watch?v=_xIwFcnHqp4) (2026-06)
- Build an MCP app via a packaged skill: push to Claude Code, it generates a spec-compliant app w/o hand-coding (official SDK kept always-compliant) [MCP Apps](https://www.youtube.com/watch?v=o-zkvb0iFDQ) (2026-05)
- Same generic loop (work → push → feedback → iterate) but swap skills to change the loop's focus: ADR skill, PRD skill, UI skill (skips checks, forces fast browser iteration), test skill (selects tests by coverage + changed files vs running whole suite) [Safe Intelligence](https://www.youtube.com/watch?v=504PvfXou5Y) (2026-06)
- Make your agent itself easy for OTHER coding agents to build/test (a "pseudo-RL" loop): expose a CLI, write AGENTS.md, add skills, wire CI/CD so long-running agents can change and end-to-end test it in a parallel thread [Cline](https://www.youtube.com/watch?v=yUmS-F9IX90) (2026-05)
- Wrap a stateful resource in a small CLI so the skill manipulates it without loading the whole thing into context — e.g. an `eval-tool` CLI to list/add/edit/replace one eval case instead of reading a giant YAML suite [incident.io](https://www.youtube.com/watch?v=L2r6vLlLgs8) (2026-05)
- Skill bridging the agent to an existing productivity suite as a trigger source: Microsoft's WorkIQ connects the coding agent to M365 (Outlook, PowerPoint) to pull a feature request straight from email into the terminal as the test trigger [Playwright](https://www.youtube.com/watch?v=FWEInOtngmM) (2026-05)
- Build your own generator/evaluator harness from stock primitives: a custom sub-agent as evaluator (harsh system prompt + detailed rubric), Playwright MCP / Claude-for-Chrome MCP (or computer use for native apps), skills for rubrics, auto mode for safer-than-`--dangerously-skip-permissions` autonomy [Anthropic](https://www.youtube.com/watch?v=mR-WAvEPRwE) (2026-05)
- Make docs cheap for agents to consume: expose an "agent sitemap" the skill reads first (picks the right doc vs looping fetch→think→fetch); serve markdown via Accept header or `.md` URL suffix (agents default to HTML, burn tokens); offer an NL search/RAG endpoint over docs (one query vs 5 fetches + logs what agents struggle with); advertise `--help` to kill hallucinated CLI params; wrap all APIs in a CLI so the agent can do everything a human did in the UI [Langfuse](https://www.youtube.com/watch?v=vNCY9kXXyDQ) (2026-05)
- Expose product docs over SSH so agents browse them as a filesystem (familiar Linux/bash navigation) instead of web-fetch; later augment the bash tools with semantic search (pgvector embeddings) over the docs [Supabase](https://www.youtube.com/watch?v=JT3OzDKrucU) (2026-05)
- Docs-as-skills can beat vector RAG: once docs were published as installable skills, loading them as agentic tool calls outperformed flat chunk-into-vector-DB retrieval (Proser, ex-Pinecone) [WorkOS-Skills](https://www.youtube.com/watch?v=pFsfax19yOM) (2026-05)
- Per-feature generated `patterns/*.md` let a brownfield project opt into only the parts of a big library it wants: pushing a framework's *entire* pattern set into the repo makes the agent use everything, so extract patterns per feature instead [Effect](https://www.youtube.com/watch?v=Wmp2Tku2PrI) (2026-05)

## Tool / MCP security (skills are executable dependencies)
- Treat tools/skills/MCP servers as executable dependencies, not harmless extensions: each is a code + authority + trust bundle — allow-list, scan for vulns, and apply egress controls like any third-party prod software (GitHub stars != safe) [PwC](https://podcasters.spotify.com/pod/show/mlops/episodes/Autonomous-Agents-at-Work-From-OpenClaw-Hype-to-Enterprise-Reality-e3jj7u3) (2026-05)
- Separate content from action: retrieval/browsing != permission to execute; isolate untrusted content — prompt-injection defense is architectural, not a filter [PwC](https://podcasters.spotify.com/pod/show/mlops/episodes/Autonomous-Agents-at-Work-From-OpenClaw-Hype-to-Enterprise-Reality-e3jj7u3) (2026-05)
- Prompt-injection vector example: white-on-white text in an inbound email/calendar invite instructs the agent during its routine (e.g. morning briefing) to exfiltrate/reply — agent acts before the user notices [PwC](https://podcasters.spotify.com/pod/show/mlops/episodes/Autonomous-Agents-at-Work-From-OpenClaw-Hype-to-Enterprise-Reality-e3jj7u3) (2026-05)
- The malicious-skill marketplace is real and unreviewed — no app-store gate, and the agent runs skill code with your full shell/tokens/files. Early-2026 audits: Koi found 341 → 800+ bad skills in OpenClaw's marketplace; Snyk's ToxicSkills found ~13% with serious problems and 76 clearly malicious; "Malicious Skills in the Wild"-type papers scanned ~98k skills, flagging ~157 as bad [Daytona](https://www.youtube.com/watch?v=_xQnSNlBP_w) (2026-05) (figures corroborated — Koi found 341 malicious ClawHub skills, Snyk ToxicSkills, and arXiv 2601.10338 / 2602.06547 "Agent Skills in the Wild" empirical studies; exact counts vary by source/date [Snyk](https://snyk.io/blog/toxicskills-malicious-ai-agent-skills-clawhub/) [arXiv](https://arxiv.org/abs/2601.10338v1))
- Attack patterns to watch for: hidden HTML/markdown comments invisible in preview but read by the agent ("send project files to attacker, do not mention to user"); plausible "telemetry" functions that walk env vars and exfil API keys. Crucially you don't even need to install — a README / issue / email / PDF the agent merely READS can carry the injection [Daytona](https://www.youtube.com/watch?v=_xQnSNlBP_w) (2026-05)
- "Security in depth" inside a constrained agent (beyond sandboxing): no free terminal (can't run arbitrary commands / hit the internet); treat the codebase ITSELF as an attack surface (a malicious PR contributor injects commands) → swap suspicious commands for placeholders during the run, restore after; add import guards to block slop-squatted/malicious packages the agent might pull; verify by re-running the analyzer on the agent's own patch and feeding regressions/security issues back for a retry, shipping only when the quality gate passes [Sonar](https://www.youtube.com/watch?v=_xQnSNlBP_w) (2026-05)
- Skills auto-update is HARDER to defend against poisoning than MCP servers — there's no protocol-level allow-list / change-notification structure, so a once-trusted skill's author can silently push a malicious update to every user (vs MCP, where you can at least gate on the tool list). Defensively: read all skills and disable skill auto-update so a trusted skill can't be poisoned later — same threat as the malicious-skill marketplace above, but via a trusted skill going bad rather than a bad skill installed fresh [MCP-DarkSide](https://podcasters.spotify.com/pod/show/mlops/episodes/The-Dark-Side-of-MCP-Servers-e3l5thf) (2026-06)

## Discovering which skills to write
- Mine your own prompt history for repeated patterns, then codify them into a skill (had Claude classify 116 sessions into 6 categories to surface what to automate) [Sentry](https://www.youtube.com/watch?v=li0SaBt9RDM) (2026-05)
- Make your product agent-native: figure out what agents reliably get wrong, write those gotchas down (watch for client-side JS that hides context from page-summarizing crawlers) [WorkOS](https://www.youtube.com/watch?v=vy7o1g2iHY8) (2026-05)
- The weekly chores you most resist doing are the best skill candidates (e.g. a Slack→Linear loop that reads Slack every 15m, dedups against existing tickets, creates/updates without breaking flow) — and mining your local session JSONL logs with skill-builder surfaces where the workflow was inefficient (10 tool calls that could be 1–2, repeated clarifying questions to pre-answer) [WorkOS-Skills](https://www.youtube.com/watch?v=pFsfax19yOM) (2026-05)
- Instrument your own usage to find where the leverage actually is — in a large mature codebase the dominant use is comprehension, not generation (measured split: 67% comprehension, 2% code gen) [Sentry](https://www.youtube.com/watch?v=li0SaBt9RDM) (2026-05)
- Generate skills with a coding agent (~80 made via Gemini CLI / Claude Code) and add a deploy/test skill (e.g. an ADB skill) so the agent deploys + tests each skill on a real target end-to-end [Function-Gemma](https://www.youtube.com/watch?v=-TiET_K-E_g) (2026-05)
- "Skills to write skills": >half of ~80 skills initially vibe-coded via Gemini CLI / Claude Code / Antigravity; an agent self-tests a skill in-loop (e.g. an ADB skill: deploy to device → exercise → validate → iterate, all from one prompt) [LiteRT-LM](https://www.youtube.com/watch?v=BKWpYIWvAo4) (2026-05)

## Measuring whether a skill helps (eval each one)
- You only know a skill helps or hurts by measuring — run scenarios with/without it and compare pass rates (the feedback signal that caught the 95%-deletion win; "trust is a pass rate, a hash, a delta score"). Treat eval numbers like an Apple Watch — not accurate in absolute terms but a reliable *directional vector*: fail a skill if score-with < score-without, gate on ~80–90% pass-rate, re-check as new models drop and delete skills that add only 1–2% once the base model improves (Claude now ships a built-in HTML before/after eval-report GUI) [WorkOS](https://www.youtube.com/watch?v=vy7o1g2iHY8) (2026-05) [WorkOS-Skills](https://www.youtube.com/watch?v=pFsfax19yOM) (2026-05)
- Prove a skill is good by wiring it into a real shipping product path — WorkOS's CLI `install` brains are entirely the public skills run via the Agent SDK (build the skill and prove it in one move) [WorkOS-Skills](https://www.youtube.com/watch?v=pFsfax19yOM) (2026-05)
- Eval each skill, then sweep models against it — compare accuracy vs token cost to drop to a cheaper/open model for skills you run often [HuggingFace](https://www.youtube.com/watch?v=JomVvNDjGb8) (2026-05)
- LLM-generated, auto-run evals validate a new knowledge entry actually resolves the specific cases that spawned it before it ships [Lovable](https://www.youtube.com/watch?v=KA5kPbdkK2E) (2026-06)
- Blank-injection holdout: for a small % where context would be injected, inject nothing; compare success of injected vs could-have-injected cohorts to measure each entry's real production value [Lovable](https://www.youtube.com/watch?v=KA5kPbdkK2E) (2026-06)
- Eval a skill by diffing filesystem + resulting traces before/after running it on sample repos, with checks as NL LLM-as-judge assertions ("OpenAI instrumentation added", "retrieval spans present"); a basic 5-fixture setup beats none — change the skill without silent regressions [Langfuse](https://www.youtube.com/watch?v=vNCY9kXXyDQ) (2026-05)

## Skill governance & lifecycle
- Skills sprawl in large orgs — curate "Darwinian" so only the best survive; many redundant skills is a liability [DeepMind](https://www.youtube.com/watch?v=7gujZrJ9L5I) (2026-05)
- Two-tier governance: project-maintained skills live in the project (robust, maintainer-owned); experimental/YOLO skills live in a separate repo [HuggingFace](https://www.youtube.com/watch?v=JomVvNDjGb8) (2026-05)
- Skill author owns writing the skill's eval/test; emerging pattern: let agents design those tests (meta-eval) [DeepMind](https://www.youtube.com/watch?v=7gujZrJ9L5I) (2026-05)
- Cluster solved-problem solutions before storing so entries generalize, not one page per exact prompt [Lovable](https://www.youtube.com/watch?v=KA5kPbdkK2E) (2026-06)
- Skill-based harness leaked context as it grew (model "forgot"/skipped tasks); moving orchestration to deterministic code fixed it — skills for knowledge, code for control flow [WorkOS](https://www.youtube.com/watch?v=vy7o1g2iHY8) (2026-05)
- Skills as a package format (context + scripts + docs) lead to context "dependency hell" → version-pin context packages mirroring library versions [Tessl](https://www.youtube.com/watch?v=bSG9wUYaHWU) (2026-05)
- Skill distribution is unsolved (no package manager, installed copies go stale, auto-upgrade harness-dependent): lean toward timestamp-on-install + skill self-flags/re-fetches when older than ~a month, vs per-harness marketplace integrations. Open Q on a getting-started skill's "target function": single-shot the perfect setup vs reach a quick aha and let user re-invoke "improve my setup" [Langfuse](https://www.youtube.com/watch?v=vNCY9kXXyDQ) (2026-05)
- Distribution has no standard registry — options seen: Vercel `skills` package (creates an `.agent/` dir, an emerging cross-tool standard, then symlinks into `.claude/skills` — gripe: installs globally with weak project scoping) [Supabase-SkillIssue](https://www.youtube.com/watch?v=GmAQKINjv1E) (2026-05), model-specific plugins bundled with MCP servers (`.claude`/`.cursor` plugin dirs in-repo). Supabase's approach: package the skill INSIDE the product repo, discoverable if the repo is accessible (skills are an agent-agnostic open standard, so start minimal then iterate/version) [Supabase](https://www.youtube.com/watch?v=JT3OzDKrucU) (2026-05)
- Skills loadable from a URL at runtime; can carry a user-prompted secret/API key [LiteRT-LM](https://www.youtube.com/watch?v=BKWpYIWvAo4) (2026-05)
- Packaging for non-technical teammates: zip the skill folder, rename `.zip`→`.skill`, drag into Claude Desktop (no versioning, no secret handling — keep creds out of the zip). Skills are cross-harness portable — the same SKILL.md runs in Claude Code, Codex, Cursor, Claude Desktop/web, and Pi — so author once, run anywhere [WorkOS-Skills](https://www.youtube.com/watch?v=pFsfax19yOM) (2026-05)
- Engineer maturity ladder: use the agent for everything → automate your work → move automation into a skill → get good at writing skills → improve skills → optimize the environment (architecture/docs) for agents [Intercom](https://www.youtube.com/watch?v=4_VQBbs2iQA) (2026-05)
- Skill quality bar: small, durable, testable, single-job — then backtest it against your existing body of code/incidents/changes to prove it operates at high quality. Treat the eng lifecycle as a factory: break into smallest composable units, one skill per unit (build door vs fit wheel) [Intercom](https://www.youtube.com/watch?v=4_VQBbs2iQA) (2026-05) [PFF](https://www.youtube.com/watch?v=VMemhtlsoNk) (2026-05)
- Prefer durable, testable, written-down "how to do work here" skills over everyone hand-rolling multi-agent orchestrators — specific impls change, the skill stays valuable [Intercom](https://www.youtube.com/watch?v=4_VQBbs2iQA) (2026-05)
- Avoid consuming external skills with strong opinions that clash with your org's conventions — they create friction [PFF](https://www.youtube.com/watch?v=VMemhtlsoNk) (2026-05); concretely, don't blindly drop a SKILL.md found on Twitter into your agent — read it and curate which parts you actually want (a borrowed design-system skill may carry off-brand things), same as not merging a PR nobody reviewed [Magic-Path](https://www.youtube.com/watch?v=m12vGjfbNlo) (2026-05)
- Skill-of-skills structure: a top-level skill + sub-skills (create/observe/...), each sub-skill fired by natural-language trigger words ("evaluate my agent", "help me build a data set"); inspect the installed SKILL.md to see its best-practice checklist + trigger phrases (low-confidence src) [Foundry](https://www.youtube.com/watch?v=iOXM3zE-2dk) (2026-05)
- Only prompt the user before installing a skill if it's more trusted than the public web; at equal trust, auto-discover + install without prompting [Langfuse](https://www.youtube.com/watch?v=vNCY9kXXyDQ) (2026-05)
- Multi-skill scaling limit: a 4B model with ~8 skills chooses well ACROSS turns (skill A→B robust) but calling MULTIPLE skills within a single turn is still unreliable — open problem even with a simple harness [Function-Gemma](https://www.youtube.com/watch?v=-TiET_K-E_g) (2026-05)
- Onboarding via a curated baseline agent image: ship company-approved MCP servers + approved auth + team skills + data access as one image, fan out to the team, then each person personalizes — vs cobbling from a coworker's repo [Red Hat](https://www.youtube.com/watch?v=F1DYkY1BlfM) (2026-05)

## Self-improvement / agent feedback
- Give the agent a "vent" tool to report tooling/docs/schema/platform friction, fired only when genuinely frustrated — tuning the threshold beats an external reviewer forced to comment every turn (overfits to noise); the agent has far more root-cause context than the end user (it lived the failure across turns), so its self-reported friction is more actionable than user complaints [Lovable](https://www.youtube.com/watch?v=KA5kPbdkK2E) (2026-06)
- Vent reports surface bugs invisible to normal monitoring (e.g. copy tool silently failing on filenames with spaces); volume spikes act as emergent incident-detection [Lovable](https://www.youtube.com/watch?v=KA5kPbdkK2E) (2026-06)
- Corroborating the vent tool, cheapest possible agent observability: a free `report` tool + one system-prompt line ("before your final answer, surface anything notable for your operators", posting to Slack/log) catches tests removed-not-fixed, a tool repeatedly failing (the agent "rants" about it in its reasoning), capability gaps (pseudo feature-requests), and sandbox bypass (write-tool permission error → agent falls back to a bash heredoc). Tool naming controls honesty — a polished model won't self-incriminate once the job's done, so a neutral name + "feedback to your creators" framing gets disclosure where "unsafe bash use" suppresses it (cf. OpenAI's confession-training work) [Raindrop](https://www.youtube.com/watch?v=-aM2EDTiaMs) (2026-05) (low-confidence src)
- Highest-signal training samples come from the stuck→unstuck transition; then ask "what should we have injected at the start to skip the friction?" — turn that into a new skill/knowledge entry [Lovable](https://www.youtube.com/watch?v=KA5kPbdkK2E) (2026-06)
- Close the loop with NL feedback the way a manager corrects a human ("you gave wrong info here," "the SOP wasn't clear so it got confused"), then iterate the spec/procedure itself — when the agent is confused the fix is usually the spec, not the model [ElevenLabs](https://podcasters.spotify.com/pod/show/mlops/episodes/Voice-Agent-Use-Cases-e3ileg6) (2026-05)
- Retro agent parses Claude/Codex JSONL transcripts to detect doom loops (same tool call repeated, excessive parallel tools) and writes lessons into per-stack markdown memory files [WorkOS](https://www.youtube.com/watch?v=vy7o1g2iHY8) (2026-05)
- Skill maintenance loop: point the agent at its own session logs to improve the skill, or run skills through a prompt optimizer (GEPA) as a "skills gym"; version skills like dotfiles [OpenClaw](https://www.youtube.com/watch?v=pmoDeA3RBZY) (2026-06)
- Subagent vs skill: reach for a subagent when you want a *separate standalone context* to do bulk work (e.g. review passes) and report back only findings, so it doesn't eat the main task's context window [WorkOS-Skills](https://www.youtube.com/watch?v=pFsfax19yOM) (2026-05)
- Memory-consolidation loop (Pi): per-domain `memory.md` files (general + framework-specific) plus an end-of-run retrospective that distills learnings ("spun in circles — next time run this command"); OpenClaw's daily-journal → consolidated-memory ("dreaming") is reported as surprisingly strong [WorkOS-Skills](https://www.youtube.com/watch?v=pFsfax19yOM) (2026-05)
- Auto-research to improve a skill (agent proposes edits against a target function) works as a human-reviewed idea generator (accepted 3/6) but is only as good as the target: minimizing "turns" backfired — optimizer deleted doc-fetch steps ("I already know this"), reintroducing staleness; anything not rewarded gets stripped as noise [Langfuse](https://www.youtube.com/watch?v=vNCY9kXXyDQ) (2026-05)
- Build skills via a feedback loop, NOT by sitting down enumerating steps: give the agent a goal, work alongside it fixing real cases, let it author the skill (e.g. a flaky-spec fixer with "cheat codes"/lookup tables + progressive disclosure); the end state is self-updating skills in a continuous-improvement loop [Intercom](https://www.youtube.com/watch?v=4_VQBbs2iQA) (2026-05)
- Optimizer loops need a human stop button: an eval/optimize "observe skill" prompt-optimizer regresses chasing a perfect score (7→8→5...), so checkpoint after each step — the agent reports the result and asks "want me to continue?" while a human picks the best version (low-confidence src) [Foundry](https://www.youtube.com/watch?v=iOXM3zE-2dk) (2026-05)
- Harness-engineering discipline: when the agent errs, fix the harness/skill so it can fix itself — never hand-fix the output; every failure is a system bug + data for the next run [WorkOS](https://www.youtube.com/watch?v=vy7o1g2iHY8) (2026-05)

## Plan-and-review as the emerging shape of SWE
- The job collapses to plan + review (code-writing shrinks). Lever: accuracy goes UP the more you plan before launching — basic = use plan mode for everything; advanced = a spec-driven framework that interrogates you with exhaustive questions until every ambiguity is resolved. 5 min planning saves much reviewing/back-and-forth [Vibe-Kanban](https://www.youtube.com/watch?v=m12vGjfbNlo) (2026-05)
- Workflow differs by work TYPE (under-discussed): plan-heavy + parallel-many-agents fits BACK-END (logic easy to express in a common language) and migrations; review-heavy + human-in-loop + sequential fits FRONT-END (interaction/visual requirements hard to fully spec) [Vibe-Kanban](https://www.youtube.com/watch?v=m12vGjfbNlo) (2026-05)

## Agent-driven design context
- For AI to produce on-brand UI, code-backed design-system components (props/variants/tokens) beat Figma mocks / storybook / a design.md as agent context; expose via MCP for a round-trip (prototype tool ↔ Claude Code/Cursor/Codex) so code↔design transfer stays high-fidelity both ways — without it agents hallucinate generic/off-brand SaaS UI [Magic-Patterns](https://www.youtube.com/watch?v=m12vGjfbNlo) (2026-05)
- Layer-level editability as design "structured output": one-shot image gen returns a frozen PNG/MP4 (state resets each prompt); instead isolate output into editable LAYERS so specialized models each own a layer (text vs SVG vs photo) and humans/agents do localized edits — relevant only if doing agent-driven design [Laika](https://www.youtube.com/watch?v=m12vGjfbNlo) (2026-05)
