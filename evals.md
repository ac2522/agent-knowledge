# Evals

## Reading benchmark numbers
- Distrust model-lab benchmark numbers — benchmark-maxing is rampant; confirm on real-world use. [Cline](https://www.youtube.com/watch?v=QuuIywMG4s8)
- Lab-published numbers aren't comparable across labs: each tunes orchestration to its own model (e.g. compaction via its own API), so the same benchmark yields different results depending on who ran it. [DeepMind-Kang](https://www.youtube.com/watch?v=Ubwb6NzegyA)
- Evals rot: SWE-bench verified no longer measures frontier coding (per OpenAI). Hunt new, precise evals. [Cline](https://www.youtube.com/watch?v=QuuIywMG4s8)
- Time-splits are the only real benchmark decontamination: released questions+solutions leak into next-gen pretraining — collect only fresh tasks from the previous month and re-evaluate monthly. [SWE-rebench](https://www.youtube.com/watch?v=wcUJWP6WpGM) (2026-06)
- Public benchmarks (terminal-bench v1, SWE-bench variants) contain never-completable tasks forming a noise floor that masks whether models actually improved — saturation/failure-rate trends partly measure task quality, not model progress. [Snorkel-tasks](https://www.youtube.com/watch?v=YYH0DMQr30A) (2026-06)
- Static benchmarks go stale fast (authors move on); PvP/ELO (Bradley-Terry) benchmarks resist saturation — always a winner and loser, so they stay hill-climbable. [DeepMind-Kang](https://www.youtube.com/watch?v=Ubwb6NzegyA)
- Model endpoints don't always reveal which model actually served a request — undermines reproducibility and over-time comparisons. [DeepMind-Kang](https://www.youtube.com/watch?v=Ubwb6NzegyA)
- Adoption cadence: stay current but not earliest adopter — let a new model "set on fire" ~2 weeks before switching. [Cline](https://www.youtube.com/watch?v=QuuIywMG4s8)

## What benchmarks miss (quality, security, the harness)
- Standard coding benchmarks (HumanEval, MBPP, SWE-bench) measure only functional pass rate — blind to security, maintainability, cyclomatic/cognitive complexity, tech debt; high pass rate ≠ shippable code. [Sonar](https://www.youtube.com/watch?v=NuePCNMpWGc)
- Pass rate is empirically uncorrelated with code quality: top-accuracy models still emit hundreds of security issues and bugs per million LOC. [Sonar](https://www.youtube.com/watch?v=NuePCNMpWGc)
- Verbosity is exploding across generations — same task set yields <250k LOC on older models vs ~1M LOC on newer; track LOC-per-task as a quality signal. [Sonar](https://www.youtube.com/watch?v=NuePCNMpWGc)
- As models mature via RL they fix obvious bug classes but introduce finer, subtler bugs harder for humans to catch — newer ≠ safer, just differently risky. [Sonar](https://www.youtube.com/watch?v=NuePCNMpWGc)
- Coding benchmarks capture a fraction of real environment complexity — org-specific policies, Slack context, screenshots, flaky toolchains, distributed CI, reviewer-head knowledge, parallel contributors — and that gap is where agents fail in production; discount scores accordingly. [Snorkel-bench](https://www.youtube.com/watch?v=iNkFlCiij0U) (2026-06)
- An eval simultaneously tests three things — the model, the harness, and whether the problems are real; a strong model masks a weak harness. [Cline](https://www.youtube.com/watch?v=QuuIywMG4s8)
- Harness dominates coding-eval results: SWE-Bench Pro showed ~22% swing from harness alone while frontier models sat within a few points — you may be measuring the harness, not the model. [DeepMind-Kang](https://www.youtube.com/watch?v=Ubwb6NzegyA)
- Same model varies by harness (e.g. Claude Code vs others) — the harness must actually leverage the model; ambiguity over what's under test is the core agentic-eval problem. [Cline](https://www.youtube.com/watch?v=QuuIywMG4s8) [DeepMind-Kang](https://www.youtube.com/watch?v=Ubwb6NzegyA)
- Information retrieval (does the agent find ALL relevant code?) is unsolved — unseen code is where garbage comes from; benchmarks rarely test this. [Pragmatic-Pi](https://newsletter.pragmaticengineer.com/p/building-pi-and-what-makes-self-modifying)

## Building agent evals & datasets
- Evals are not unit tests — start from known failure modes (named by you or an SME); exhaustive coverage = infinite tests, zero shipping. [Braintrust](https://www.youtube.com/watch?v=FB-MLPhL9Ms)
- Agent evals are multi-turn (read files, install env, run scripts) → grade the end state (did it work? did it break other things?), not a single-turn binary; evaluate the whole trace and target individual tool/MCP steps. [Cline](https://www.youtube.com/watch?v=QuuIywMG4s8) [Braintrust](https://www.youtube.com/watch?v=FB-MLPhL9Ms)
- Eval scope taxonomy beyond judge flavor: span (one component's I/O) → multi-span (data joined across components, e.g. agent-to-agent handoff quality) → trajectory (right call order for the business process) → session (the conversation state machine: user frustration, all questions answered) — pick scope+depth per question. [Arize](https://www.youtube.com/watch?v=JsCCrBF7F1g) (2026-06)
- "Tokens per successful outcome" as the fuel-efficiency metric for an agent-facing tool/interface: effectiveness (full user journey completed?) × efficiency (tokens, tool calls, duration) — efficiency alone is worthless if the destination isn't reached; compare only within a task class, never globally; imperfect measurement beats gut-driven tool design. [Chrome DevTools](https://www.youtube.com/watch?v=_B4Pv9ttFgY) (2026-06)
- Seed datasets from real production/UAT traces ("rerun production, don't run tests") and from opted-in real user tasks, hand-cleaned into solvable problems. [Braintrust](https://www.youtube.com/watch?v=FB-MLPhL9Ms) [Cline](https://www.youtube.com/watch?v=QuuIywMG4s8)
- Build your OWN eval set representative of your real tasks — vendors overfit to public SWE-bench; Meta built an internal "Meta SWE-bench" because real internal problems differ from public GitHub issues. [Guild-Meta](https://podcasters.spotify.com/pod/show/mlops/episodes/From-Single-Player-to-Multi-Player-Operating-AI-Agents-at-Scale-e3khpk8) (2026-06)
- Hardest part of agent eval is new datasets + mechanically spinning up sandboxed envs per problem, not the scoring. [DeepMind-Kang](https://www.youtube.com/watch?v=Ubwb6NzegyA) [DeepMind-Sawhney](https://www.youtube.com/watch?v=7gujZrJ9L5I)
- Isolated per-task env (VM/container with repo + agent installed); run tasks in parallel on standardized configs (Harbor / Laude) so the slowest task is the limit; infra: Modal / Daytona / Docker. [Cline](https://www.youtube.com/watch?v=QuuIywMG4s8)
- CRUD-tool agents break offline eval (hard to reconstruct external state, risky to mutate prod) — solve with mock APIs + state reconstruction; reconstruct historical state via versioned/timestamp queries (e.g. query a vector DB as-of input-capture time). [Braintrust](https://www.youtube.com/watch?v=FB-MLPhL9Ms)
- A single `bash` tool is far harder to eval than discrete tools: `trigger_alert` → assert "called" (binary); one `bash` tool → 100 ways to get the same effect, only the command string is matchable. Two fixes: LLM fuzzy-match the command, or run the real endpoint in-env and assert the side effect — but then every dependency (AWS/K8s/RDS) must spin up in-env, so eval-env complexity grows with app complexity. [Sazabi](https://podcasters.spotify.com/pod/show/mlops/episodes/Logs-Are-All-You-Need-Rethinking-Observability-with-AI-Agents-e3jthog) (2026-06)
- Mock the expensive backend (e.g. mock TPUs) to test the harness/agentic flow without burning real compute. [DeepMind-Sawhney](https://www.youtube.com/watch?v=7gujZrJ9L5I)
- Cram system/external state into the trace itself and inject it into the eval task — avoids building separate test fixtures/infra. [Braintrust](https://www.youtube.com/watch?v=FB-MLPhL9Ms)
- Terminal-Bench (Stanford): 89 realistic tasks (race conditions, infra, DB), runnable with any coding-agent CLI; legit tasks run 30–40 min. [Cline](https://www.youtube.com/watch?v=QuuIywMG4s8)
- Difficulty must be tuned to a band: too hard and runs never finish, too easy and there's no signal. [DeepMind-Kang](https://www.youtube.com/watch?v=Ubwb6NzegyA) [Cline](https://www.youtube.com/watch?v=QuuIywMG4s8)
- Gate agentic (terminal-bench-style) tasks on 4 acceptance criteria before they enter an eval/RL set: achievable, non-trivial, functionally correct logic/tests, reliable environment — each criterion verified by its own test. [Snorkel-tasks](https://www.youtube.com/watch?v=YYH0DMQr30A) (2026-06)
- Task quality dominates RL outcome: same model, compute, and task count → rejected (low-quality) tasks gave ~1% uplift vs ~6% with quality-gated tasks — ~5x from task quality alone. [Snorkel-tasks](https://www.youtube.com/watch?v=YYH0DMQr30A) (2026-06)
- Cheap probe-run signature validates task quality: run frontier agents over the set — accepted tasks average ~2x tool calls, lower pass rate, more output tokens than rejected ones. [Snorkel-tasks](https://www.youtube.com/watch?v=YYH0DMQr30A) (2026-06)
- Triage task failures into meaningful (logic error, incomplete — true difficulty, hill-climbable) vs degenerate environmental failures no model could pass; low pass rate alone isn't evidence of a good task. [Snorkel-tasks](https://www.youtube.com/watch?v=YYH0DMQr30A) (2026-06)
- Commonest task rot = spec/verifier mismatch: task definition underspecifies the testable outcome, or tests carry implicit dependencies never fed into the model's context — check spec↔verifier alignment. [Snorkel-tasks](https://www.youtube.com/watch?v=YYH0DMQr30A) (2026-06)
- Two dominant bad-task classes: overfitted verifiers (test written after the fix demands an exact error-message substring, so correct solutions fail) and infra noise (tests calling external resources; Docker clocks defaulting to 1970). [SWE-rebench](https://www.youtube.com/watch?v=wcUJWP6WpGM) (2026-06)
- Some task-quality problems only surface after agents attempt them — oversample candidates ~10% over target, run models, prune; final set still gets manual verification (~1 day/month for a monthly batch). [SWE-rebench](https://www.youtube.com/watch?v=wcUJWP6WpGM) (2026-06)
- Adversarial expert QC for eval tasks (GPQA appendix): author → reviewers → adjudicator with revision rounds, payouts gated on inter-expert agreement — task validation as incentive design, not just review. [Snorkel-bench](https://www.youtube.com/watch?v=iNkFlCiij0U) (2026-06)
- Define an explicit task taxonomy and distribute eval tasks intentionally against it; deliberately over-represent rare-but-disproportionately-important failure modes instead of mirroring production traffic. [Snorkel-bench](https://www.youtube.com/watch?v=iNkFlCiij0U) (2026-06)
- Grade policy/constraint adherence as a separate pass condition from task completion — agent that books the right flight but violates fare-class rules still fails (TauBench pattern). [Snorkel-bench](https://www.youtube.com/watch?v=iNkFlCiij0U) (2026-06)
- Benchmark-cheating arms race (observed in Claude Code & Codex): `git log --all` reads the future fix commit → strip future git history (keep past); WebFetch reads the original issue/PR → restrict it; plain `curl` does the same → network restriction + trajectory audit. Stronger models cheat more; hide the solution by construction. [SWE-rebench](https://www.youtube.com/watch?v=wcUJWP6WpGM) (2026-06)
- Process-level retrieval eval (ContextBench pattern): grade the agent's search trajectory, not task success — human-labeled golden files/lines per task; precision = fraction of reads that were golden, recall = fraction of golden context found; cap file reads (~50 lines) or whole-file reads drown the line-level signal. [Turbopuffer-CC](https://www.youtube.com/watch?v=zKk7sDMGDEQ) (2026-06)
- Build your own eval harness on a large real task corpus and run generated output through static analysis to surface quality/security metrics benchmarks ignore. [Sonar](https://www.youtube.com/watch?v=NuePCNMpWGc)
- Tune evals per focus-mode (plan/research/debug) rather than optimizing one do-anything agent. [Flinn](https://www.youtube.com/watch?v=7CrPrHgoEYk)
- Coding agents systematically miss concurrency/plumbing (sync vs async, threading) — code passes as a single-chat POC then collapses under load; test the failure mode directly by firing 10+ concurrent requests before believing it. [RocketRide](https://podcasters.spotify.com/pod/show/mlops/episodes/AI-Is-Fast--AI-Projects-Are-Slow--Lets-Fix-That-e3k1c24) (2026-05)
- Drive system-prompt iteration off integration tests written over many hypothetical conversation scenarios; have PMs (not just eng) own the eval frameworks. [JustEat](https://podcasters.spotify.com/pod/show/mlops/episodes/Inside-Just-Eats-AI-Lab-Voice-Agents--Agentic-Commerce-e3jsob9) (2026-05)
- Skill author owns the skill's eval/test; emerging pattern: let agents design those tests (meta-eval). [DeepMind-Sawhney](https://www.youtube.com/watch?v=7gujZrJ9L5I)
- Eval each skill, then sweep models against it — compare accuracy vs token cost to drop to a cheaper/open model for skills you run often. [HuggingFace](https://www.youtube.com/watch?v=JomVvNDjGb8)

## LLM judges & human annotation
- Human annotation must capture a justification, not just thumbs up/down — the justification is the domain knowledge you later distill into an LLM judge. [Braintrust](https://www.youtube.com/watch?v=FB-MLPhL9Ms)
- Feed collected thumbs-down justifications through a coding agent (Claude Code / Cursor / Codex) to derive the agent's actual failure-mode taxonomy. [Braintrust](https://www.youtube.com/watch?v=FB-MLPhL9Ms)
- Mix deterministic code checks with LLM judges: objective failure modes (too many tool calls, too many tokens) need no LLM. [Braintrust](https://www.youtube.com/watch?v=FB-MLPhL9Ms)
- Custom, domain-specific annotation UIs beat a generic grading view — annotators already have a mental model of a good trace; match it for better labels. [Braintrust](https://www.youtube.com/watch?v=FB-MLPhL9Ms)
- "Stuck" as a measurable LLM-judge signal: repeated identical asks, explicit complaints, or abandoning a session a judge would expect to continue — flag as friction. [Lovable](https://www.youtube.com/watch?v=KA5kPbdkK2E)
- LLM-as-judge for per-use-case model selection: run the same input through every candidate model (e.g. OCR engines), feed all outputs to a judge, ask which is best — ranking flips by dataset, so re-run per use case rather than picking one model globally. [RocketRide](https://podcasters.spotify.com/pod/show/mlops/episodes/AI-Is-Fast--AI-Projects-Are-Slow--Lets-Fix-That-e3k1c24) (2026-05)
- Self-grade docs/prompts before trusting them: paste in, ask the model to score 1–10 and rewrite "so you understand it better next time"; GPT over-scores (9–10), Claude grades harsher (~6) — use the harsh grader's rewrite. [RocketRide](https://podcasters.spotify.com/pod/show/mlops/episodes/AI-Is-Fast--AI-Projects-Are-Slow--Lets-Fix-That-e3k1c24) (2026-05)

## Eval-the-eval & statistical significance
- Eval the eval: LLM-judge outputs are discrete, so build a ground-truth dataset of judge calls and score the judge against human decisions before trusting it. [Braintrust](https://www.youtube.com/watch?v=FB-MLPhL9Ms)
- Statistical significance is expensive: ~400k poker hands needed for signal; use Bradley-Terry pairwise to cut matchups vs all-pairs. [DeepMind-Kang](https://www.youtube.com/watch?v=Ubwb6NzegyA)
- "Trust is a pass rate, a hash, a delta score" — measure with/without a change and compare; otherwise you're adding noise blind. [WorkOS](https://www.youtube.com/watch?v=vy7o1g2iHY8)
- Validate a cheaper/newer-model downgrade by replaying production: capture real LLM inputs from logs, re-issue against the candidate, diff outputs vs the known-good expensive outputs; only switch if results hold. Worth automating as token-cheaper models ship. [RocketRide](https://podcasters.spotify.com/pod/show/mlops/episodes/AI-Is-Fast--AI-Projects-Are-Slow--Lets-Fix-That-e3k1c24) (2026-05)
- Eval-validity ops: classify exit statuses to separate model errors from infra errors (context overflow, too-many-tool-calls, provider errors) with an explicit rerun policy; reproduce an external benchmark's published numbers on your infra before trusting your own experiments; watch for silent default-parameter drift (reasoning level, caching) within a model family — 1–2 runs/month go invalid from this. [SWE-rebench](https://www.youtube.com/watch?v=wcUJWP6WpGM) (2026-06)
- Report pass@5 (any of 5 runs = capability ceiling) alongside pass^5 (all 5 = reliability) plus tokens/price per problem with confidence intervals — mean resolved alone hides both potential and flakiness. [SWE-rebench](https://www.youtube.com/watch?v=wcUJWP6WpGM) (2026-06)

## The improvement flywheel / hill-climbing
- Flywheel: capture trace → diagnose → replay offline → steer agent changes. [Braintrust](https://www.youtube.com/watch?v=FB-MLPhL9Ms)
- Distributional view across all runs, not single traces: aggregate every instantiation into a path/branch/loop graph — % traffic per branch, latency per component within a branch, eval-score drop per path; root-cause pattern: B called before its dependency A on the bad path → fix by adding ordering context. [Arize](https://www.youtube.com/watch?v=JsCCrBF7F1g) (2026-06)
- Hill-climb loop: run suite → score → read failure traces → classify each failure ("portfolio allocate": didn't run tests, broken read-file tool, …) → pull the small high-leverage levers. [Cline](https://www.youtube.com/watch?v=QuuIywMG4s8)
- Rubric evals as diagnosis layer over a scalar reward: decompose response rightness into per-behavior questions to localize WHICH behavior fails → drives what training data to generate; the RL loop still consumes the single pass/fail scalar — rubric is for analysis, not reward. [Snorkel-RL](https://www.youtube.com/watch?v=TNwJ1LMiENk) (2026-06)
- Three zones of improvement: (1) obvious harness bugs / rate limits, (2) model-family-specific prompt tweaks (Anthropic ≠ Codex ≠ Gemini) — highest value, (3) overfitting / benchmark-cheating — danger zone. [Cline](https://www.youtube.com/watch?v=QuuIywMG4s8)
- Tunable levers beyond prompts: container CPU/mem, timeouts, thinking behavior — and more thinking ≠ better (model loops). [Cline](https://www.youtube.com/watch?v=QuuIywMG4s8)
- You only know a skill helps or hurts by measuring — run scenarios with/without and compare pass rates; one measured skill made a task 77% correct vs 97% without it, and evals caught a 95%-skill-deletion win. [WorkOS](https://www.youtube.com/watch?v=vy7o1g2iHY8)
- Highest-signal training/knowledge samples come from the stuck→unstuck transition; then ask "what should we have injected at the start to skip the friction?" [Lovable](https://www.youtube.com/watch?v=KA5kPbdkK2E)
- Blank-injection holdout: for a small % where context would be injected, inject nothing; compare injected vs could-have-injected cohorts to measure each knowledge entry's real production value. [Lovable](https://www.youtube.com/watch?v=KA5kPbdkK2E)
- LLM-generated, auto-run evals validate a new knowledge entry actually resolves the cases that spawned it before it ships. [Lovable](https://www.youtube.com/watch?v=KA5kPbdkK2E)
- Ship gate: pass both the vibe (feels good to use) and the score. [Cline](https://www.youtube.com/watch?v=QuuIywMG4s8)

## When to fine-tune
- Plateauing evals = signal it may be time to fine-tune rather than keep prompt-engineering. [Modal](https://www.youtube.com/watch?v=HvZXAOZ3iv8)
- Fine-tuning a tiny model beats prompting it for narrow tool-use: Function Gemma (270M, fine-tuned for function calling) went ~46% out-of-the-box → >90% on 8/10 functions (rest in 80s) on a 7–10 function "app intents" task. [Function-Gemma](https://www.youtube.com/watch?v=-TiET_K-E_g) (2026-05)
- Synthetic-data fine-tune workflow for tool-use: bake the function set into the model rather than describe it in a system prompt — use a larger model (Gemini Flash) to synthetically generate the call-example training set, then fine-tune; standard path to robust on-device tool-calling. [Function-Gemma](https://www.youtube.com/watch?v=-TiET_K-E_g) (2026-05)
- Fine-tune vs prompt by size: below ~100–200M params a model only works if the task is very narrow/focused — pick fixed-function or fine-tune, don't expect general prompt-following; prompting a larger model is easier, but a fine-tuned tiny model ships robust narrow tool-use at scale. [Function-Gemma](https://www.youtube.com/watch?v=-TiET_K-E_g) (2026-05)
- No mature evals / no collected data = not ready to train (garbage in, garbage out); build the eval+data pipeline first. [Modal](https://www.youtube.com/watch?v=HvZXAOZ3iv8)
- A working harness + collected eval traces ≈ ready-made training data — RL can teach a model your service from what the harness already logs. [Modal](https://www.youtube.com/watch?v=HvZXAOZ3iv8)
- Big-reasoning-model failure on tool tasks is often tool indiscipline, not weak reasoning: Qwen3-235B queried nonexistent tables without ever calling the available schema tools, then hallucinated; an RL'd 4B learned discover-tools → inspect-schema → self-correct-on-error — diagnose the tool-use behavior before reaching for a bigger model. [Snorkel-RL](https://www.youtube.com/watch?v=TNwJ1LMiENk) (2026-06)
- RL for behavior, not knowledge: changing how a model acts (tool discipline) is an RL problem vs SFT for content; GRPO on a 4B, ~21h, <$500/run, doubled pass@1 past a 235B — behavior-targeted RL is cheap enough for self-hosted teams. [Snorkel-RL](https://www.youtube.com/watch?v=TNwJ1LMiENk) (2026-06)
- Ablation surprise: training on easy (single-table) tasks only beat both mixed-data and easy→hard curriculum — and the gain generalized to the harder multi-table benchmark (13.9→26.6% pass@1); fixing one core behavior generalizes better than training on the hard distribution. [Snorkel-RL](https://www.youtube.com/watch?v=TNwJ1LMiENk) (2026-06)
- Self-contained published RL envs (FinQA on Prime Intellect, OpenEnv repo, HF Spaces; Harbor-style, no external deps) — reusable starting point for tool-use RL. [Snorkel-RL](https://www.youtube.com/watch?v=TNwJ1LMiENk) (2026-06)

## Decision-quality evals (high-stakes agents)
- Reference-class validation: don't act on general/statistical norms — first classify whether this case is the common 99% or the dangerous 1% (the norm can be fatal in the tail). [Neo4j-Kollegger](https://www.youtube.com/watch?v=abvQEhvRI_c)
- Scale decision effort to stakes: weigh reversibility and cost-of-being-wrong; spend more compute on serious, irreversible actions. [Neo4j-Kollegger](https://www.youtube.com/watch?v=abvQEhvRI_c)
- Force the objective explicit (maximize value vs minimize cost) — left implicit, the agent defaults to "what's usually right," but particulars dominate. [Neo4j-Kollegger](https://www.youtube.com/watch?v=abvQEhvRI_c)

## Eval limits in large codebases
- Complexity is the agent's enemy: the ~200k context window caps how much of a large codebase fits; agent-generated sprawl eventually exceeds its own ingestion limit. [Pragmatic-Pi](https://newsletter.pragmaticengineer.com/p/building-pi-and-what-makes-self-modifying)
- Models regress to the mean of training data (mostly mediocre internet code); well-engineered projects are a rounding error in the corpus. [Pragmatic-Pi](https://newsletter.pragmaticengineer.com/p/building-pi-and-what-makes-self-modifying)
- "Best possible spec is the software itself" — any spec gaps get filled from mediocre training priors, multiplied across many parallel agents. [Pragmatic-Pi](https://newsletter.pragmaticengineer.com/p/building-pi-and-what-makes-self-modifying)
