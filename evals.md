# Evals

## Reading benchmark numbers
- Distrust model-lab benchmark numbers — benchmark-maxing is rampant; confirm on real-world use. [Cline](https://www.youtube.com/watch?v=QuuIywMG4s8) (2026-06)
- Lab-published numbers aren't comparable across labs: each tunes orchestration to its own model (e.g. compaction via its own API), so the same benchmark yields different results depending on who ran it. [DeepMind-Kang](https://www.youtube.com/watch?v=Ubwb6NzegyA) (2026-05)
- Evals rot: SWE-bench verified no longer measures frontier coding (per OpenAI). Hunt new, precise evals. [Cline](https://www.youtube.com/watch?v=QuuIywMG4s8) (2026-06)
- Time-splits are the only real benchmark decontamination: released questions+solutions leak into next-gen pretraining — collect only fresh tasks from the previous month and re-evaluate monthly. [SWE-rebench](https://www.youtube.com/watch?v=wcUJWP6WpGM) (2026-06)
- Public benchmarks (terminal-bench v1, SWE-bench variants) contain never-completable tasks forming a noise floor that masks whether models actually improved — saturation/failure-rate trends partly measure task quality, not model progress. [Snorkel-tasks](https://www.youtube.com/watch?v=YYH0DMQr30A) (2026-06)
- Static benchmarks go stale fast (authors move on); PvP/ELO (Bradley-Terry) benchmarks resist saturation — always a winner and loser, so they stay hill-climbable. [DeepMind-Kang](https://www.youtube.com/watch?v=Ubwb6NzegyA) (2026-05)
- Model endpoints don't always reveal which model actually served a request — undermines reproducibility and over-time comparisons. [DeepMind-Kang](https://www.youtube.com/watch?v=Ubwb6NzegyA) (2026-05)
- Adoption cadence: stay current but not earliest adopter — let a new model "set on fire" ~2 weeks before switching. [Cline](https://www.youtube.com/watch?v=QuuIywMG4s8) (2026-06)
- ...but once you've tuned prompts/instructions to a specific model, treat a major version bump like an on-prem upgrade you can defer — a new model can break a setup that worked, so pin the version for tuned/production agents and re-validate before cutting over ("if it ain't broke" applies to LLMs). Reconciles with the cadence above: explore new models promptly, but gate the production cutover on a re-run of your evals. [CI/CD Erez](https://newsletter.pragmaticengineer.com/p/cicd-with-robert-erez) (2026-06)

## What benchmarks miss (quality, security, the harness)
- Standard coding benchmarks (HumanEval, MBPP, SWE-bench) measure only functional pass rate — blind to security, maintainability, cyclomatic/cognitive complexity, tech debt; high pass rate ≠ shippable code. Pass rate is empirically uncorrelated with code quality: top-accuracy models still emit hundreds of security issues and bugs per million LOC. [Sonar](https://www.youtube.com/watch?v=NuePCNMpWGc)
- Verbosity is exploding across generations — same task set yields <250k LOC on older models vs ~1M LOC on newer; track LOC-per-task as a quality signal. [Sonar](https://www.youtube.com/watch?v=NuePCNMpWGc)
- As models mature via RL they fix obvious bug classes but introduce finer, subtler bugs harder for humans to catch — newer ≠ safer, just differently risky. [Sonar](https://www.youtube.com/watch?v=NuePCNMpWGc)
- Coding benchmarks capture a fraction of real environment complexity — org-specific policies, Slack context, screenshots, flaky toolchains, distributed CI, reviewer-head knowledge, parallel contributors — and that gap is where agents fail in production; discount scores accordingly. [Snorkel-bench](https://www.youtube.com/watch?v=iNkFlCiij0U) (2026-06)
- An eval simultaneously tests three things — the model, the harness, and whether the problems are real; a strong model masks a weak harness, and ambiguity over what's under test is the core agentic-eval problem. Harness dominates coding-eval results: SWE-Bench Pro showed ~22% swing from harness alone while frontier models sat within a few points (same model varies by harness, e.g. Claude Code vs others) — the harness must actually leverage the model, or you're measuring the harness, not the model. [Cline](https://www.youtube.com/watch?v=QuuIywMG4s8) (2026-06) [DeepMind-Kang](https://www.youtube.com/watch?v=Ubwb6NzegyA) (2026-05)
- Information retrieval (does the agent find ALL relevant code?) is unsolved — unseen code is where garbage comes from; benchmarks rarely test this. [Pragmatic-Pi](https://newsletter.pragmaticengineer.com/p/building-pi-and-what-makes-self-modifying) (2026-04)

## Building agent evals & datasets
- Evals are not unit tests — start from known failure modes (named by you or an SME); exhaustive coverage = infinite tests, zero shipping. [Braintrust](https://www.youtube.com/watch?v=FB-MLPhL9Ms) (2026-05)
- ...but structure them LIKE unit tests: prompt + input → output → grading criteria (pass/fail), stored as YAML next to the prompts in the same language as the code (Go); keep cases focused/minimal, not giant fixtures. Anti-pattern: production eval cases as raw multi-MB YAML blobs (whole incident report) — unmaintainable and they blow the coding-agent context window on read/edit. [incident.io](https://www.youtube.com/watch?v=L2r6vLlLgs8) (2026-05)
- Grading criteria = multiple independent checks (e.g. "looks like pirate speak" AND "meaning preserved input→output"), each asserted separately. [incident.io](https://www.youtube.com/watch?v=L2r6vLlLgs8) (2026-05)
- "Steal an eval from production" button: one-click pull a failing real interaction into the eval suite as a test case. [incident.io](https://www.youtube.com/watch?v=L2r6vLlLgs8) (2026-05)
- Validation contract = assertions written DURING planning, BEFORE code, implementation-independent (hundreds for a complex project), each feature → ≥1 assertion — counters "tests written after merely confirm the decisions you already made." [Factory-Alvoeiro](https://www.youtube.com/watch?v=ow1we5PzK-o) (2026-05)
- Two unseen-code adversarial validators per milestone: a scrutiny validator (tests/typecheck/lint + per-feature code-review agents) and a user-testing validator (QA via computer-use on the live app); first-pass validation reliably fails — the QA loop IS the value, not the green check. [Factory-Alvoeiro](https://www.youtube.com/watch?v=ow1we5PzK-o) (2026-05)
- Agent evals are multi-turn (read files, install env, run scripts) → grade the end state (did it work? did it break other things?), not a single-turn binary; evaluate the whole trace and target individual tool/MCP steps. [Cline](https://www.youtube.com/watch?v=QuuIywMG4s8) (2026-06) [Braintrust](https://www.youtube.com/watch?v=FB-MLPhL9Ms) (2026-05)
- Eval scope taxonomy beyond judge flavor: span (one component's I/O) → multi-span (data joined across components, e.g. agent-to-agent handoff quality) → trajectory (right call order for the business process) → session (the conversation state machine: user frustration, all questions answered) — pick scope+depth per question. [Arize](https://www.youtube.com/watch?v=JsCCrBF7F1g) (2026-06)
- "Tokens per successful outcome" as the fuel-efficiency metric for an agent-facing tool/interface: effectiveness (full user journey completed?) × efficiency (tokens, tool calls, duration) — efficiency alone is worthless if the destination isn't reached; compare only within a task class, never globally; imperfect measurement beats gut-driven tool design. [Chrome DevTools](https://www.youtube.com/watch?v=_B4Pv9ttFgY) (2026-06)
- Seed datasets from real production/UAT traces ("rerun production, don't run tests") and from opted-in real user tasks, hand-cleaned into solvable problems. [Braintrust](https://www.youtube.com/watch?v=FB-MLPhL9Ms) (2026-05) [Cline](https://www.youtube.com/watch?v=QuuIywMG4s8) (2026-06)
- Build your OWN eval set representative of your real tasks — vendors overfit to public SWE-bench; Meta built an internal "Meta SWE-bench" because real internal problems differ from public GitHub issues. [Guild-Meta](https://podcasters.spotify.com/pod/show/mlops/episodes/From-Single-Player-to-Multi-Player-Operating-AI-Agents-at-Scale-e3khpk8) (2026-06) — for dialogue/voice there are no good E2E multi-turn benchmarks (TauBench is far easier than real conditions), so an internal benchmark on your real data is the only way to capture your distribution/edge cases. [ElevenLabs](https://podcasters.spotify.com/pod/show/mlops/episodes/Voice-Agent-Use-Cases-e3ileg6) (2026-05)
- Curate a small named-scenario suite from real production failures, not synthetic: chess-coach used ~16 hand-built cases (tactical patterns, blunders, hallucination-limiting), each asserting the LLM mentions/understands a specific thing given the context; LLM-as-judge scores, a domain-expert (SME) gives final say (the evaluator need not be the builder), and models run via OpenRouter so the harness is model-agnostic and new models swap in cheaply. More "thinking" lost on this grounded translation task (heavier reasoning model = lower pass rate + much higher latency) — re-run the suite per release and pick the model per task. [Take Take Take](https://www.youtube.com/watch?v=FlzpEGHNVKQ) (2026-05)
- Failure compounds with conversation length: longer multi-turn trajectories = more failure chances, so treat "recovery from a wrong turn" as its own eval dimension — can the agent self-correct after a misunderstanding and ask the right clarifying question at the right time (asking too eagerly erodes trust), not just first-turn correctness. [ElevenLabs](https://podcasters.spotify.com/pod/show/mlops/episodes/Voice-Agent-Use-Cases-e3ileg6) (2026-05)
- Long-session evals to catch late-conversation degradation: context bugs surface late (turn 15-20+) after a user has walked through many pages in one chat — too late if you wait for user reports. Make it testable — load N turns (e.g. 10) and assert on turn N+1 — turning "does the context strategy hold over long sessions?" into a repeatable signal. [Arize-context](https://www.youtube.com/watch?v=esY99nYXxR4) (2026-05)
- Hardest part of agent eval is new datasets + mechanically spinning up sandboxed envs per problem, not the scoring. [DeepMind-Kang](https://www.youtube.com/watch?v=Ubwb6NzegyA) (2026-05) [DeepMind-Sawhney](https://www.youtube.com/watch?v=7gujZrJ9L5I) (2026-05)
- Isolated per-task env (VM/container with repo + agent installed); run tasks in parallel on standardized configs (Harbor / Laude) so the slowest task is the limit; infra: Modal / Daytona / Docker. [Cline](https://www.youtube.com/watch?v=QuuIywMG4s8) (2026-06)
- CRUD-tool agents break offline eval (hard to reconstruct external state, risky to mutate prod) — solve with mock APIs + state reconstruction; reconstruct historical state via versioned/timestamp queries (e.g. query a vector DB as-of input-capture time). [Braintrust](https://www.youtube.com/watch?v=FB-MLPhL9Ms) (2026-05)
- A single `bash` tool is far harder to eval than discrete tools: `trigger_alert` → assert "called" (binary); one `bash` tool → 100 ways to get the same effect, only the command string is matchable. Two fixes: LLM fuzzy-match the command, or run the real endpoint in-env and assert the side effect — but then every dependency (AWS/K8s/RDS) must spin up in-env, so eval-env complexity grows with app complexity. [Sazabi](https://podcasters.spotify.com/pod/show/mlops/episodes/Logs-Are-All-You-Need-Rethinking-Observability-with-AI-Agents-e3jthog) (2026-06)
- Avoid building separate test fixtures/infra: cram system/external state into the trace itself and inject it into the eval task [Braintrust](https://www.youtube.com/watch?v=FB-MLPhL9Ms) (2026-05); mock the expensive backend (e.g. mock TPUs) to test the harness/agentic flow without burning real compute. [DeepMind-Sawhney](https://www.youtube.com/watch?v=7gujZrJ9L5I) (2026-05)
- Terminal-Bench (Stanford): 89 realistic tasks (race conditions, infra, DB), runnable with any coding-agent CLI; legit tasks run 30–40 min. [Cline](https://www.youtube.com/watch?v=QuuIywMG4s8) (2026-06)
- Difficulty must be tuned to a band: too hard and runs never finish, too easy and there's no signal. [DeepMind-Kang](https://www.youtube.com/watch?v=Ubwb6NzegyA) (2026-05) [Cline](https://www.youtube.com/watch?v=QuuIywMG4s8) (2026-06)
- Gate agentic (terminal-bench-style) tasks on 4 acceptance criteria before they enter an eval/RL set: achievable, non-trivial, functionally correct logic/tests, reliable environment — each criterion verified by its own test. [Snorkel-tasks](https://www.youtube.com/watch?v=YYH0DMQr30A) (2026-06)
- Task quality dominates RL outcome: same model, compute, and task count → rejected (low-quality) tasks gave ~1% uplift vs ~6% with quality-gated tasks — ~5x from task quality alone. Cheap probe-run signature validates quality: run frontier agents over the set — accepted tasks average ~2x tool calls, lower pass rate, more output tokens than rejected ones. [Snorkel-tasks](https://www.youtube.com/watch?v=YYH0DMQr30A) (2026-06)
- Triage task failures into meaningful (logic error, incomplete — true difficulty, hill-climbable) vs degenerate environmental failures no model could pass; low pass rate alone isn't evidence of a good task. [Snorkel-tasks](https://www.youtube.com/watch?v=YYH0DMQr30A) (2026-06)
- Commonest task rot = spec/verifier mismatch: task underspecifies the testable outcome, or tests carry implicit deps never fed into the model's context — check spec↔verifier alignment. Two dominant bad-task classes: overfitted verifiers (test written after the fix demands an exact error-message substring, so correct solutions fail) and infra noise (tests calling external resources; Docker clocks defaulting to 1970). [Snorkel-tasks](https://www.youtube.com/watch?v=YYH0DMQr30A) (2026-06) [SWE-rebench](https://www.youtube.com/watch?v=wcUJWP6WpGM) (2026-06)
- Some task-quality problems only surface after agents attempt them — oversample candidates ~10% over target, run models, prune; final set still gets manual verification (~1 day/month for a monthly batch). [SWE-rebench](https://www.youtube.com/watch?v=wcUJWP6WpGM) (2026-06)
- Adversarial expert QC for eval tasks (GPQA appendix): author → reviewers → adjudicator with revision rounds, payouts gated on inter-expert agreement — task validation as incentive design, not just review. [Snorkel-bench](https://www.youtube.com/watch?v=iNkFlCiij0U) (2026-06)
- Define an explicit task taxonomy and distribute eval tasks intentionally against it; deliberately over-represent rare-but-disproportionately-important failure modes instead of mirroring production traffic. [Snorkel-bench](https://www.youtube.com/watch?v=iNkFlCiij0U) (2026-06)
- Grade policy/constraint adherence as a separate pass condition from task completion — agent that books the right flight but violates fare-class rules still fails (TauBench pattern). [Snorkel-bench](https://www.youtube.com/watch?v=iNkFlCiij0U) (2026-06)
- Tag each eval as a guardrail eval (ship-blocker, e.g. hallucinated stock price) vs a north-star eval (nice-to-have) so a soft-fail doesn't block ship. [Arize](https://www.youtube.com/watch?v=Xfl50508LZM) (2026-05)
- Spend eval budget on the capability eval (push the model, an expensive judge is OK) and compress regression evals — usually drop ~80% of regressions and keep a representative sample; on live/production traces don't run capability evals at all (nothing's changing). [Arize](https://www.youtube.com/watch?v=Xfl50508LZM) (2026-05)
- CI eval cost control: on prompt-change PRs run only a small subset of the eval dataset; run the full set (300–500+ rows) only on merge to main — keeps expensive behavioral/tool-call evals affordable. [Databricks](https://www.youtube.com/watch?v=ObTPqBGsEbA) (2026-06)
- Benchmark-cheating arms race (observed in Claude Code & Codex): `git log --all` reads the future fix commit → strip future git history (keep past); WebFetch reads the original issue/PR → restrict it; plain `curl` does the same → network restriction + trajectory audit. Stronger models cheat more; hide the solution by construction. [SWE-rebench](https://www.youtube.com/watch?v=wcUJWP6WpGM) (2026-06)
- Process-level retrieval eval (ContextBench pattern): grade the agent's search trajectory, not task success — human-labeled golden files/lines per task; precision = fraction of reads that were golden, recall = fraction of golden context found; cap file reads (~50 lines) or whole-file reads drown the line-level signal. [Turbopuffer-CC](https://www.youtube.com/watch?v=zKk7sDMGDEQ) (2026-06)
- Build your own eval harness on a large real task corpus and run generated output through static analysis to surface quality/security metrics benchmarks ignore. [Sonar](https://www.youtube.com/watch?v=NuePCNMpWGc)
- Library-vendor codegen-quality eval loop: run evals once/day that regenerate reports so any docs/example/library change shows whether agent output got better or worse — method is human-written best-practice code vs model-generated, LLM-judge scoring similarity; can feed RL to fine-tune an open model on the library's patterns. [Effect](https://www.youtube.com/watch?v=Wmp2Tku2PrI) (2026-05)
- Tune evals per focus-mode (plan/research/debug) rather than optimizing one do-anything agent. [Flinn](https://www.youtube.com/watch?v=7CrPrHgoEYk) (2026-05)
- Coding agents systematically miss concurrency/plumbing (sync vs async, threading) — code passes as a single-chat POC then collapses under load; test the failure mode directly by firing 10+ concurrent requests before believing it. [RocketRide](https://podcasters.spotify.com/pod/show/mlops/episodes/AI-Is-Fast--AI-Projects-Are-Slow--Lets-Fix-That-e3k1c24) (2026-05)
- Drive system-prompt iteration off integration tests written over many hypothetical conversation scenarios; have PMs (not just eng) own the eval frameworks. [JustEat](https://podcasters.spotify.com/pod/show/mlops/episodes/Inside-Just-Eats-AI-Lab-Voice-Agents--Agentic-Commerce-e3jsob9) (2026-05)
- Skill author owns the skill's eval/test; emerging pattern: let agents design those tests (meta-eval). [DeepMind-Sawhney](https://www.youtube.com/watch?v=7gujZrJ9L5I) (2026-05)
- Eval each skill, then sweep models against it — compare accuracy vs token cost to drop to a cheaper/open model for skills you run often. [HuggingFace](https://www.youtube.com/watch?v=JomVvNDjGb8) (2026-05)
- Eval the skill DOCUMENT itself: run a markdown doc through evals like CI but grade agent BEHAVIOR (tools called, reasoning), not code — setup was 6 scenarios × 4 agents from 2 vendors × 3 conditions (baseline / MCP only / MCP+skill), scored on a 4-grade test-completeness score (Braintrust; models: Claude Code Opus 4.6 & Sonnet 4.6, Codex GPT-5.4 & GPT-5.4-mini). MCP+skill beat every other condition on every model. [Supabase](https://www.youtube.com/watch?v=JT3OzDKrucU) (2026-05)
- Skill-eval cycle = OpenAI's "Systematically Evaluate Agent Skills" (metrics → write → eval → grade → iterate, mirrors TDD); concrete local harness uses agent-skills `eval.json` (prompt + expected + assertions), runs two conditions via claude print/headless-as-binary, a DB-reset script between runs, and a grading.json diff; assertions are deterministic OR LLM-judge. [Supabase-SkillIssue](https://www.youtube.com/watch?v=GmAQKINjv1E) (2026-05)
- Eval bugs are harder to catch than test bugs: a mis-written assertion reports inconsistent pass/fail across non-deterministic runs, and a judge can hallucinate grades. The hardest part of skill-evals is authoring representative scenarios (it requires already knowing the expected behavior), not building the harness. [Supabase-SkillIssue](https://www.youtube.com/watch?v=GmAQKINjv1E) (2026-05)
- Write evals for CONTEXT, don't YOLO it — even a 2-line CLAUDE.md edit is a change to test. Escalating context-testing levels: lint (skill format/spec) → "grammarly" (ask the agent if it understands) → LLM-judge vs company rules → judge + tools/sandbox-agent/end-to-end curl test → commit-scoped eval scenarios. [Context-Debois](https://www.youtube.com/watch?v=bSG9wUYaHWU) (2026-05)
- Consistency-as-eval (audience idea): run N parallel definitions of the same context; cross-agreement signals spec quality. [Context-Debois](https://www.youtube.com/watch?v=bSG9wUYaHWU) (2026-05) (low-confidence src)

### Functionality coverage vs code coverage (behavioral E2E)
- AI-generated tests are often "self-affirming": the unit suite is green but system behavior was never validated — code coverage ≠ functionality coverage. [Playwright](https://www.youtube.com/watch?v=FWEInOtngmM) (2026-05)
- Prefer behavioral/E2E tests over unit tests tied to implementation details (renaming a `calculate` method breaks impl-tied tests though behavior is unchanged); test against stable contracts (public API, exported module) so tests survive refactors. [Playwright](https://www.youtube.com/watch?v=FWEInOtngmM) (2026-05)
- Playwright as an agent-driven E2E verification loop: agent opens the page, types inputs, clicks filters, asserts results — confirms the app actually works, not just that asserts pass; auto-screenshots every run, attach them to the PR as functionality evidence. [Playwright](https://www.youtube.com/watch?v=FWEInOtngmM) (2026-05)
- One test per feature; trigger a test on a feature request (behavior), not on adding a class method. [Playwright](https://www.youtube.com/watch?v=FWEInOtngmM) (2026-05)

## LLM judges & human annotation
- Human annotation must capture a justification, not just thumbs up/down — the justification is the domain knowledge you later distill into an LLM judge; feed collected thumbs-down justifications through a coding agent (Claude Code / Cursor / Codex) to derive the agent's actual failure-mode taxonomy. [Braintrust](https://www.youtube.com/watch?v=FB-MLPhL9Ms) (2026-05)
- Best implicit feedback signal = what the user did NEXT, not thumbs up/down (almost nobody clicks, and adversarial users click "bad"): "No, I meant X" / re-asking = strong negative; "thanks" or leaving = positive (cf. Google ranking fast return-to-search as a bad page). [Pydantic-Colvin](https://www.youtube.com/watch?v=A48uhxfxbsM) (2026-05)
- Privacy-constrained evals (legal/medical, e.g. Harvey/Legora): never exfiltrate inputs/outputs — record only categorical performance (good/bad, up to ~50 grades) so you can eval without leaking generated content. [Pydantic-Colvin](https://www.youtube.com/watch?v=A48uhxfxbsM) (2026-05)
- Mix deterministic code checks with LLM judges: objective failure modes (too many tool calls, too many tokens) need no LLM. [Braintrust](https://www.youtube.com/watch?v=FB-MLPhL9Ms) (2026-05)
- Custom, domain-specific annotation UIs beat a generic grading view — annotators already have a mental model of a good trace; match it for better labels. [Braintrust](https://www.youtube.com/watch?v=FB-MLPhL9Ms) (2026-05)
- Use the LLM-judge as a first-pass filter over conversations, then deep-dive the actual transcripts — read WHY a result was good/bad, don't trust the score alone. [iFood](https://podcasters.spotify.com/pod/show/mlops/episodes/The-Latency-Goldilocks-Zone-Explained-e3j384v) (2026-05)
- Survey/self-report metrics mislead (e.g. Sean Ellis "how would you feel if this went away" PMF score): users answer inconsistently — ground truth comes from reading real conversations, not surveyed sentiment. [iFood](https://podcasters.spotify.com/pod/show/mlops/episodes/The-Latency-Goldilocks-Zone-Explained-e3j384v) (2026-05)
- "Stuck" as a measurable LLM-judge signal: repeated identical asks, explicit complaints, or abandoning a session a judge would expect to continue — flag as friction. [Lovable](https://www.youtube.com/watch?v=KA5kPbdkK2E) (2026-06)
- Cheap frustration signal: Claude Code's leaked `user_prompt_keywords.ts` is a regex over user messages ("WTF", "this sucks", "horrible") that flips a boolean — track frustration rate per release as a near-free regression signal; it works in aggregate even when individual hits are noisy/non-English (a 10% rise across millions of users is strong signal regardless of per-case misses). [Raindrop](https://www.youtube.com/watch?v=-aM2EDTiaMs) (2026-05) (low-confidence src)
- LLM-as-judge for per-use-case model selection: run the same input through every candidate model (e.g. OCR engines), feed all outputs to a judge, ask which is best — ranking flips by dataset, so re-run per use case rather than picking one model globally. [RocketRide](https://podcasters.spotify.com/pod/show/mlops/episodes/AI-Is-Fast--AI-Projects-Are-Slow--Lets-Fix-That-e3k1c24) (2026-05)
- Self-grade docs/prompts before trusting them: paste in, ask the model to score 1–10 and rewrite "so you understand it better next time"; GPT over-scores (9–10), Claude grades harsher (~6) — use the harsh grader's rewrite. [RocketRide](https://podcasters.spotify.com/pod/show/mlops/episodes/AI-Is-Fast--AI-Projects-Are-Slow--Lets-Fix-That-e3k1c24) (2026-05)
- Out of the box Claude is a bad QA/critic agent — sycophancy/generosity bias makes it find a bug then say "fix later, might take two weeks" and move on; tune the judge prompt hard toward harshness. A generator/evaluator (GAN-style) split beats one session grading its own work — separate context windows, prompts, roles, with adversarial pressure; works because tuning a standalone critic to be harsh is tractable while tuning a builder to self-criticize is not. [Anthropic](https://www.youtube.com/watch?v=mR-WAvEPRwE) (2026-05)

### Who runs evals (domain-expert roles)
- Domain-relevance must be granular: "a doctor" isn't enough for a medical-coding eval — you need direct experience of the exact use case to know where it breaks. [Notius](https://www.youtube.com/watch?v=kfSDc2eVLo4) (2026-05)
- Three domain-expert eval roles that progress as you scale: oracle (reads traces/outputs, finds failure modes, tweaks prompts directly) → evaluator (defines quality metrics + builds the measurement system, hands fixes to engineers) → architect (designs a self-improving loop with minimal human-in-loop). [Notius](https://www.youtube.com/watch?v=kfSDc2eVLo4) (2026-05)
- Pick oracle (taste-based review) vs evaluator (metric-based) by asking: can quality be captured in an objective metric? If no → stay oracle; if yes but manual engineer iteration is fast enough → evaluator; if iteration is too slow / edge-case variation too high → architect (automated improvement). [Notius](https://www.youtube.com/watch?v=kfSDc2eVLo4) (2026-05)
- Serving as oracle first (deep failure-mode knowledge from reading traces) is what makes someone good at defining eval metrics — review-before-metricize ordering. [Notius](https://www.youtube.com/watch?v=kfSDc2eVLo4) (2026-05)
- "Decentralized oracle": when one reviewer can't cover all specialties/geos/note-types, fan out many domain reviewers each owning a subset of outputs + their own prompt variants (thousands of per-context prompt versions). [Notius](https://www.youtube.com/watch?v=kfSDc2eVLo4) (2026-05)

### Grading subjective quality / taste
- You CAN grade subjective quality/taste if you hold a strong opinion and write it down: a design rubric used 4 weighted criteria — design, originality, craft, functionality — weighted toward design+originality (functionality already strong) and reweighted per model. [Anthropic](https://www.youtube.com/watch?v=mR-WAvEPRwE) (2026-05)
- Calibrate taste with few-shot reference examples ("this is a Read.cv-pro look; this is AI slop") so the judge's taste converges on yours and avoids generic AI-slop aesthetics; generalizes across projects. [Anthropic](https://www.youtube.com/watch?v=mR-WAvEPRwE) (2026-05)
- Counterweight on grading style: code style has no ground truth — "is this file structure better?" splits humans ~80/20, so a library's "good patterns" are opinion, and only hard signals (compiles vs not) are objective; favor per-project evals over chasing absolute style correctness. [Effect](https://www.youtube.com/watch?v=Wmp2Tku2PrI) (2026-05)

### Writing the judge rubric (Voss)
- LLM-judge rubric = 5 parts: (1) judge role/domain context, (2) explicit observable criteria each mapped to a failure you actually saw in traces (never "good"/"helpful"), (3) clearly delimited data (XML tags — models are good at start/end boundaries), (4) labeled examples — the most-skipped + highest-value part (LLMs follow an example far better than an instruction list), (5) constrained output (one word: actionable/not). [Arize](https://www.youtube.com/watch?v=Xfl50508LZM) (2026-05)
- Don't ask judges for 1–10 ratings ("what's the difference between a 6 and a 7?" = noise unless each step is defined); use binary or at most 3 buckets (correct/partial/incorrect) — LLMs are bad at scoring, good at A/B. [Arize](https://www.youtube.com/watch?v=Xfl50508LZM) (2026-05)
- Pairwise eval (give the judge two outputs, ask which is better) beats numeric rating; ideal for A/B-ing prompt versions or model upgrades. [Arize](https://www.youtube.com/watch?v=Xfl50508LZM) (2026-05)
- Chain-of-thought for judges measurably improves judging — have it explain its reasoning BEFORE emitting the label. [Arize](https://www.youtube.com/watch?v=Xfl50508LZM) (2026-05)
- "God evaluator" anti-pattern: one judge testing accuracy+tone+completeness+policy+format at once is uncalibratable and gives no why-it-failed — split into one judge per dimension. [Arize](https://www.youtube.com/watch?v=Xfl50508LZM) (2026-05)
- Test judges in BOTH directions: cases where the behavior should occur AND where it shouldn't — else the agent cheats the eval (e.g. always web-searches and "passes" a does-it-search eval). [Arize](https://www.youtube.com/watch?v=Xfl50508LZM) (2026-05)
- Correctness/factuality judges fail on future-dated or very-recent data — the judge's own training cutoff makes it wrong; switch to a faithfulness (grounded-in-provided-context) eval instead. [Arize](https://www.youtube.com/watch?v=Xfl50508LZM) (2026-05)
- Faithfulness eval needs a separate `context` column = only the research/retrieval turn's output; split the agent into research-then-write so you can feed just the research as grounding and ask "did the report use only this?". [Arize](https://www.youtube.com/watch?v=Xfl50508LZM) (2026-05)
- Suppress tracing while running judges — else your judge LLM's own calls get captured as traces and pollute the data. [Arize](https://www.youtube.com/watch?v=Xfl50508LZM) (2026-05)

## Eval-the-eval & statistical significance
- Eval the eval: LLM-judge outputs are discrete, so build a ground-truth dataset of judge calls and score the judge against human decisions before trusting it. [Braintrust](https://www.youtube.com/watch?v=FB-MLPhL9Ms) (2026-05)
- Inter-rater reliability between two human experts on the same rubric is often only 0.2–0.3 — so a judge agreeing with you at 0.4 is doing better than two humans agree with each other; judge-disagrees-with-you only matters if it disagrees MORE than a human would. [Arize](https://www.youtube.com/watch?v=Xfl50508LZM) (2026-05)
- "Is the failure fair?" check (Anthropic meta-eval): if a failing trace looks fine to you, the eval is wrong, not the agent — CoreBench case: Opus scored 42%, root cause was the eval demanding exact "96.12" while Claude returned "96.124991"; fixing the eval → 95%. [Arize](https://www.youtube.com/watch?v=Xfl50508LZM) (2026-05)
- 0% score consistently ≈ broken task, not a bad agent — usually you made the eval task something only a human could do; write a reference solution per task in advance. [Arize](https://www.youtube.com/watch?v=Xfl50508LZM) (2026-05)
- Read the score-band for what it means on a hard dynamic task: 100% perfect is SUSPICIOUS (overfit or trivially easy), ~20% pass = stuck/broken, and a healthy hard task should have "play" and tolerate some mistakes — don't chase 100%. [MLOps-Sandboxing](https://podcasters.spotify.com/pod/show/mlops/episodes/Sandboxing--Agent-Harnesses--and-Agent-Teamwork-e3kts75) (2026-06)
- Statistical significance is expensive: ~400k poker hands needed for signal; use Bradley-Terry pairwise to cut matchups vs all-pairs. [DeepMind-Kang](https://www.youtube.com/watch?v=Ubwb6NzegyA) (2026-05)
- "Trust is a pass rate, a hash, a delta score" — measure with/without a change and compare; otherwise you're adding noise blind. [WorkOS](https://www.youtube.com/watch?v=vy7o1g2iHY8)
- Run each eval case with N repeats and judge on a pass-rate threshold, not a single run — absorbs LLM nondeterminism. [incident.io](https://www.youtube.com/watch?v=L2r6vLlLgs8) (2026-05); evals are non-deterministic by nature — track success RATE against an error budget, not exact pass/fail. [Context-Debois](https://www.youtube.com/watch?v=bSG9wUYaHWU) (2026-05)
- Validate a cheaper/newer-model downgrade by replaying production: capture real LLM inputs from logs, re-issue against the candidate, diff outputs vs the known-good expensive outputs; only switch if results hold. Worth automating as token-cheaper models ship. [RocketRide](https://podcasters.spotify.com/pod/show/mlops/episodes/AI-Is-Fast--AI-Projects-Are-Slow--Lets-Fix-That-e3k1c24) (2026-05)
- Eval-validity ops: classify exit statuses to separate model errors from infra errors (context overflow, too-many-tool-calls, provider errors) with an explicit rerun policy; reproduce an external benchmark's published numbers on your infra before trusting your own experiments; watch for silent default-parameter drift (reasoning level, caching) within a model family — 1–2 runs/month go invalid from this. [SWE-rebench](https://www.youtube.com/watch?v=wcUJWP6WpGM) (2026-06)
- Report pass@5 (any of 5 runs = capability ceiling) alongside pass^5 (all 5 = reliability) plus tokens/price per problem with confidence intervals — mean resolved alone hides both potential and flakiness. [SWE-rebench](https://www.youtube.com/watch?v=wcUJWP6WpGM) (2026-06)

## The improvement flywheel / hill-climbing
- Flywheel / hill-climb loop: capture trace → diagnose → replay offline → run suite → score → read failure traces → classify each failure ("portfolio allocate": didn't run tests, broken read-file tool, …) → pull the small high-leverage levers, steering agent changes. [Braintrust](https://www.youtube.com/watch?v=FB-MLPhL9Ms) (2026-05) [Cline](https://www.youtube.com/watch?v=QuuIywMG4s8) (2026-06)
- Prompt versioning as change management, not casual commits: make each prompt-change commit document the production failure that triggered it + what it corrects, so you can later trace why every version exists. [Databricks](https://www.youtube.com/watch?v=ObTPqBGsEbA) (2026-06)
- Distributional view across all runs, not single traces: aggregate every instantiation into a path/branch/loop graph — % traffic per branch, latency per component within a branch, eval-score drop per path; root-cause pattern: B called before its dependency A on the bad path → fix by adding ordering context. [Arize](https://www.youtube.com/watch?v=JsCCrBF7F1g) (2026-06)
- Failure-mode-driven eval: run each trace through a second LLM that labels it against a documented set (e.g. "context window flooded from too-wide log query", "6 attempts when 1 would do") to mechanize trace classification. [MLOps-Sandboxing](https://podcasters.spotify.com/pod/show/mlops/episodes/Sandboxing--Agent-Harnesses--and-Agent-Teamwork-e3kts75) (2026-06)
- Trace-linked evals: attach eval results to the trace so detection→diagnosis is one hop — swap a model, the eval flags a tool-call metric drop, jump to the trace to see which tool stopped being called. [Foundry](https://www.youtube.com/watch?v=iOXM3zE-2dk) (2026-05) (low-confidence src)
- Build your own LLM tracing tooling rather than buying SaaS: full visibility into each tool call + reasoning/search trail, structured your way, with a UI non-engineers (product, CX) can use — not raw CloudWatch queries. LLMs make this cheap enough to be worth it, and it's what turns black-box agent runs into a debuggable feedback loop. [Granola](https://www.youtube.com/watch?v=ON5LIT0M4do) (2026-05)
- Agent-level eval ladder (not a single LLM call): intent resolution → tool-call correctness → task adherence/completion across the whole workflow; task adherence is the flaky one to tune. [Foundry](https://www.youtube.com/watch?v=iOXM3zE-2dk) (2026-05) (low-confidence src)
- Eval-gated changes: every change (model swap, cache vs web search) immediately re-runs evals + compares for regression; agent versions are just identifiers, with auto-rollback to the best-scoring version. [Foundry](https://www.youtube.com/watch?v=iOXM3zE-2dk) (2026-05) (low-confidence src)
- Primary harness-debugging loop is reading traces by hand, not running more experiments — find where the agent's judgment diverged from yours, tune the prompt there (same muscle as reading a stack trace). Piping transcripts to another agent that greps them and proposes prompt updates closes the loop on building the harness itself, but it's a first pass only — hand-reading still wins. [Anthropic](https://www.youtube.com/watch?v=mR-WAvEPRwE) (2026-05)
- "Empathize with the model" when reading traces: simulate its POV (e.g. for Claude-for-Chrome the team navigated pages with eyes closed, opening for a static glimpse every 10s) to understand why it acted as it did, then fix the instructions. [Anthropic](https://www.youtube.com/watch?v=mR-WAvEPRwE) (2026-05)
- Reading traces > judging output alone: a report written perfectly but to disk (failed write) looks like garbage on output yet had all the right research — only the trace reveals it; suspiciously-precise numbers for a non-public company = a hallucination tell. [Arize](https://www.youtube.com/watch?v=Xfl50508LZM) (2026-05)
- Prioritize failures by severity × frequency, not raw frequency — a rare go-completely-off-the-rails beats a frequent minor hallucination. [Arize](https://www.youtube.com/watch?v=Xfl50508LZM) (2026-05)
- Harness quality isn't one eval at the end: define "what good looks like", let generator+evaluator derive the tests/contracts, and measure hill-climbing from start score to end score within a run (not comparable across products/runs). [Anthropic](https://www.youtube.com/watch?v=mR-WAvEPRwE) (2026-05)
- Rubric evals as diagnosis layer over a scalar reward: decompose response rightness into per-behavior questions to localize WHICH behavior fails → drives what training data to generate; the RL loop still consumes the single pass/fail scalar — rubric is for analysis, not reward. [Snorkel-RL](https://www.youtube.com/watch?v=TNwJ1LMiENk) (2026-06)
- Three zones of improvement: (1) obvious harness bugs / rate limits, (2) model-family-specific prompt tweaks (Anthropic ≠ Codex ≠ Gemini) — highest value, (3) overfitting / benchmark-cheating — danger zone. [Cline](https://www.youtube.com/watch?v=QuuIywMG4s8) (2026-06)
- Tunable levers beyond prompts: container CPU/mem, timeouts, thinking behavior — and more thinking ≠ better (model loops). [Cline](https://www.youtube.com/watch?v=QuuIywMG4s8) (2026-06)
- You only know a skill helps or hurts by measuring — run scenarios with/without and compare pass rates; one measured skill made a task 77% correct vs 97% without it, and evals caught a 95%-skill-deletion win. [WorkOS](https://www.youtube.com/watch?v=vy7o1g2iHY8)
- Highest-signal training/knowledge samples come from the stuck→unstuck transition; then ask "what should we have injected at the start to skip the friction?" [Lovable](https://www.youtube.com/watch?v=KA5kPbdkK2E) (2026-06)
- Blank-injection holdout: for a small % where context would be injected, inject nothing; compare injected vs could-have-injected cohorts to measure each knowledge entry's real production value. [Lovable](https://www.youtube.com/watch?v=KA5kPbdkK2E) (2026-06)
- LLM-generated, auto-run evals validate a new knowledge entry actually resolves the cases that spawned it before it ships. [Lovable](https://www.youtube.com/watch?v=KA5kPbdkK2E) (2026-06)
- Close the loop: a CI action fed eval feedback can auto-optimize the context itself. [Context-Debois](https://www.youtube.com/watch?v=bSG9wUYaHWU) (2026-05)
- Ship gate: pass both the vibe (feels good to use) and the score. [Cline](https://www.youtube.com/watch?v=QuuIywMG4s8) (2026-06)
- Auto-approve PRs via a tested agent suite: backtest on past PRs + human-labeled outputs to set a confidence level, and shape PRs toward safe/simple so they're auto-approvable (~17.6% auto-approval rate); this eval-gated auto-approval stayed SOC2/ISO27001/HIPAA compliant with no human-in-loop — needs auditing controls + knowing exactly what agents do. [Intercom](https://www.youtube.com/watch?v=4_VQBbs2iQA) (2026-05)
- Same-context self-validation rubber-stamps its own work (confirmation bias); sub-agent validation (separate context) actually catches things — but there's still no objective way to know a prompt/skill change improved: A/B is confounded by model non-determinism, so iteration stays partly subjective. [Ralph-Parsons](https://www.youtube.com/watch?v=2TLXsxkz0zI) (2026-05)

### Prompt / agent-config optimization (GEPA)
- GEPA (Genetic Pareto, Lakshya Agrawal/Berkeley) = string optimizer over a prompt or any JSON/key-value object — breeds Pareto-frontier candidates and proposes new ones ("ask an agent for a new prompt, keep the bits that score better, repeat to a time/score budget"); available in DSPy but not only DSPy. [Pydantic-Colvin](https://www.youtube.com/watch?v=A48uhxfxbsM) (2026-05)
- Optimization payoff is biggest when forcing a cheap/dumb model to match a frontier model on PRIVATE untrained, high-volume data; on public-knowledge tasks a frontier model already gets it right so the gain is marginal, and since an optimized prompt is model-specific (change the model → re-run optimization) it rarely pays for general tasks the next model supersedes. Shopify: agent + Qwen + GEPA cut fraud/tax classification from $5M/yr → ~$73K/yr while improving quality; a PE firm optimizing a Qwen-mini for weeks vs GPT-5 over 200M invoices saw a ~$10M delta. [Pydantic-Colvin](https://www.youtube.com/watch?v=A48uhxfxbsM) (2026-05)
- Prefer deterministic golden-set evals over LLM-judge ("lunatics running the asylum"); golden-set sources when no reference exists = run the agent-written code and check it compiles/runs/uses-real-libs, or close a feedback loop back to the source to measure accuracy. [Pydantic-Colvin](https://www.youtube.com/watch?v=A48uhxfxbsM) (2026-05)
- The hard part of evals is defining "right," and it gets harder as models get smarter; when the true outcome is unmeasurable (stop-smoking agent — can't wait 40yrs) fall back to a proxy metric (output contains/excludes required words). Much of real eval reduces to crude string checks. [Pydantic-Colvin](https://www.youtube.com/watch?v=A48uhxfxbsM) (2026-05)
- GEPA overfits: it sees only a small train subset, so it learned to explicitly EXCLUDE uncles/aunts present in the golden set → 96% but a systematic error. Fix = ≥2× data covering the full space + a separate train/eval split (classic ML train/test). [Pydantic-Colvin](https://www.youtube.com/watch?v=A48uhxfxbsM) (2026-05)
- Variance reduction = run each case many times (repeat count); hedge funds reportedly spend ~$20K/night on overnight eval suites. The evaluator returns a LIST of floats, so you fold variance + accuracy + cost + latency into one optimization target. [Pydantic-Colvin](https://www.youtube.com/watch?v=A48uhxfxbsM) (2026-05)
- GEPA verbosity failure: a GPT-5-mini proposer generated bloated system prompts (slower, no better). Fix = give it a key/value store not free-form text — split the prompt into ~200 sentences and let it SELECT the best 20; a cap forces concision. [Pydantic-Colvin](https://www.youtube.com/watch?v=A48uhxfxbsM) (2026-05)
- Optimization can target beyond the prompt — model choice, compaction strategy, how tools are registered, code-mode inclusion — i.e. the whole agent config (exact model may then matter less). [Pydantic-Colvin](https://www.youtube.com/watch?v=A48uhxfxbsM) (2026-05)
- Coding agents are the hard case: trajectory space is so sparse/wide that even 5K eval cases overfit; coding-agent cos reportedly run few/no evals (per Boris Cherny, "mostly vibes"). Broader task → harder to optimize → just use a clever frontier agent. [Pydantic-Colvin](https://www.youtube.com/watch?v=A48uhxfxbsM) (2026-05)

### Continuous / production eval & behavior auditing
- 5-axis ongoing eval run continuously to catch drift: quality (LLM-as-judge on a fixed use-case set), performance (track P99 not just P50/median — the tail hides token blowups), safety, cost (per-run not per-agent, to find which runs spike), business impact. [PwC](https://podcasters.spotify.com/pod/show/mlops/episodes/Autonomous-Agents-at-Work-From-OpenClaw-Hype-to-Enterprise-Reality-e3jj7u3) (2026-05)
- Audit behavior, not just outputs: a safe-looking answer can hide unsafe behavior — real signal is abnormal tool calls, strange network egress, odd retries, scope expansion. [PwC](https://podcasters.spotify.com/pod/show/mlops/episodes/Autonomous-Agents-at-Work-From-OpenClaw-Hype-to-Enterprise-Reality-e3jj7u3) (2026-05)
- Log chain-of-thought into a system-of-record so each decision is traceable; mine the logs to find where task scope is expanding (signals new tools / CoT revisions needed). [PwC](https://podcasters.spotify.com/pod/show/mlops/episodes/Autonomous-Agents-at-Work-From-OpenClaw-Hype-to-Enterprise-Reality-e3jj7u3) (2026-05)
- Measuring agent-context success is squishy — when there's no clean score, track sentiment trend plus wall-clock/token metrics over time (echoes Boris Cherny on Claude Code being run "on vibes"). [Mergeable-Werry](https://www.youtube.com/watch?v=5ID22ACI7IM) (2026-05)
- Kill a saturated eval KPI once it plateaus: an "in-scope: can the agent answer the prompt" metric plateaued ~95–96%, so they stopped tracking the asymptotic metric and shifted to value/coverage metrics (gross-booking attribution, % users engaged). [Despegar](https://podcasters.spotify.com/pod/show/mlops/episodes/Building-MCP-Before-MCP-Existed-Inside-Despegars-Sofia-Agent-e3j1si9) (2026-05)
- The 70%-accuracy demo→prod gap is task-shaped: a coding agent at 70% saves keystrokes and is cheap to verify, but an SRE agent at 70% wastes 30 stressful minutes and kills trust. Past simple tasks, VERIFICATION (not search) is the hard problem — subtle issues (latency +20%) need company-specific context/conventions to rank causes. [MLOps-Sandboxing](https://podcasters.spotify.com/pod/show/mlops/episodes/Sandboxing--Agent-Harnesses--and-Agent-Teamwork-e3kts75) (2026-06)

### Adaptive / self-modifying evals
- "Eval calcification": evals rot faster than benchmark-leakage alone explains because the *application self-modifies* — the harness writes its own skills, adapts to user intent, ships at token-speed — so a static benchmark can't track a target that rewrites itself between runs. [Malleable-Koc](https://www.youtube.com/watch?v=4VhbYlfC7Gs) (2026-05)
- Per-user intent engineering breaks ground truth: when the harness self-optimizes a different experience per person, there's no single correct answer to grade — the eval must handle ambiguity/personality, not exact-match. [Malleable-Koc](https://www.youtube.com/watch?v=4VhbYlfC7Gs) (2026-05)
- Invert the eval: stop treating it as the starting dataset (input→expected output); make the eval the *desired end-state + a reward signal* and let the machine self-optimize toward it (Karpathy-style auto-research loop). [Malleable-Koc](https://www.youtube.com/watch?v=4VhbYlfC7Gs) (2026-05)
- Treat the eval set itself as living code: an agent watches production traces, detects when the question/customer distribution has shifted, and rewrites the suite. Pair with a telemetry-in-the-loop self-healing harness — fed its own telemetry + thresholds it hits an error → fixes itself → continues, rather than pre-predicting every failure. [Malleable-Koc](https://www.youtube.com/watch?v=4VhbYlfC7Gs) (2026-05)
- 80/20 framing: ~80% of agent behavior is static/intentful and definable up front; the ~20% that keeps changing (weird, off-distribution questions) is what wrecks the business — assign an agent to watch and adapt evals to that 20%, don't over-invest the fixed 80%. [Malleable-Koc](https://www.youtube.com/watch?v=4VhbYlfC7Gs) (2026-05)
- "Context gap scanner" = eval a knowledge base by replaying archived work items against it at scale, scoring each clean/stale/incomplete/missing/tribal → a prioritized Kanban (3 steps: generate probes → run → analyze); prioritize gaps by recurrence across the corpus. Cheap recurring scan (~<$1, <100k tokens/domain). [DemandContext-Raj](https://www.youtube.com/watch?v=_QAVExf_1uw) (2026-05)
- Staleness handling: flag KB entries past an age threshold but surface them for a human decision — don't auto-discard. [DemandContext-Raj](https://www.youtube.com/watch?v=_QAVExf_1uw) (2026-05)

## When to fine-tune
- Plateauing evals = signal it may be time to fine-tune rather than keep prompt-engineering. [Modal](https://www.youtube.com/watch?v=HvZXAOZ3iv8) (2026-06)
- Fine-tuning a tiny model beats prompting it for narrow tool-use: Function Gemma (270M, fine-tuned for function calling) went ~46% out-of-the-box → >90% on 8/10 functions (rest in 80s) on a 7–10 function "app intents" task. [Function-Gemma](https://www.youtube.com/watch?v=-TiET_K-E_g) (2026-05)
- Synthetic-data fine-tune workflow for tool-use: bake the function set into the model rather than describe it in a system prompt — use a larger model (Gemini Flash) to synthetically generate the call-example training set, then fine-tune; standard path to robust on-device tool-calling. Corroborated as the small-model recipe (synthetic data from a bigger model → fine-tune the small one) with a measured ~20–40 pt eval lift, function-calling 46% → 86–90%. [Function-Gemma](https://www.youtube.com/watch?v=-TiET_K-E_g) (2026-05) [LiteRT-LM-Brick](https://www.youtube.com/watch?v=BKWpYIWvAo4) (2026-05)
- Fine-tune vs prompt by size: below ~100–200M params a model only works if the task is very narrow/focused — pick fixed-function or fine-tune, don't expect general prompt-following; prompting a larger model is easier, but a fine-tuned tiny model ships robust narrow tool-use at scale. [Function-Gemma](https://www.youtube.com/watch?v=-TiET_K-E_g) (2026-05)
- No mature evals / no collected data = not ready to train (garbage in, garbage out); build the eval+data pipeline first. [Modal](https://www.youtube.com/watch?v=HvZXAOZ3iv8) (2026-06)
- A working harness + collected eval traces ≈ ready-made training data — RL can teach a model your service from what the harness already logs. [Modal](https://www.youtube.com/watch?v=HvZXAOZ3iv8) (2026-06)
- Big-reasoning-model failure on tool tasks is often tool indiscipline, not weak reasoning: Qwen3-235B queried nonexistent tables without ever calling the available schema tools, then hallucinated; an RL'd 4B learned discover-tools → inspect-schema → self-correct-on-error — diagnose the tool-use behavior before reaching for a bigger model. [Snorkel-RL](https://www.youtube.com/watch?v=TNwJ1LMiENk) (2026-06)
- RL for behavior, not knowledge: changing how a model acts (tool discipline) is an RL problem vs SFT for content; GRPO on a 4B, ~21h, <$500/run, doubled pass@1 past a 235B — behavior-targeted RL is cheap enough for self-hosted teams. [Snorkel-RL](https://www.youtube.com/watch?v=TNwJ1LMiENk) (2026-06)
- Ablation surprise: training on easy (single-table) tasks only beat both mixed-data and easy→hard curriculum — and the gain generalized to the harder multi-table benchmark (13.9→26.6% pass@1); fixing one core behavior generalizes better than training on the hard distribution. [Snorkel-RL](https://www.youtube.com/watch?v=TNwJ1LMiENk) (2026-06)
- Self-contained published RL envs (FinQA on Prime Intellect, OpenEnv repo, HF Spaces; Harbor-style, no external deps) — reusable starting point for tool-use RL. [Snorkel-RL](https://www.youtube.com/watch?v=TNwJ1LMiENk) (2026-06)
- Agent training data doesn't exist in the wild (you can't scrape an agent using tools) — stand up an RL environment + reward and the environment itself becomes the synthetic-data pipeline: generate trajectories, keep the good ones by reward, rejection-sample to bootstrap the first SFT dataset. To train/eval without a real environment, mock the tools and mock the *user* with an LLM trained on real transcripts so it behaves like a realistic (annoying, repetitive, panicked) customer. [Adaptive ML](https://www.youtube.com/watch?v=X6NShR2ccOg) (2026-05)
- Reward-source hierarchy, cleanest first: systematic/verifiable (does the code run? syntax?) → direct business KPI (e.g. support containment rate) → open-ended qualities (tone, guideline adherence) via LLM-as-judge. Human-in-the-loop is mostly defining judge rubrics + judge system-prompts + scenarios (minutes-to-hours, once) — NOT running weekly annotation campaigns (expensive, and people won't actually do them). [Adaptive ML](https://www.youtube.com/watch?v=X6NShR2ccOg) (2026-05)
- Judge→reward-model scaling path: with only ~10–20 feedbacks, refine an LLM-judge driven by a big prompted model (e.g. Qwen 235B); at production scale (thousands of feedbacks) train a reward model from that data to scale the signal. Implicit production signals (Cursor-style tab accept/reject) can become a reward model too — single-signal vs multi-rollout is use-case-dependent, pick by eval. [Adaptive ML](https://www.youtube.com/watch?v=X6NShR2ccOg) (2026-05)
- "Last mile to production" is a myth — the MVP is the *first* mile; prompt-only or SFT-only can't systematically absorb defects found en route (prompt tweaks trade one defect for another; SFT means rebuilding the dataset weekly). Small RL-tuned ~10B models (Gemma/Mistral/Qwen) beat large ones on the two prod gating constraints — tokenomics at scale and latency (speech-to-speech needs ~1/3s; >1/2s feels broken) — and agents widen that advantage (~10x tokens, less error budget since they mutate the DB). ## Decision-quality evals (high-stakes agents)
- Reference-class validation: don't act on general/statistical norms — first classify whether this case is the common 99% or the dangerous 1% (the norm can be fatal in the tail). [Neo4j-Kollegger](https://www.youtube.com/watch?v=abvQEhvRI_c) (2026-05)
- Scale decision effort to stakes: weigh reversibility and cost-of-being-wrong; spend more compute on serious, irreversible actions. [Neo4j-Kollegger](https://www.youtube.com/watch?v=abvQEhvRI_c) (2026-05)
- Force the objective explicit (maximize value vs minimize cost) — left implicit, the agent defaults to "what's usually right," but particulars dominate. [Neo4j-Kollegger](https://www.youtube.com/watch?v=abvQEhvRI_c) (2026-05)
- Design a task-specific eval metric that captures the real-world objective, not just correctness — for AI tutors the relevant metric was "cognitive load placed on the student," evaluated across frontier models. [10DS](https://www.youtube.com/watch?v=ObNKGf9YR0g) (2026-05)

## Red-teaming & safety-failure evals
- Red-team a model against its intended user-manipulation failure mode BEFORE exposing it to users — e.g. a policy-simulation tool was red-teamed against the "should I cut income tax to 0%?" leading-question/sycophancy attack before shipping. [10DS](https://www.youtube.com/watch?v=ObNKGf9YR0g) (2026-05)
- LLM-as-second-judge to audit self-reports: a "delivery red teaming" tool reads incoming status reports and flags meta-signals — optimism bias, disproportionately self-rating own risks as "amber," whether stated mitigations are usually effective — i.e. use an agent to audit the credibility of human/agent self-reports, not just summarize them. [10DS](https://www.youtube.com/watch?v=ObNKGf9YR0g) (2026-05)
- AISI's `inspect` is a referenced environment for testing what agents actually do given autonomy + tools, run in a safe isolated sandbox — tooling pointer for agent-behavior evals. [10DS](https://www.youtube.com/watch?v=ObNKGf9YR0g) (2026-05)
- Agentic red-teaming has agent-specific attacks beyond model safety: "prohibited actions" — give a taxonomy of disallowed actions (fill passwords, leak data) and have a red-team agent generate prompts trying to manipulate the agent into them; the Crescendo attack = slow escalating multi-turn, expensive but thorough. [Foundry](https://www.youtube.com/watch?v=iOXM3zE-2dk) (2026-05) (low-confidence src)

## Eval limits in large codebases
- Complexity is the agent's enemy: the ~200k context window caps how much of a large codebase fits; agent-generated sprawl eventually exceeds its own ingestion limit. [Pragmatic-Pi](https://newsletter.pragmaticengineer.com/p/building-pi-and-what-makes-self-modifying) (2026-04)
- Models regress to the mean of training data (mostly mediocre internet code); well-engineered projects are a rounding error in the corpus. [Pragmatic-Pi](https://newsletter.pragmaticengineer.com/p/building-pi-and-what-makes-self-modifying) (2026-04)
- "Best possible spec is the software itself" — any spec gaps get filled from mediocre training priors, multiplied across many parallel agents. [Pragmatic-Pi](https://newsletter.pragmaticengineer.com/p/building-pi-and-what-makes-self-modifying) (2026-04)
